# 什么是规划执行观察循环？

## 面试问题

什么是规划执行观察循环？

## 考察重点

- 是否理解 Agent Loop 的核心机制，以及它与普通 API 调用的本质区别
- 是否能说清楚规划、执行、观察三个阶段的各自作用和相互关系
- 是否了解 Claude Code 的 while(true) 循环和 Codex 的事件驱动模型的实现差异
- 是否能从源码级别理解 Agent Loop 的停止条件和循环控制

## 核心结论

规划执行观察循环（Agent Loop）是智能体系统的核心控制结构，由"规划下一步"、"执行动作"、"观察结果"三个阶段组成。这不是一个新鲜概念，但生产环境中的实现细节才是真正考验功力的地方——如何优雅地处理循环终止、错误恢复、上下文膨胀，直接决定系统是否能上线。

## Agent Loop 的三层结构

### 第一层：规划（Plan）

模型根据当前状态决定下一步行动。这个"状态"包括：
- 原始用户目标
- 已执行的步骤历史
- 上一步的观察结果
- 可用的工具列表
- 剩余的上下文空间

**规划有两种策略**：

| 策略 | 适用场景 | 代表系统 |
|-----|---------|---------|
| **ReAct**（边做边想） | 简单任务、交互式探索 | 早期 LangChain Agent |
| **Plan-and-Execute**（先想后做） | 复杂任务、需要全局规划 | Claude Code、Codex |

### 第二层：执行（Act）

模型输出结构化的工具调用请求。注意：这是"请求"而非"执行"。

```json
{
  "name": "shell",
  "arguments": {"command": ["ls", "-la"]}
}
```

真正的执行由框架完成：
1. 解析工具名和参数
2. 权限检查（白名单、危险命令检测）
3. 沙箱执行
4. 捕获输出

### 第三层：观察（Observe）

执行结果被转成文本，追加到上下文。模型基于新的上下文决定：
- 任务完成？→ 终止循环
- 需要继续？→ 回到第一层
- 出错了？→ 尝试恢复或报告

## Claude Code 的 Agent Loop 实现

Claude Code 的核心循环在 `query.ts` 中，代码结构如下：

```typescript
// query.ts — line 307
while (true) {
  let { toolUseContext } = state;
  const { ... } = state;
  // ... 1,421 lines of state machine logic ...
  state = next;
}
```

关键特点：

1. **不是递归**：早期版本用递归，长对话会栈溢出。while(true) + 状态对象避免了这个问题

2. **10 步迭代**：每次循环包含：
   - Context Compression（4 阶段压缩）
   - Token Budget Check
   - Call Model API
   - Streaming Tool Execution
   - Error Recovery
   - Stop Hooks
   - Token Budget Check (Again)
   - Tool Execution
   - Attachment Injection
   - Assemble and Loop

3. **9 个 continue 点**：每个代表一次状态转换，可追踪调试

4. **Circuit Breaker 机制**：`hasAttemptedReactiveCompact` 防止上下文压缩死循环

## Codex 的 Agent Loop 实现

Codex 采用事件驱动模型，通过 HTTP SSE 流式返回：

```
data: {"type":"response.reasoning_summary_text.delta","delta":"ha ", ...}
data: {"type":"response.output_item.added","item":{...}}
data: {"type":"response.output_text.delta","delta":"forty-", ...}
data: {"type":"response.completed","response":{...}}
```

核心循环：

```
用户输入 → 构建提示 → 调用 Responses API
                              ↓
                    收到 SSE 事件流
                              ↓
              ┌───────────────────────────────┐
              │ 模型可能多次发起 function_call │
              │ 或直接返回 output_text.done   │
              └───────────────────────────────┘
                              ↓
              如果是 function_call → 执行工具 → 追加结果到 input → 再次调用 API
                              ↓
              如果是 output_text.done → 返回给用户 → 循环结束
```

## 停止条件设计

停止条件是 Agent Loop 最关键的部分。设计不当会导致：

| 问题 | 后果 |
|-----|------|
| 没有停止条件 | 无限循环，耗尽 token |
| 停止条件太激进 | 任务未完成就退出 |
| 停止条件太宽松 | 执行不必要的步骤，浪费资源 |

### 常见的停止条件

1. **模型主动停止**：模型输出 `output_text` 而非 `function_call`
2. **Token 耗尽**：上下文窗口达到上限（Claude Code 有 4 阶段压缩缓解）
3. **步骤超限**：`turnCount > maxTurns`
4. **时间超限**：任务执行超过 `maxTime`
5. **用户中断**：用户主动取消
6. **Circuit Breaker**：连续失败 N 次，强制停止

### Claude Code 的停止检查点

在每次循环迭代结束时检查：

```typescript
if (state.turnCount > MAX_TURNS) {
  state.transition = { reason: "max_turns_exceeded" };
  break;
}
if (state.tokenBudget <= 0) {
  state.transition = { reason: "token_budget_exhausted" };
  break;
}
if (state.stopSignalReceived) {
  state.transition = { reason: "user_stopped" };
  break;
}
```

## 目标漂移问题

Agent Loop 最常见的问题是**目标漂移（Goal Drift）**：模型在执行过程中逐渐偏离原始目标。

### 原因

1. **上下文膨胀**：历史信息淹没关键目标
2. **局部最优**：模型在某个子目标上反复优化
3. **外部干扰**：用户后续输入改变了方向
4. **规划错误**：初始规划本身就有偏差

### 解决思路

| 方法 | 实现 |
|-----|------|
| 周期性目标检查 | 每 N 步检查当前状态是否还在目标范围内 |
| 用户确认 | 关键步骤前请求用户确认 |
| 目标重置 | 上下文压缩时保留原始目标 |
| 失败计数 | 连续失败超过阈值，触发恢复 |

Claude Code 的 `reactive compact` 就是解决目标漂移的机制之一。

## 工程实践：如何设计一个健壮的 Agent Loop

### 1. 状态结构设计

```typescript
interface AgentState {
  // 进度追踪
  turnCount: number;
  steps: Array<{
    action: string;
    observation: string;
    timestamp: number;
  }>;

  // 资源管理
  tokenBudget: number;
  tokenBudgetUsed: number;

  // 错误处理
  errorCount: number;
  lastError: Error | null;

  // 控制流
  transition: { reason: string } | null;
  stopSignalReceived: boolean;

  // Circuit Breaker
  hasAttemptedReactiveCompact: boolean;
  consecutiveFailures: number;
}
```

### 2. 结构化日志

每次循环迭代都要记录：
```json
{
  "turn": 42,
  "transition_reason": "tool_call_completed",
  "tool_name": "shell",
  "tool_args": {"command": ["git", "status"]},
  "tool_result": {"status": "success", "output_length": 1024},
  "token_budget_used_pct": 67.2,
  "error_count": 0
}
```

### 3. 优雅降级

当 Agent Loop 出现问题时，应该有序降级：

```
正常 → 警告 → 降级 → 放弃
  ↓      ↓       ↓       ↓
继续   记录    压缩    报告
执行   日志    上下文   用户
```

## 真实踩坑案例

### 案例：上下文窗口耗尽导致的"半成品"任务

某团队的智能体在执行代码生成任务时，前景光明地生成了 200 行代码，然后因为上下文耗尽戛然而止。用户拿到的是不完整的代码，既不能运行，也不知道模型"想做什么"。

**问题根因**：上下文管理没有考虑"任务完整性"。

**修复方案**：
1. 引入"检查点"机制，每生成 N 行代码就保存一次
2. 上下文压缩时，优先保留任务目标和已完成的关键步骤
3. 任务开始前检查 token 预算是否足够
