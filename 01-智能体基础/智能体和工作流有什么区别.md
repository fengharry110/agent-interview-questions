# 智能体和工作流有什么区别？

## 面试问题

智能体和工作流有什么区别？

## 考察重点

- 是否理解"智能体"和"工作流"在控制系统复杂度上的不同定位
- 是否能说清楚什么时候该用工作流、什么时候该用智能体
- 是否了解混合使用两者的场景和实现方式
- 是否理解工作流的确定性 vs 智能体的不确定性的工程含义

## 核心结论

工作流是**人在环内**的确定性执行管道，智能体是**人在环外**的自主决策系统。两者不是替代关系，而是适用于不同复杂度级别的工具。生产系统中，常见的做法是用工作流控制宏观流程，用智能体处理局部不确定性任务。

---

## 本质区别：控制权的分配

### 工作流：人定义控制流

```
用户 → [节点A] → [节点B] → [节点C] → 用户
         ↓          ↓          ↓
      人工干预    人工干预    人工干预
```

工作流中，每个节点做什么、由谁执行、什么时候执行，都是**预先定义好**的。系统只是按图索骥地执行。

### 智能体：模型决定控制流

```
用户 → 模型决定下一步 → 可能是工具X → 观察结果
                      → 可能是工具Y → 观察结果
                      → 直接返回用户
```

智能体中，下一步做什么由模型根据上下文**动态决定**。同一个输入，不同时候可能走不同的路径。

---

## 对比矩阵

| 维度 | 工作流 | 智能体 |
|-----|-------|-------|
| **控制流** | 预先定义 | 动态决定 |
| **人在环内** | 是，通常每个节点 | 否，通常只在边界 |
| **执行确定性** | 高，完全按定义执行 | 低，依赖模型输出 |
| **可预测性** | 高，路径可枚举 | 低，需要测试覆盖 |
| **适用复杂度** | 低-中（流程固定） | 中-高（需要判断） |
| **调试难度** | 低，节点边界清晰 | 高，黑盒决策 |
| **成本控制** | 容易，按节点计费 | 难，token 消耗不确定 |
| **延迟** | 可预期 | 不可预期 |

## 典型场景分析

### 场景一：客服工单处理

**用工作流更合适**：
```
用户提交工单 → 分类（规则判断）→ 优先级（规则判断）→ 分配（规则判断）→ 处理
```

这个流程是固定的：分类 → 优先级 → 分配，每一步都有明确规则。用工作流可以精确控制成本和延迟。

**什么时候需要智能体**：
- 当用户的问题不在知识库里，需要智能体去检索、推理、生成答案
- 当工单内容复杂，需要多步工具调用才能完成

### 场景二：代码审查

**用智能体更合适**：
```
代码变更 → 智能体决定审查重点 → 可能调用的工具：
                                        - 查找相关测试
                                        - 检查安全漏洞
                                        - 分析性能影响
                                        - 生成审查意见
```

代码审查需要根据代码内容**动态决定**审查重点，用工作流很难预先定义所有可能的审查路径。

### 场景三：报表生成

**混合架构**：
```
用户请求 → 工作流：数据获取 → 工作流：数据清洗 → 智能体：生成分析 → 工作流：格式化 → 用户
                        ↓                    ↓
                    固定API调用           LLM生成
```

宏观流程是固定的（获取 → 清洗 → 生成 → 格式化），但"生成分析"这一步需要灵活性，由智能体处理。

## 工程实现差异

### 工作流的工程特点

1. **DAG 定义**：通常用 YAML/JSON 定义节点和依赖关系
2. **节点类型**：代码节点、LLM 节点、HTTP 节点、条件节点
3. **状态管理**：每个节点的输入输出，失败重试策略
4. **可观测性**：节点级别的执行记录，精确知道哪一步花了多少时间

```yaml
# 示例：LangChain Expression Language (LCEL)
chain:
  steps:
    - name: classify
      type: llm
      model: gpt-4
      prompt: "Classify: {input}"
    - name: route
      type: conditional
      condition: "classify.topic == 'billing'"
      branches:
        billing: [process_refund]
        other: [general_response]
```

### 智能体的工程特点

1. **Agent Loop**：while(true) + 状态机（参考 Claude Code 的 1421 行循环）
2. **工具注册**：动态的工具列表，模型自行选择
3. **上下文管理**：动态增长的上下文，需要压缩和预算控制
4. **可观测性**：执行轨迹，需要专门的结构化日志

```typescript
// 智能体的核心循环
while (true) {
  const context = await compressContext(state);
  const decision = await model.decide(context, tools);
  if (decision.type === 'tool_call') {
    const result = await executeTool(decision);
    state.history.push({ decision, result });
  } else if (decision.type === 'response') {
    return decision.text;
  }
}
```

## 什么时候不该用智能体

| 场景 | 原因 | 替代方案 |
|-----|------|---------|
| 流程固定、很少变化 | 智能体的灵活性是浪费 | 工作流 |
| 需要精确控制延迟 | 模型推理延迟不可预测 | 规则引擎 |
| 需要精确控制成本 | token 消耗难以预估 | 工作流 |
| 决策必须可解释 | 模型决策是黑盒 | 规则引擎 + 决策树 |
| 任务非常简单 | 用智能体是杀鸡用牛刀 | 规则判断 |
| 需要人工审核 | 智能体的自主决策反而是负担 | 工作流 |

---

## 什么时候该用智能体

| 场景 | 原因 | 示例 |
|-----|------|-----|
| 流程不确定 | 需要根据输入动态决定 | 智能客服 |
| 需要多步推理 | 固定流程无法覆盖 | 代码审查 |
| 工具组合不确定 | 需要根据任务选工具 | 数据分析 |
| 输出格式灵活 | 强结构化反而限制能力 | 文案生成 |
| 探索性任务 | 需要边做边想 | 研究助手 |

## 面试追问方向

1. **"如果一个任务可以用工作流完成，你还会用智能体吗？考虑哪些因素？"**

   考察候选人对技术选型的权衡能力。

2. **"工作流中的一个 LLM 节点和智能体有什么区别？"**

   考察候选人对两者边界条件的理解。

3. **"你如何在一个工作流中嵌入智能体？有没有具体的实现方式？"**

   考察候选人的工程经验。

4. **"Claude Code 是一个智能体，但它也执行固定的工作流程（比如先读取文件、再编辑）。这算不算工作流？"**

   考察候选人对工作流和智能体融合的理解。

5. **"如果智能体执行的步骤数不可预测，你如何控制成本？"**

   考察候选人对成本控制的理解。

## 真实踩坑案例

### 案例：智能体用在不该用的地方

某团队用智能体实现"一句话回复"功能——用户发一条消息，智能体生成回复。结果：
- 成本：每次调用 0.1 元
- 延迟：平均 3 秒
- 质量：70% 回答和固定回复没区别

后来改成规则匹配 + 工作流，成本降到 0.001 元，延迟降到 100ms。

**教训**：当任务足够简单、有固定模式时，不要为了"智能"而用智能体。
