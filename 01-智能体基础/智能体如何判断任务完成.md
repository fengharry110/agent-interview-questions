# 智能体如何判断任务完成？

## 面试问题

智能体如何判断任务完成？

## 考察重点

- 是否理解"任务完成"的多层含义：用户满意度、目标达成、可观测性
- 是否能说清楚显式停止条件和隐式停止条件的区别
- 是否了解 Claude Code 和 Codex 在停止条件上的具体实现
- 是否有处理"部分完成"和"模糊完成"的经验

## 核心结论

任务完成不是单一条件判断，而是一个**多维度评估**的过程。最可靠的系统是"让模型自己说完成了，然后让用户最终确认"。没有人比用户更知道任务是否真正完成。

## 任务完成的三层含义

### 第一层：执行层面

模型不再调用工具，只输出文本响应。

这是 Codex 的停止条件：当模型输出 `response.output_text.done` 而不是 `function_call` 时，认为执行循环可以结束。

```json
// Codex SSE 事件
data: {"type":"response.output_text.done","text":"任务已完成", ...}
```

但这只是**必要条件**，不是**充分条件**。模型可能：
- 误判任务完成，实际上还有步骤没做
- 输出了文本但没有真正执行用户想要的操作
- 输出了错误的结果

### 第二层：目标层面

实际达成了用户请求的目标。

这需要验证：
- 用户请求的"删除文件"，文件是否真的被删了
- 用户请求的"生成报告"，报告内容是否完整
- 用户请求的"修复 bug"，bug 是否真的修复了

**问题**：模型无法可靠地自我验证。验证需要：
1. 独立的检查机制（如测试套件）
2. 结构化的输出验证
3. 用户的最终确认

### 第三层：用户层面

用户认为任务完成了。

这意味着：
- 输出符合用户的预期格式
- 任何疑问都已被解答
- 用户没有进一步的问题或修改需求

## 显式停止条件 vs 隐式停止条件

### 显式停止条件

系统在代码中明确定义什么时候停止：

```typescript
// Claude Code 风格
if (state.turnCount > MAX_TURNS) break;
if (state.tokenBudget <= 0) break;
if (state.stopSignalReceived) break;

// Codex 风格
if (response.type === 'output_text.done') break;
```

优点：可预测、可控制、可调试
缺点：可能过于死板，无法处理边缘情况

### 隐式停止条件

依赖模型自己的能力判断什么时候停止：

```json
{
  "name": "task_complete",
  "arguments": {
    "summary": "完成了什么",
    "verification": "如何验证完成",
    "next_steps": "如果用户还有需求"
  }
}
```

优点：灵活，能处理复杂场景
缺点：不可靠，模型可能误判

### 实际做法：混合策略

```
显式条件作为安全网
    ↓
隐式条件作为主要判断
    ↓
用户确认作为最终验证
```

## Claude Code 的停止条件实现

### 每次循环结束时的检查

```typescript
// query.ts 中的停止检查
const shouldStop = () => {
  // 1. 用户主动停止
  if (state.stopSignalReceived) return true;

  // 2. 达到最大轮次
  if (state.turnCount >= state.maxTurns) return true;

  // 3. Token 预算耗尽
  if (state.tokenBudget <= 0) return true;

  // 4. Circuit breaker 触发
  if (state.hasAttemptedReactiveCompact && state.consecutiveFailures > 0) {
    return true;
  }

  // 5. 模型输出表示任务完成
  if (state.lastModelOutput.type === 'final_response') return true;

  return false;
};
```

### reactive compact 作为安全网

当检测到潜在问题时，Claude Code 会尝试压缩上下文：

```typescript
interface State {
  hasAttemptedReactiveCompact: boolean;
  // 如果已经尝试过压缩但仍然有问题，就强制停止
}
```

## 部分完成和模糊完成的处理

### 部分完成

当任务没有完全完成但必须停止时（资源耗尽、用户中断等）：

```json
{
  "status": "partial_complete",
  "completed_steps": [
    "分析了 SQL 查询",
    "识别出 3 个性能问题",
    "优化了第一个问题"
  ],
  "remaining_steps": [
    "优化第二个问题（索引）",
    "验证所有优化",
    "生成测试用例"
  ],
  "can_resume": true,
  "resume_from_step": 3
}
```

### 模糊完成

当任务"看起来"完成了，但不确定时：

```json
{
  "status": "uncertain",
  "model_believes": "已完成",
  "confidence": 0.7,
  "verification_needed": [
    "运行测试验证修复",
    "用户确认输出格式"
  ],
  "suggestion": "建议用户确认后再结束会话"
}
```

## 工程实践：设计可靠的完成判断

### 1. 多重验证机制

```typescript
async function isTaskComplete(state: AgentState, task: Task): Promise<VerifyResult> {
  const checks = [
    // 显式条件
    checkTurnLimit(state),
    checkTokenBudget(state),
    checkUserStop(state),

    // 目标验证
    checkGoalAchieved(state, task),
    checkOutputFormat(state, task),

    // 隐式条件
    checkModelConfidence(state),
  ];

  const results = await Promise.all(checks);
  const failedChecks = results.filter(r => !r.passed);

  return {
    complete: failedChecks.length === 0,
    confidence: results.filter(r => r.passed).length / results.length,
    failedChecks,
    suggestion: failedChecks.length > 0 ? failedChecks[0].suggestion : null
  };
}
```

### 2. 可恢复的状态保存

```typescript
interface Checkpoint {
  turnCount: number;
  completedSteps: Array<{action: string, result: any}>;
  currentPlan: string[];
  originalGoal: string;
  tokenBudget: number;
}

// 保存检查点，以便在中断后恢复
function saveCheckpoint(state: AgentState): Checkpoint {
  return {
    turnCount: state.turnCount,
    completedSteps: state.history,
    currentPlan: state.currentPlan,
    originalGoal: state.originalGoal,
    tokenBudget: state.tokenBudget
  };
}
```

### 3. 用户确认流程

```
系统判断任务完成
    ↓
展示完成摘要和验证结果
    ↓
用户确认 or 补充需求
    ↓
结束 or 继续执行
```

## 真实踩坑案例

### 案例：模型"假装"完成

某团队的智能体在执行数据导入任务时：
- 应该导入 1000 条数据
- 实际只导入了 10 条就返回"完成"
- 剩下的 990 条因为某种错误被静默跳过

模型在输出中自信地说"已完成"，但实际上大量数据丢失。

**问题根因**：没有独立的验证机制，完全相信模型输出。

**修复方案**：
1. 任务开始前记录预期结果（如记录数据条数）
2. 任务结束后独立验证（如查询数据库确认）
3. 不匹配时触发警告和重试
