# 智能体和检索增强生成有什么区别？

## 面试问题

智能体和检索增强生成有什么区别？

## 考察重点

- 是否理解 RAG 是"**增强输入**"，智能体是"**增强行动能力**"
- 是否能说清楚两者在架构上的位置：RAG 在输入端，智能体在执行端
- 是否了解两者可以结合使用：RAG-Agent
- 是否有实际项目中平衡 RAG 和 Agent 能力的经验

## 核心结论

RAG 和智能体解决不同层次的问题：RAG 解决"模型不知道什么"的问题（知识获取），智能体解决"模型不能做什么"的问题（行动执行）。RAG 让模型"读得多"，智能体让模型"做得多"。两者是正交的，可以叠加使用。

---

## RAG vs 智能体：解决的问题不同

### RAG：解决"模型不知道什么"

```
用户问："你们公司去年的营收是多少？"
  → 模型不知道（知识截止日期、隐私数据）
  → RAG 从数据库检索相关信息
  → 模型结合检索结果生成答案
```

RAG 的核心职责：**扩展模型的知识边界**

### 智能体：解决"模型不能做什么"

```
用户说："帮我把这份报告发给张总"
  → 模型知道怎么写邮件，但不知道怎么"发"
  → 智能体调用邮件工具
  → 执行发送操作
```

智能体的核心职责：**扩展模型的操作边界**

---

## 架构位置不同

### RAG 在输入端

```
用户输入 → 检索增强 → 模型推理 → 输出
              ↑
         RAG 在这里
         扩展模型的知识
```

RAG 优化的是"**模型知道什么**"：
- 检索相关文档
- 注入到上下文
- 模型基于更多知识回答

### 智能体在执行端

```
用户目标 → 理解 → 规划 → 工具调用 → 观察 → 决策继续/停止
                      ↑
              智能体在这里
              执行实际操作
```

智能体优化的是"**模型能做什么**"：
- 调用 API/数据库
- 执行实际操作
- 观察结果并调整

---

## 对比矩阵

| 维度 | RAG | 智能体 |
|-----|-----|--------|
| **解决的问题** | 模型知识不足 | 模型行动能力不足 |
| **架构位置** | 输入端 | 执行端 |
| **核心能力** | 检索 + 注入 | 规划 + 工具调用 |
| **调用外部系统** | 否（只读检索） | 是（读写操作） |
| **有状态** | 通常无状态 | 有状态（任务生命周期） |
| **失败模式** | 检索不到、噪声干扰 | 执行失败、循环失控 |
| **典型场景** | 知识问答、文档理解 | 自动化办公、代码助手 |

## 两者结合：RAG-Agent

在实际生产系统中，RAG 和智能体经常结合使用：

```
用户："查看这个项目的最新代码，找出问题并修复"

智能体接收任务
    ↓
RAG：检索项目文档、API 文档
    ↓
智能体理解代码结构
    ↓
RAG：检索相关的 issue 和 PR
    ↓
智能体定位问题
    ↓
智能体修复代码
    ↓
智能体运行测试验证
```

### RAG-Agent 的架构

```typescript
interface RAGAgentState {
  // RAG 部分
  retrievedDocs: Document[];
  contextWindow: string;

  // Agent 部分
  taskGoal: string;
  steps: Step[];
  tools: Tool[];

  // 两者结合
  knowledgeContext: string;  // RAG 注入的知识
  actionContext: string;    // Agent 的执行历史
}
```

### 什么时候需要 RAG + Agent

| 场景 | 是否需要结合 |
|-----|------------|
| 纯知识问答（FAQ） | 单独 RAG 即可 |
| 需要执行操作的问答 | Agent，需要时加 RAG |
| 复杂任务需要领域知识 | RAG + Agent |
| 代码生成和调试 | Agent，需要时 RAG 检索文档 |

## RAG 和工具调用的关系

### RAG 是"只读工具"

RAG 从外部知识库检索信息，本质上是一个**只读工具**：

```json
{
  "name": "retrieve",
  "description": "从知识库检索相关文档",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {"type": "string"}
    }
  }
}
```

### 工具调用是"读写操作"

智能体的工具可以是读写操作：

```json
[
  {
    "name": "read_file",
    "description": "读取文件",
    "parameters": {...}
  },
  {
    "name": "write_file",
    "description": "写入文件",
    "parameters": {...}
  },
  {
    "name": "send_email",
    "description": "发送邮件",
    "parameters": {...}
  }
]
```

### 工具调用的范围

| 工具类型 | 示例 | RAG 能做吗？ |
|---------|------|------------|
| **只读检索** | 搜索文档、查数据库 | 是（RAG 本身就是检索） |
| **写入操作** | 修改文件、发送消息 | 否 |
| **外部操作** | 调用 API、执行命令 | 否 |
| **状态改变** | 更新数据库、创建资源 | 否 |

## 工程实践：区分 RAG 和 Agent 的职责

### 1. 判断标准

问自己一个问题：**模型需要知道，还是需要做到？**

- 需要知道 → 用 RAG
- 需要做到 → 用 Agent
- 都需要 → RAG + Agent

### 2. 架构分层

```
第一层：RAG（知识层）
  - 提供领域知识
  - 检索文档、数据

第二层：Agent（行动层）
  - 调用工具执行
  - 管理任务状态
  - 处理异常恢复
```

### 3. 避免过度设计

| 场景 | 错误做法 | 正确做法 |
|-----|---------|---------|
| FAQ 问答 | 用 Agent 调用"回答问题"工具 | 单独用 RAG |
| 需要计算后回答 | RAG 返回数据，模型计算 | RAG 返回数据，模型计算 |
| 需要操作后回答 | Agent 执行操作，返回结果 | Agent 执行操作，返回结果 |

## 面试追问方向

1. **"你的智能体项目是如何决定哪些用 RAG、哪些用 Agent 的？"**

   考察候选人的架构设计能力。

2. **"RAG 检索到的信息和模型的上下文冲突时，你怎么处理？"**

   考察候选人对知识冲突处理的理解。

3. **"如果一个任务的每一步都需要先 RAG 再执行，你是设计成一个 Agent 还是两个 Agent 协作？"**

   考察候选人对 Agent 分工的理解。

4. **"RAG 和 Long Context Window（长上下文）哪个更好？为什么？"**

   考察候选人对两者权衡的理解。
