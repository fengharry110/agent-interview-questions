# 智能体为什么容易失控？

## 面试问题

智能体为什么容易失控？

## 考察重点

- 是否理解"失控"的本质：模型行为偏离预期但系统无法及时发现或干预
- 是否能说清楚失控的常见模式：无限循环、目标漂移、危险操作、上下文污染
- 是否了解 Claude Code 和 Codex 在防止失控方面的具体设计
- 是否有实际调试和修复失控问题的经验

## 核心结论

智能体失控的本质是**反馈闭环的缺失或失效**。当模型开始偏离预期行为时，如果没有有效的检测和干预机制，问题会像滚雪球一样越滚越大，直到资源耗尽或造成破环。防止失控不是给智能体更多"聪明"，而是给它更多"约束"。

## 失控的本质：反馈闭环失效

### 正常情况

```
用户目标 → 模型规划 → 执行动作 → 观察结果 → 验证是否符合目标 → 继续或停止
                                    ↓
                              ✓ 符合 → 继续
                              ✗ 不符合 → 调整或停止
```

### 失控情况

```
用户目标 → 模型规划 → 执行动作 → 观察结果 → 验证？
                                    ↓
                              ✗ 验证失效（假阳性/假阴性）
                                    ↓
                              继续执行 → 偏离越来越远 → 失控
```

失控的根本原因是"验证"环节出了问题——模型可能：
- 错误地认为自己在正确方向上
- 无法感知到偏离
- 有动机（模型倾向于"多做"）继续执行

## 失控的常见模式

### 模式一：无限循环

模型在同一个问题上反复执行相同的动作序列。

**典型案例**：
```
用户："帮我修这个 bug"
  → 智能体尝试修复
  → 运行测试，失败
  → 尝试另一种修复
  → 运行测试，失败
  → 尝试第三种修复
  → ... 循环 50 次后上下文耗尽
```

**原因**：
- 模型倾向于"继续尝试"而不是"承认失败"
- 没有检测"重复动作"的机制
- 停止条件设置不当

**Claude Code 的防护**：
- `consecutiveFailures` 计数器，连续失败 N 次后触发警告
- `turnCount` 上限，防止无限循环
- 工具执行结果的结构化记录，方便检测重复

### 模式二：目标漂移

模型在执行过程中逐渐偏离原始目标，转向其他任务。

**典型案例**：
```
用户："帮我优化这个查询性能"
  → 智能体开始分析 SQL
  → 发现索引问题，创建索引
  → 发现注释缺失，添加注释
  → 发现文档过时，更新文档
  → ... 30 步后，用户想要的是查询优化，结果在做代码规范
```

**原因**：
- 子目标侵蚀了主目标
- 模型把"相关"误认为"应该做"
- 缺乏周期性目标检查

**防护机制**：
- 上下文压缩时保留原始目标
- 周期性检查当前状态是否还在目标范围内
- 用户确认机制：执行关键步骤前暂停

### 模式三：危险操作

模型执行了不应该执行的有害操作。

**典型案例**：
```
用户："帮我清理一下项目中的临时文件"
  → 智能体执行 rm -rf node_modules/*
  → 哦，node_modules 不是临时文件，但已经被删了
```

**原因**：
- 模型对"临时文件"的理解和用户不同
- 缺乏危险操作检测
- 没有沙箱或备份机制

**Claude Code 的防护**：
- 7 种权限模式，危险操作需要确认
- ML 分类器检测危险命令
- 沙箱执行环境

### 模式四：上下文污染

错误信息或无关内容污染了上下文，影响后续决策。

**典型案例**：
```
用户问了一个关于 X 的问题
  → 智能体尝试解决方案 A，失败（包含大量错误日志）
  → 上下文现在充满了关于 A 为什么失败的日志
  → 智能体被这些失败信息误导，放弃本可以成功的 A
```

**原因**：
- 工具输出直接进入上下文，没有过滤
- 失败信息产生了锚定效应
- 缺乏上下文管理策略

## Claude Code 的失控防护设计

### 1. 状态机 + Circuit Breaker

```typescript
interface State {
  // Circuit Breaker
  consecutiveFailures: number;
  hasAttemptedReactiveCompact: boolean;

  // 限制器
  turnCount: number;
  maxTurns: number;
  tokenBudget: number;
  maxTokenBudget: number;
}
```

### 2. 权限分级

```typescript
enum PermissionMode {
  ACCEPT_ALL,        // 所有操作直接执行
  ACCEPT_SAFE,      // 安全操作直接执行，危险操作需确认
  DENY_ALL,         // 所有操作都需确认
  READ_ONLY,        // 只读操作
}
```

### 3. ML 分类器

Claude Code 使用机器学习模型判断操作的危险性，而非简单的关键词匹配。这可以检测：
- 潜在的有害命令
- 数据删除风险
- 安全漏洞利用尝试

## 工程实践：如何设计失控防护

### 1. 多层防护网

```
第一层：输入验证
  - 用户输入是否在预期范围内
  - 是否有明显的注入攻击

第二层：规划审核
  - 模型生成的计划是否合理
  - 是否有明显的危险步骤

第三层：执行监控
  - 工具输出是否异常
  - 执行时间是否超限

第四层：结果验证
  - 结果是否符合预期
  - 是否需要用户确认
```

### 2. 量化指标

| 指标 | 作用 | 阈值示例 |
|-----|------|---------|
| `turnCount` | 检测循环 | > 100 触发警告 |
| `consecutiveFailures` | 检测重复失败 | > 3 触发警告 |
| `tokenGrowthRate` | 检测上下文膨胀 | > 2x 触发压缩 |
| `goalRelevanceScore` | 检测目标漂移 | < 0.5 触发重审 |

### 3. 优雅降级

```
失控检测到
    ↓
警告 + 记录上下文
    ↓
尝试自动恢复（压缩上下文、重置状态）
    ↓
如果恢复失败，暂停执行，等待用户指令
```

## 真实踩坑案例

### 案例：无限循环导致 AWS 账单爆表

某团队用智能体处理数据清洗任务。智能体在某个边缘情况下陷入了无限循环：
- 每分钟调用 API 数十次
- 持续运行了 8 小时
- 账单从预期的 $10 变成了 $4,000

**根因**：
- 没有设置 `maxTurns` 或 `timeout`
- 没有监控 API 调用频率
- 没有及时发现异常

**修复方案**：
1. 强制设置 `maxSteps = 500`，超过后强制停止
2. 实现调用频率监控，异常时触发告警
3. 增加成本预算控制，超预算时暂停
