# 智能体的状态应该如何管理？

## 面试问题

智能体的状态应该如何管理？

## 考察重点

- 是否理解"状态"在智能体中的多层含义：对话状态、任务状态、系统状态
- 是否能说清楚状态持久化和状态恢复的实现方式
- 是否了解 Claude Code 的状态机设计（turnCount、tokenBudget 等）
- 是否有处理状态冲突、状态丢失等边界情况的经验

## 核心结论

智能体的状态管理是防止"失忆"和"重复"的关键。状态至少分为三层：**对话状态**（当前轮次）、**任务状态**（跨轮次）、**系统状态**（资源预算）。好的状态管理能让智能体在中断后无缝恢复，在长任务中保持目标一致。

## 状态的三个层次

### 第一层：对话状态（Conversation State）

当前对话轮次内的状态，类似传统聊天机器人的 session。

```typescript
interface ConversationState {
  conversationId: string;
  userMessage: string;
  modelResponse: string;
  // 仅限当前轮次
}
```

### 第二层：任务状态（Task State）

跨轮次保持的任务进度，这是智能体和聊天机器人的关键区别。

```typescript
interface TaskState {
  taskId: string;
  userGoal: string;           // 原始目标
  currentPlan: string[];      // 当前计划
  completedSteps: Step[];     // 已完成步骤
  currentStepIndex: number;   // 当前步骤
  context: Map<string, any>;  // 任务相关的上下文
  canResume: boolean;         // 是否可恢复
}
```

### 第三层：系统状态（System State）

资源预算和执行控制相关的状态。

```typescript
interface SystemState {
  turnCount: number;
  maxTurns: number;
  tokenBudget: number;
  maxTokenBudget: number;
  tokenUsage: { input: number; output: number };

  // Circuit Breaker
  consecutiveFailures: number;
  hasAttemptedReactiveCompact: boolean;

  // 控制
  stopSignalReceived: boolean;
  transition: { reason: string } | null;
}
```

### 状态持久化：检查点机制

```typescript
interface Checkpoint {
  state: State;
  timestamp: number;
  completedSteps: Array<{
    action: string;
    observation: string;
    success: boolean;
  }>;
}

// 保存检查点
function saveCheckpoint(state: State): Checkpoint {
  return {
    state: deepClone(state),
    timestamp: Date.now(),
    completedSteps: state.history.map(h => ({
      action: h.decision.type,
      observation: h.result,
      success: !h.error
    }))
  };
}

// 从检查点恢复
async function restoreFromCheckpoint(checkpoint: Checkpoint): Promise<State> {
  return {
    ...checkpoint.state,
    // 重置循环控制，允许重新开始
    turnCount: 0,
    stopSignalReceived: false,
    transition: null
  };
}
```

## 状态管理的常见问题

### 问题一：状态丢失

当智能体执行中断（网络故障、进程崩溃）时，如果状态没有持久化，需要从头开始。

**解决方案**：周期性保存检查点

```typescript
// 每 N 步保存一次检查点
if (state.turnCount % 10 === 0) {
  await saveCheckpoint(state);
}
```

### 问题二：状态冲突

多个并发任务可能修改同一份状态。

**解决方案**：乐观锁或版本号

```typescript
interface TaskState {
  version: number;  // 每次修改递增
}

async function updateState(taskId: string, updates: Partial<TaskState>) {
  const current = await getState(taskId);
  const result = await db.update(taskId, {
    ...updates,
    version: current.version + 1
  }, {
    where: { id: taskId, version: current.version }
  });

  if (result === 0) {
    throw new ConcurrencyError('State was modified by another process');
  }
}
```

### 问题三：状态膨胀

随着任务执行，状态不断增长，可能耗尽内存。

**解决方案**：状态压缩和摘要

```typescript
function compressState(state: TaskState): TaskState {
  return {
    ...state,
    // 将多个步骤摘要成一个
    completedSteps: summarizeSteps(state.completedSteps),
    // 清理不再需要的临时数据
    temporaryData: null
  };
}
```

## 工程实践：状态管理架构

### 1. 状态存储分层

```
┌─────────────────────────────────────────────────┐
│                 内存状态                          │
│    (快速访问，用于当前执行)                        │
│    - TaskState                                   │
│    - SystemState                                 │
└─────────────────────────────────────────────────┘
                    ↓ 定期持久化
┌─────────────────────────────────────────────────┐
│               Redis / 数据库                      │
│    (持久化，支持恢复)                              │
│    - Checkpoints                                 │
│    - 历史任务状态                                 │
└─────────────────────────────────────────────────┘
```

### 2. 状态更新模式

```typescript
class StateManager {
  private state: State;
  private dirty: boolean = false;

  update(updates: Partial<State>) {
    this.state = { ...this.state, ...updates };
    this.dirty = true;
  }

  async persist() {
    if (this.dirty) {
      await this.storage.save(this.state);
      this.dirty = false;
    }
  }

  // 定期持久化
  startPeriodicPersist(intervalMs: number) {
    setInterval(() => this.persist(), intervalMs);
  }
}
```

### 3. 状态恢复流程

```
任务中断
    ↓
检测到非正常退出（心跳超时、进程退出）
    ↓
从最近的检查点加载状态
    ↓
验证状态完整性（checksum、版本号）
    ↓
恢复执行（跳过已完成的步骤）
    ↓
继续 Agent Loop
```
