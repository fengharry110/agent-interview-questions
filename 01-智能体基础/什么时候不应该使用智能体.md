# 什么时候不应该使用智能体？

## 面试问题

什么时候不应该使用智能体？

## 考察重点

- 是否理解智能体不是万能的，有其适用范围
- 是否能说清楚什么时候该用规则引擎、工作流、或简单 API
- 是否了解智能体的成本和延迟问题
- 是否有实际项目中"不用智能体反而更好"的经历

## 核心结论

智能体是"高能力、高成本、高不确定性"的系统。应该避免在以下场景使用：流程固定且简单、需要精确控制成本和延迟、需要可预测的行为、或者只是跟风"用 AI"。能用规则和工作流解决的，不要用智能体。

---

## 什么时候不该用智能体

### 场景一：流程固定且简单

**问题**：智能体的灵活性是浪费。

```
例子：FAQ 问答
  - 用户问："你们公司地址在哪？"
  - 答案是固定的：直接查知识库返回

如果用智能体：
  - 调用模型 → 检索知识库 → 生成回答
  - 成本：0.01 元
  - 延迟：500ms

正确做法：
  - 关键词匹配 → 直接返回答案
  - 成本：0.0001 元
  - 延迟：10ms
```

**替代方案**：规则匹配、关键词检索、简单决策树

### 场景二：需要精确控制成本

**问题**：智能体的 token 消耗不可预测。

```
例子：批量处理 10000 条数据
  - 每条数据的复杂度不同
  - 有的可能 100 tokens，有的可能 5000 tokens
  - 最终成本完全不可预测

如果用智能体：
  - 总成本 = Σ(每条数据的 token 消耗 × 单价)
  - 可能超预算 10 倍

正确做法：
  - 用工作流定义固定的处理流程
  - 每个节点的成本可计算
  - 总成本可预测
```

**替代方案**：工作流 + 固定 LLM 节点

### 场景三：需要精确控制延迟

**问题**：模型推理延迟不可预测。

```
例子：实时客服
  - 用户期望 1 秒内响应
  - 模型推理可能需要 2-5 秒
  - 用户体验很差

如果用智能体：
  - 每个用户请求可能触发多次工具调用
  - 总延迟 = Σ(每次调用的延迟)
  - 可能达到 30 秒以上

正确做法：
  - 用规则引擎处理常见问题
  - 复杂问题才转人工
```

**替代方案**：规则引擎 + 人工兜底

### 场景四：需要可预测的行为

**问题**：智能体的输出有随机性。

```
例子：生成格式化报告
  - 要求：严格按照模板生成
  - 每次格式必须完全一致

如果用智能体：
  - 模型可能每次措辞略有不同
  - 可能"自由发挥"添加额外内容
  - 需要大量后处理来保证格式

正确做法：
  - 用模板引擎直接替换变量
  - 输出 100% 可预测
```

**替代方案**：模板引擎、规则生成

### 场景五：需要人工审核

**问题**：智能体的自主决策反而是负担。

```
例子：审核贷款申请
  - 每个申请都需要人工批准
  - 智能体不能自主决定通过/拒绝

如果用智能体：
  - 智能体分析后还是要等人工确认
  - 增加了复杂度，没有减少工作量

正确做法：
  - 智能体只做信息收集和初步分析
  - 人工做最终决策
```

**替代方案**：工作流 + 人工节点

### 场景六：任务非常简单

**问题**：杀鸡用牛刀。

```
例子：计算器功能
  - 用户说："1 + 1 等于多少？"
  - 直接计算即可，不需要模型

如果用智能体：
  - 模型处理 → 工具调用 → 返回
  - 浪费资源

正确做法：
  - 正则匹配 → 计算 → 返回
```

**替代方案**：正则匹配、简单函数

## 判断标准

### 什么时候用智能体

| 条件 | 说明 |
|-----|------|
| 流程不确定 | 无法预先定义所有可能的处理路径 |
| 需要多步推理 | 需要综合多种信息做决策 |
| 工具组合不确定 | 根据输入动态决定用哪些工具 |
| 输出格式灵活 | 不需要严格的结构化输出 |
| 探索性任务 | 需要边做边想 |

### 什么时候不用智能体

| 条件 | 说明 |
|-----|------|
| 流程固定 | 所有场景的处理路径都是预定义的 |
| 成本敏感 | 需要精确控制每次调用的成本 |
| 延迟敏感 | 需要毫秒级响应时间 |
| 可预测性要求高 | 输出必须严格一致 |
| 有人工审核 | 最终决策需要人来做 |
| 任务简单 | 用规则或简单函数就能解决 |

### 决策流程图

```
任务来了
    ↓
是固定流程吗？
    ├─ 是 → 用规则/工作流
    └─ 否 ↓
需要调用外部系统吗？
    ├─ 否 → 考虑用简单 LLM 调用
    └─ 是 ↓
工具组合是动态的吗？
    ├─ 否 → 用工作流 + LLM 节点
    └─ 是 ↓
需要多步推理/规划吗？
    ├─ 否 → 用工作流 + LLM 节点
    └─ 是 → 用智能体
```

## 成本和延迟对比

| 方案 | 单次成本 | P50 延迟 | P99 延迟 | 可预测性 |
|-----|---------|---------|---------|---------|
| 关键词匹配 | $0.00001 | 5ms | 20ms | 100% |
| 规则引擎 | $0.0001 | 10ms | 50ms | 100% |
| 工作流 + LLM | $0.001 | 200ms | 500ms | 90% |
| 智能体 | $0.01-0.5 | 1s | 30s | 50% |

## 面试追问方向

1. **"你最近做的项目中，有哪些原本想用智能体但后来改成其他方案的？为什么？"**

   考察候选人的实际经验和对技术选型的反思。

2. **"如果用户坚持要用智能体来解决一个明显不适合的场景，你会怎么说服他们？"**

   考察候选人的沟通能力和技术判断力。

3. **"智能体和工作流各适合什么场景？你如何在同一个系统中使用它们？"**

   考察候选人对混合架构的理解。

4. **"当智能体的成本超出预算时，你怎么优化？"**

   考察候选人对成本控制的理解。
