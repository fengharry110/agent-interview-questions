# 什么时候不应该使用 Agent？

## 面试问题

什么情况下不应该使用 Agent？

## 考察重点

- 是否有工程判断力
- 是否知道 Agent 会带来成本、延迟和不确定性
- 是否能根据任务复杂度选择方案

## 参考回答

当任务流程固定、规则明确、没有复杂决策时，通常不需要 Agent。比如简单表单校验、固定 SQL 查询、固定格式转换，用普通程序更稳定、更便宜、更容易测试。

如果任务对确定性、安全性、低延迟要求非常高，也要谨慎使用 Agent。Agent 的输出和工具选择具有一定不确定性，可能需要额外的权限控制、日志、人工确认和评测体系。

Agent 更适合目标明确但路径不固定的任务，例如代码修复、资料研究、多系统协作、复杂客服工单处理等。

## 追问方向

- 如何判断一个需求是否适合 Agent？
- Agent 和传统 workflow engine 如何选择？
- 如何降低 Agent 的不确定性？

## 学习建议

回答时要体现“能不用 Agent 就不用 Agent”的工程意识，这会让面试官觉得你不是只会追热点。
