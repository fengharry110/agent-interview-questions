# 多智能体如何共享上下文？

## 面试问题

多智能体如何共享上下文？

## 考察重点

- 是否理解多智能体系统中上下文共享的核心挑战（一致性、冗余、延迟）
- 是否能说清楚不同的共享策略（全量共享、按需共享、汇总共享）的优缺点
- 是否了解上下文压缩和摘要技术在共享场景中的应用
- 是否有解决过"上下文同步冲突"的实际经验

## 核心结论

多智能体的上下文共享核心是在"信息完整性"和"token 成本"之间做权衡：全量共享保证完整性但成本高，按需共享节省成本但可能遗漏关键信息，汇总共享最经济但信息损失最大；实际工程中通常采用混合策略。

## 上下文共享的三种模式

### 模式一：全量共享（Shared Global Context）

所有智能体共享同一个完整上下文。每个智能体都可以读写全局上下文。

```
用户输入 → [全局上下文] → Agent A 读取 → Agent A 写入
            → Agent B 读取 → Agent B 写入
            → Agent C 读取 → Agent C 写入
```

```python
class GlobalContext:
    def __init__(self):
        self.messages: list[Message] = []
        self.artifacts: dict[str, any] = {}
        self.lock = asyncio.Lock()

    async def read(self, agent_id: str) -> list[Message]:
        async with self.lock:
            return self.messages.copy()

    async def write(self, agent_id: str, message: Message):
        async with self.lock:
            self.messages.append(message)
            # 记录是谁写的，方便追踪
            message.author = agent_id
```

**优点**：信息完整、实现简单
**缺点**：上下文膨胀极快、每个智能体都要处理不相关的信息、token 成本爆炸

```
全局上下文增长曲线：
  Agent A 工作 → 上下文增加 5K tokens
  Agent B 工作 → 上下文再增加 5K tokens（包含 A 的全部）
  Agent C 工作 → 上下文再增加 5K tokens（包含 A+B 的全部）
  → 3 个智能体后，上下文已经 15K tokens
  → 10 个智能体后，上下文 > 50K tokens
  → 成本 O(n²)
```

### 模式二：按需共享（On-Demand Context）

智能体只维护自己的私有上下文，需要其他智能体的信息时主动请求。

```
Agent A 的私有上下文    Agent B 的私有上下文
    [私有信息 A]            [私有信息 B]
          |                       |
          └───── 请求/响应 ──────┘
                (只交换需要的)
```

```python
class OnDemandContext:
    def __init__(self, agent_id: str):
        self.agent_id = agent_id
        self.private_context: list[Message] = []
        self.shared_service: ContextService = ContextService()

    async def request_info(
        self, target_agent: str, query: str
    ) -> str:
        # 向特定智能体请求信息
        return await self.shared_service.query(
            requester=self.agent_id,
            target=target_agent,
            query=query
        )

    async def publish_info(self, key: str, value: any):
        # 发布信息到共享总线
        await self.shared_service.publish(
            agent_id=self.agent_id,
            key=key,
            value=value
        )

class ContextService:
    def __init__(self):
        self.published: dict[str, dict] = {}

    async def query(
        self, requester: str, target: str, query: str
    ) -> str:
        context = self.published.get(target, {})
        # 智能体 B 根据查询生成摘要回应
        response = await self.summarize_relevant(context, query)
        return response

    async def publish(self, agent_id: str, key: str, value: any):
        if agent_id not in self.published:
            self.published[agent_id] = {}
        self.published[agent_id][key] = value
```

**优点**：token 成本可控、每个智能体只关注相关信息
**缺点**：请求/响应的额外延迟、可能遗漏"你不知道你不知道"的信息

### 模式三：汇总共享（Summarized Context）

每个智能体把自己的工作结果汇总成摘要，共享给其他智能体。

```
Agent A 原始上下文 (10K tokens)
    ↓ 摘要
Agent A 摘要 (1K tokens) → 写入共享区域

Agent B 看到：
  Agent A 摘要 (1K) + 自己的上下文 (5K) = 6K
  而不是：
  Agent A 全部 (10K) + 自己的上下文 (5K) = 15K
```

```python
class SummarizedContext:
    def __init__(self):
        self.summaries: dict[str, Summary] = {}

    async def add_summary(self, agent_id: str, summary: Summary):
        self.summaries[agent_id] = summary

    async def get_context(self, agent_id: str) -> str:
        # 获取所有摘要（排除自己的）
        others_summaries = [
            s.content
            for aid, s in self.summaries.items()
            if aid != agent_id
        ]
        return "\n---\n".join(others_summaries)

class Summary:
    def __init__(self, content: str, metadata: dict):
        self.content = content
        self.metadata = metadata
        self.token_count = len(content) / 4
        self.timestamp = time.time()
```

**优点**：token 成本最低、易于扩展
**缺点**：摘要过程本身有信息损失、摘要的质量直接影响协作效果

## 上下文压缩策略对比

| 策略 | 信息保留率 | Token 节省 | 实现复杂度 | 适用场景 |
|------|-----------|-----------|-----------|---------|
| 全量共享 | 100% | 0% | 低 | 小规模（2-3 个智能体） |
| 选择性保留 | 80-90% | 50-70% | 中 | 中等规模 |
| 摘要压缩 | 60-80% | 70-90% | 高 | 大规模（10+ 个智能体） |
| 结构化提取 | 70-85% | 60-80% | 中 | 需要精确参数传递 |
| 按需获取 | 可变 | 80%+ | 高 | 智能体之间弱依赖 |

## 结构化上下文共享

除了文本摘要，还可以使用结构化数据格式共享。

### 共享数据格式

```typescript
interface SharedContext {
  // 全局任务信息
  task: {
    id: string;
    goal: string;
    status: 'planning' | 'executing' | 'reviewing' | 'done';
    progress: number;  // 0-100
  };

  // 制品仓库：智能体产生的产物
  artifacts: Record<string, Artifact>;

  // 决策记录：关键决策和理由
  decisions: Decision[];

  // 待处理事项
  action_items: ActionItem[];

  // 冲突记录
  conflicts: Conflict[];
}

interface Artifact {
  id: string;
  type: 'code' | 'doc' | 'data' | 'plan' | 'review';
  content: string;
  summary: string;  // 摘要版本
  produced_by: string;
  version: number;
  dependencies: string[];  // 依赖的其他 artifact
  timestamp: number;
}

interface Decision {
  id: string;
  made_by: string;
  description: string;
  rationale: string;
  alternatives: string[];
  timestamp: number;
}
```

### 共享上下文管理器实现

```python
class StructuredContextManager:
    def __init__(self):
        self.context = SharedContext()
        self.change_log = []

    async def update_artifact(
        self, agent_id: str, artifact: Artifact
    ):
        # 更新制品，如果是更新则版本号 +1
        existing = self.context.artifacts.get(artifact.id)
        if existing:
            artifact.version = existing.version + 1

        self.context.artifacts[artifact.id] = artifact
        self.change_log.append({
            "action": "update_artifact",
            "agent": agent_id,
            "artifact_id": artifact.id,
            "version": artifact.version
        })

    async def get_relevant_context(
        self, agent_id: str, task_type: str
    ) -> dict:
        # 根据智能体角色和任务类型，过滤出相关的上下文
        relevant = {
            "task": self.context.task,
            "artifacts": {},
            "decisions": []
        }

        # 只返回该智能体需要关注的制品
        for aid, artifact in self.context.artifacts.items():
            if self._is_relevant(artifact, agent_id, task_type):
                relevant["artifacts"][aid] = artifact.summary

        # 返回最近的关键决策
        relevant["decisions"] = self.context.decisions[-5:]

        return relevant

    def _is_relevant(
        self, artifact: Artifact,
        agent_id: str, task_type: str
    ) -> bool:
        # 简单规则：依赖当前任务的制品都相关
        if artifact.produced_by == agent_id:
            return False  # 不需要看自己的产出
        if artifact.type == task_type:
            return True
        return False
```

## 上下文同步的策略

### 策略一：写时同步（Synchronous Write）

智能体修改共享上下文时，立即通知所有相关方。

```python
class SyncWriter:
    async def write_and_notify(
        self, artifact: Artifact, subscribers: list[str]
    ):
        await self.store.save(artifact)
        # 同步通知所有订阅者
        for subscriber in subscribers:
            await self.notify(subscriber, artifact)
```

**适合**：关键信息必须及时同步
**不适合**：频繁更新的场景，通知风暴

### 策略二：定时同步（Periodic Sync）

智能体定期从共享上下文中拉取最新信息。

```python
class PeriodicSyncAgent(Agent):
    def __init__(self, sync_interval: int = 30):
        super().__init__()
        self.sync_interval = sync_interval  # seconds
        self.last_sync = 0

    async def ensure_context_fresh(self):
        now = time.time()
        if now - self.last_sync > self.sync_interval:
            await self.sync_from_shared()
            self.last_sync = now

    async def execute(self, step: TaskStep):
        # 执行前确保上下文是最新的
        await self.ensure_context_fresh()
        return await super().execute(step)
```

**适合**：对时效性要求不高的场景
**不适合**：需要实时协作的场景

### 策略三：事件驱动同步（Event-Driven Sync）

智能体只订阅自己感兴趣的事件类型，有事件发生时被动接收。

```python
class EventBus:
    def __init__(self):
        self.subscribers: dict[str, list[callable]] = {}

    def subscribe(self, event_type: str, callback: callable):
        if event_type not in self.subscribers:
            self.subscribers[event_type] = []
        self.subscribers[event_type].append(callback)

    async def emit(self, event_type: str, data: any):
        if event_type in self.subscribers:
            for callback in self.subscribers[event_type]:
                await callback(data)

class EventDrivenAgent(Agent):
    def __init__(self, event_bus: EventBus):
        super().__init__()
        self.event_bus = event_bus
        # 订阅感兴趣的事件
        self.event_bus.subscribe(
            f"artifact.{self.role}.updated",
            self.on_artifact_updated
        )
        self.event_bus.subscribe(
            "task.status_changed",
            self.on_task_status_changed
        )

    async def on_artifact_updated(self, artifact: Artifact):
        if self._is_relevant(artifact):
            self.context.add_artifact(artifact.summary)

    async def on_task_status_changed(self, status: dict):
        self.context.task_status = status
```

**适合**：复杂的协作场景、智能体之间有明确的依赖关系
**不适合**：简单场景，过度设计

## 上下文冲突处理

### 冲突类型

```
类型一：数据冲突
  Agent A 写入：数据库连接串 = "host1:3306"
  Agent B 写入：数据库连接串 = "host2:3306"
  → 谁是正确的？

类型二：状态冲突
  Agent A 标记任务状态：已完成
  Agent B 标记任务状态：未完成
  → 该信谁？

类型三：逻辑冲突
  Agent A 的代码：使用 PostgreSQL
  Agent B 的代码：使用 Redis
  → 如果 Agent A 依赖 Agent B 的 Redis 缓存
  → PostgreSQL 和 Redis 之间没有直接冲突
  → 但如果两者都做持久化存储，就需要协调
```

### 解决方案

```python
class ConflictResolver:
    def __init__(self, context: SharedContext):
        self.context = context

    async def resolve(self, conflict: Conflict) -> Resolution:
        # 策略1：时间戳优先
        if conflict.type == "data":
            return self.resolve_by_timestamp(conflict)

        # 策略2：权限等级优先
        if conflict.type == "state":
            return self.resolve_by_authority(conflict)

        # 策略3：人工介入
        if conflict.type == "logic":
            return await self.resolve_by_human(conflict)

    def resolve_by_timestamp(self, conflict: Conflict) -> Resolution:
        # 后写入的覆盖先写入的
        sorted_versions = sorted(
            conflict.versions,
            key=lambda v: v.timestamp,
            reverse=True
        )
        return Resolution(
            winner=sorted_versions[0].author,
            resolved_value=sorted_versions[0].content,
            strategy="last_write_wins"
        )

    def resolve_by_authority(self, conflict: Conflict) -> Resolution:
        # 按智能体优先级选择
        authority_order = [
            "orchestrator", "reviewer",
            "executor", "assistant"
        ]
        winner = min(
            conflict.versions,
            key=lambda v: authority_order.index(v.author)
            if v.author in authority_order
            else len(authority_order)
        )
        return Resolution(...)
```

## 工程实践：上下文共享架构

### 分层上下文设计

```
第0层：全局共享上下文
  - 任务目标、约束条件、最终输出规范
  - 所有智能体都可读

第1层：角色上下文
  - 当前角色的专用工具和知识
  - 只有该角色可读写

第2层：私有上下文
  - 智能体的中间产物和思考过程
  - 只有该智能体可读写

第3层：交换上下文
  - 智能体之间明确共享的信息
  - 按需读写
```

```python
class LayeredContext:
    def __init__(self, role: str):
        self.global_ctx = SharedContext()  # 第0层
        self.role_ctx = RoleContext(role)  # 第1层
        self.private_ctx = []             # 第2层
        self.exchange_ctx = {}            # 第3层

    def build_agent_context(self) -> str:
        # 按照优先级组装上下文
        layers = [
            ("Global Context", self.global_ctx.summarize()),
            ("Role Context", self.role_ctx.get_relevant()),
            ("Exchange Context", self.format_exchange()),
            ("Private Context", self.summarize_private()),
        ]
        return "\n\n".join(
            f"=== {name} ===\n{content}"
            for name, content in layers
            if content
        )
```

## 常见误区

- 默认所有智能体需要完整的全局上下文，忽视上下文膨胀问题
- 共享上下文时不加锁，导致多个智能体同时写入产生冲突
- 汇总共享时只用简单的"前几句截断"，而不是智能摘要
- 没有考虑共享上下文的版本管理，出问题时无法回溯
- 不同智能体使用不同格式写入上下文，导致其他智能体无法解析

## 追问方向

1. **"如果共享上下文太大超出模型限制，你的降级策略是什么？"**

   考察候选人对上下文超限的工程处理。

2. **"智能体 A 更新了共享上下文，智能体 B 还在用旧版本，怎么处理这种不一致？"**

   考察候选人对分布式系统一致性的理解。

3. **"你如何衡量共享策略的 token 效率？有没有量化指标？"**

   考察候选人是否有实际评测经验。

4. **"如果要求上下文 100% 不丢失信息，但成本必须控制在 2 倍以内，你怎么设计？"**

   考察候选人在约束条件下的设计能力。

## 学习建议

分别用全量共享、摘要共享、按需共享三种方式实现同一个多智能体协作任务（比如"写代码并审查"），记录每种方式的 token 消耗、信息保留率和最终质量。你会直观地看到：全量共享最简单但最贵，摘要共享最省钱但需要精心设计摘要策略，按需共享最灵活但实现最复杂。面试时能说出具体的数据对比会非常有说服力。
