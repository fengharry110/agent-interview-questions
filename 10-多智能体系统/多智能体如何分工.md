# 多智能体如何分工？

## 面试问题

多智能体如何分工？

## 考察重点

- 是否能给出明确的分工策略，而不是模糊地说"按功能分"
- 是否理解不同分工模式（按领域、按阶段、按角色）的适用场景
- 是否能说清楚分工的粒度如何把握
- 是否有实际设计和优化过分工方案的经验

## 核心结论

多智能体分工的核心原则是"**高内聚、低耦合**"：每个智能体的职责边界清晰，智能体之间的通信最小化；常见的分工模式有按领域分、按流程阶段分、按角色分三种，选择标准是看智能体之间是否需要频繁交换信息。

## 三种分工模式

### 1. 按领域分工（Domain-Based）

每个智能体负责一个专业领域，拥有该领域的全部工具和知识。

```
用户输入：开发一个电商后台管理系统
    ↓
Orchestrator 拆解任务
    ├── FrontendAgent（前端开发）
    │   ├── 工具：React, CSS, HTML
    │   └── 输出：前端代码
    ├── BackendAgent（后端开发）
    │   ├── 工具：Node.js, SQL, Redis
    │   └── 输出：API 代码
    ├── DBAgent（数据库）
    │   ├── 工具：Schema设计, Migration, Query优化
    │   └── 输出：数据库脚本
    ├── DevOpsAgent（部署）
    │   ├── 工具：Docker, CI/CD, Cloud
    │   └── 输出：部署配置
    └── TestAgent（测试）
        ├── 工具：UnitTest, E2E, Performance
        └── 输出：测试报告
```

**优点**：专业知识集中、工具选择准确率高、易于扩展
**缺点**：领域之间有重叠时容易冲突、需要协调者做大量上下文传递

```python
class DomainBasedSystem:
    def __init__(self):
        self.agents = {
            "frontend": FrontendAgent(),
            "backend": BackendAgent(),
            "database": DBAgent(),
            "devops": DevOpsAgent(),
            "test": TestAgent(),
        }

    async def develop_system(self, requirements: str) -> SystemOutput:
        # 规划阶段：Orchestrator 理解需求并分解
        plan = await self.orchestrator.plan(requirements)

        # 执行阶段：各领域并行或串行
        results = {}
        for phase in plan.phases:
            tasks = await self.orchestrator.distribute(
                phase, self.agents.keys()
            )

            # 并行执行独立任务
            parallel_tasks = [
                self.agents[name].execute(task)
                for name, task in tasks.items()
                if not task.has_dependency
            ]
            phase_results = await asyncio.gather(*parallel_tasks)
            results.update(phase_results)

        return await self.orchestrator.integrate(results)
```

### 2. 按流程阶段分工（Stage-Based）

每个智能体负责流程中的一个阶段，上一个智能体的输出是下一个智能体的输入。

```
用户输入 → PlanningAgent → ExecutionAgent → ReviewAgent → 输出
             (规划)          (执行)           (审查)

PlanningAgent:
  输入：用户需求
  输出：详细执行计划
  上下文：只关注"要做什么"

ExecutionAgent:
  输入：执行计划
  输出：执行结果
  上下文：只关注"怎么做"

ReviewAgent:
  输入：执行结果
  输出：质量评估 + 修改建议
  上下文：只关注"做得对不对"
```

**优点**：流程清晰、每个智能体上下文聚焦、易于调试
**缺点**：串行执行、无法并行、信息在阶段传递中可能丢失

```python
class StageBasedPipeline:
    def __init__(self):
        self.planner = PlanningAgent()
        self.executor = ExecutionAgent()
        self.reviewer = ReviewAgent()

    async def process(self, task: str) -> str:
        iteration = 0
        max_iterations = 3

        # 阶段一：规划
        plan = await self.planner.create_plan(task)

        while iteration < max_iterations:
            iteration += 1

            # 阶段二：执行
            result = await self.executor.execute(plan)

            # 阶段三：审查
            review = await self.reviewer.review(
                plan, result, task
            )

            if review.approved:
                return result

            # 根据审查反馈修改计划
            plan = await self.planner.revise_plan(
                plan, review.feedback
            )

        return result  # 达到最大迭代次数，直接返回
```

### 3. 按角色分工（Role-Based）

每个智能体扮演特定角色，角色之间通过协作完成任务。这种模式模仿人类团队协作。

```
         ┌──────────────┐
         │ ProductManager │
         │  (理解需求)     │
         └──────┬───────┘
                │
    ┌───────────┼───────────┐
    ↓           ↓           ↓
┌────────┐ ┌────────┐ ┌────────┐
│  Developer │ │  Designer  │ │  Tester   │
│ (实现)  │ │ (设计)  │ │ (验证)  │
└────┬───┘ └────┬───┘ └────┬───┘
     │          │          │
     └──────────┼──────────┘
                ↓
         ┌──────────────┐
         │  Tech Lead   │
         │  (技术决策)   │
         └──────────────┘
```

```python
class RoleBasedTeam:
    def __init__(self):
        self.pm = ProductManagerAgent()
        self.dev = DeveloperAgent()
        self.designer = DesignerAgent()
        self.tester = TesterAgent()
        self.tech_lead = TechLeadAgent()

    async def collaborate(self, task: str) -> dict:
        # PM 理解需求
        requirements = await self.pm.analyze_requirements(task)

        # 技术负责人评估
        tech_spec = await self.tech_lead.create_spec(requirements)

        # 设计和开发并行
        design, implementation = await asyncio.gather(
            self.designer.create_design(tech_spec),
            self.dev.implement(tech_spec)
        )

        # 验证
        test_result = await self.tester.test(
            implementation, requirements
        )

        return {
            "design": design,
            "implementation": implementation,
            "test_result": test_result,
            "technical_review": await self.tech_lead.review(
                implementation
            )
        }
```

## 分工的设计原则

### 原则一：职责单一

每个智能体只做一件事，且把这个事做到最好。

```
好的分工：
  SearchAgent → 只搜索
  ExtractAgent → 只提取信息
  WriteAgent → 只写内容

坏的分工：
  SmartAgent → 搜索 + 提取 + 写作 + 翻译 + ...
  → 工具太多，模型选择困难
  → 上下文太长，注意力分散
```

### 原则二：通信最小化

智能体之间的信息交换越少越好。

```python
# 不好的设计：频繁通信
class ChattyArchitecture:
    async def process(self, task: str):
        # Agent A 每做一步就广播
        for i, step in enumerate(steps):
            result = await agent_a.execute(step)
            # 每次都要通知其他智能体
            await self.broadcast(f"step {i} done: {result}")
            await agent_b.receive_update(result)

# 好的设计：只在必要时通信
class MinimalCommunication:
    async def process(self, task: str):
        agent_a_results = []
        for step in steps:
            result = await agent_a.execute(step)
            agent_a_results.append(result)

        # 一次批量传递
        await agent_b.receive_batch(agent_a_results)
```

### 原则三：边界清晰

智能体之间的职责边界不能有模糊地带。

```
模糊边界导致的问题：

  Agent A 认为"生成测试代码"属于自己
  Agent B 也认为"生成测试代码"属于自己
  → 两人都生成测试代码，互相覆盖
  → 或者两人都认为对方会做，测试代码没人写

解决方案：明确职责矩阵

| 任务类型       | 负责智能体 | 协作智能体 | 审批者 |
|--------------|-----------|-----------|-------|
| 搜索信息       | Search    | -         | -     |
| 代码生成       | Developer | Reviewer  | TechLead |
| 代码审查       | Reviewer  | -         | Developer |
| 测试生成       | Tester    | Developer | -     |
```

## 分工粒度的把握

### 粒度过粗的问题

```
一个智能体负责"数据处理"
  → 包含：数据清洗、转换、聚合、分析、可视化
  → 工具集：30+ 个数据处理工具
  → 问题：工具太多，智能体经常选错工具
  → 解决方案：拆成 DataCleaningAgent、DataAnalysisAgent 等
```

### 粒度过细的问题

```
拆成：
  DataReadingAgent（读数据）
  DataParsingAgent（解析数据）
  DataCleaningAgent（清洗数据）
  DataTransformingAgent（转换数据）
  DataAggregatingAgent（聚合数据）
  DataAnalyzingAgent（分析数据）
  DataVisualizingAgent（可视化）

问题：
  - 7 个智能体完成一个简单任务
  - 每个智能体之间需要频繁通信
  - 大部分时间花在"下一个智能体是谁"的决策上
  - 成本爆炸，延迟极高
```

### 粒度选择标准

```
| 因素 | 倾向粗粒度 | 倾向细粒度 |
|------|-----------|-----------|
| 工具数量 | < 10 个 | > 20 个 |
| 领域数 | 1-2 个 | 3+ 个 |
| 领域差异 | 差异小 | 差异大 |
| 并行需求 | 不需要 | 需要 |
| 调试成本 | 高 | 接受 |
| 团队经验 | 新手 | 有经验 |

经验法则：
  1. 初始从粗粒度开始
  2. 当工具选择准确率 < 85% 时考虑拆分
  3. 拆分后每个智能体工具数控制在 5-10 个
  4. 如果两个智能体频繁通信，考虑合并
```

## 动态分工 vs 静态分工

### 静态分工（固定角色）

```python
class StaticDivision:
    # 角色在系统初始化时固定
    def __init__(self):
        self.researcher = ResearchAgent()
        self.writer = WritingAgent()
        self.editor = EditingAgent()

    # 每个任务走固定流程
    async def handle(self, task: str):
        research = await self.researcher.do_research(task)
        draft = await self.writer.write(research)
        final = await self.editor.edit(draft)
        return final
```

**优点**：简单、可预测、易于调试
**缺点**：不灵活，某些任务可能不需要所有角色

### 动态分工（按需分配）

```python
class DynamicDivision:
    def __init__(self):
        self.agent_pool = {
            "research": ResearchAgent(),
            "writing": WritingAgent(),
            "editing": EditingAgent(),
            "coding": CodingAgent(),
            "testing": TestingAgent(),
            "design": DesignAgent(),
        }

    async def handle(self, task: str):
        # 先分析任务需要哪些技能
        required_skills = await self.skill_analyzer.analyze(task)

        # 动态组建团队
        team = {
            skill: self.agent_pool[skill]
            for skill in required_skills
        }

        # 动态生成协作流程
        workflow = await self.workflow_generator.generate(
            task, team.keys()
        )

        # 执行
        result = await self.execute_workflow(workflow, team)
        return result
```

**优点**：灵活、资源利用率高
**缺点**：复杂、需要额外的任务分析模块

## 工程实践：分工方案的设计流程

### 1. 任务分析

```python
class TaskAnalyzer:
    def analyze(self, task: str) -> TaskProfile:
        # 分析任务需要的技能
        skills = self.extract_skills(task)

        # 分析任务的并行潜力
        parallelism = self.assess_parallelism(task)

        # 分析任务的信息流
        info_flow = self.analyze_information_flow(task)

        return TaskProfile(
            required_skills=skills,
            parallelism=parallelism,
            dependency_graph=info_flow
        )
```

### 2. 角色设计

```python
class RoleDesigner:
    def design_roles(self, profile: TaskProfile) -> list[Role]:
        roles = []

        for skill_group in self.cluster_skills(profile.required_skills):
            role = Role(
                name=skill_group.name,
                tools=skill_group.tools,
                context_window=skill_group.estimated_context,
                expected_communication=self.estimate_comm(
                    skill_group, profile.dependency_graph
                )
            )
            roles.append(role)

        return roles
```

### 3. 验证和调优

```python
class DivisionValidator:
    async def validate(self, roles: list[Role], test_cases: list[str]):
        metrics = {
            "tool_accuracy": [],   # 工具选择准确率
            "comm_overhead": [],   # 通信开销
            "context_usage": [],   # 上下文利用率
            "error_rate": [],      # 错误率
        }

        for test in test_cases:
            result = await self.run_test(roles, test)
            metrics["tool_accuracy"].append(
                result.tool_selection_accuracy
            )
            metrics["comm_overhead"].append(
                result.communication_tokens / result.total_tokens
            )

        return metrics
```

## 常见误区

- 一开始就设计非常精细的分工，没有先验证粗粒度是否够用
- 分工时只考虑功能，不考虑智能体之间的信息依赖强度
- 让一个智能体既做规划又做执行又做审查，违反了职责单一原则
- 分工粒度过细导致"通信开销 > 执行开销"
- 静态分工应对所有任务，忽略了不同任务需要不同分工模式

## 追问方向

1. **"你的分工方案是静态的还是动态的？怎么处理临时出现的任务？"**

   考察候选人对系统灵活性的考虑。

2. **"分工太粗工具选择不准，分工太细通信开销大，你怎么平衡？"**

   考察候选人对 trade-off 的理解。

3. **"如果 A 智能体需要频繁调用 B 智能体的结果，你会直接合并还是保持分离？"**

   考察候选人对模块聚合度的判断。

4. **"分工之后，如何保证每个智能体都在正确的上下文里工作？"**

   考察候选人对上下文管理的理解。

## 学习建议

选一个你熟悉的复杂任务（比如"写一个技术方案文档"），尝试用不同的分工方式实现：先按领域（调研、写作、审校），再按阶段（草稿、修改、定稿），再按角色（作者、审稿人、批准人）。比较三种方式的成本、质量和延迟差异。在实践中你会深刻理解：分工方式的选择直接影响系统的成败。
