# 规划者和执行者和审查者架构有什么优缺点？

## 面试问题

规划者和执行者和审查者架构有什么优缺点？

## 考察重点

- 是否理解 Planner-Executor-Critic 是"分工-执行-验证"三元组在智能体系统中的映射
- 是否能说清楚三种角色的职责边界和交互关系
- 是否了解这种架构的典型失败模式
- 是否有实际实现或优化过这种架构的经验

## 核心结论

Planner-Executor-Critic 架构通过将"思考"（规划）、"行动"（执行）和"反思"（审查）分离到不同智能体，实现了关注点分离和专业化，但引入了串行延迟、信息传递损耗和角色间推诿三个核心缺陷。

---

## 架构定义

```
用户请求
    ↓
┌──────────┐
│  Planner │  负责：拆解任务、制定步骤、定义成功标准
│  规划者   │  关注："要做什么"和"做到什么程度算好"
└────┬─────┘
     ↓ 执行计划
┌──────────┐
│ Executor │  负责：按计划执行、调用工具、产生输出
│  执行者   │  关注："怎么做"和"做出来"
└────┬─────┘
     ↓ 执行结果
┌──────────┐
│  Critic  │  负责：评估结果、检查质量、提出改进建议
│  审查者   │  关注："做得对不对"和"怎么改进"
└────┬─────┘
     ↓ 通过/不通过
  反馈循环
```

## 角色职责详解

### Planner（规划者）

```python
class PlannerAgent:
    def __init__(self):
        self.planning_tools = [
            decompose_task,
            identify_dependencies,
            estimate_effort,
            define_acceptance_criteria
        ]

    async def create_plan(self, task: str) -> Plan:
        # 1. 理解任务
        task_analysis = await self.analyze_task(task)

        # 2. 拆解子任务
        subtasks = await self.decompose(task_analysis)

        # 3. 识别依赖关系
        dependencies = await self.identify_deps(subtasks)

        # 4. 排序和优先级
        ordered = await self.order_subtasks(
            subtasks, dependencies
        )

        # 5. 定义成功标准
        criteria = await self.define_criteria(ordered)

        return Plan(
            steps=ordered,
            dependencies=dependencies,
            success_criteria=criteria
        )
```

**关注点**：What（做什么）、Why（为什么做）、When（什么时候做）
**不需要关注**：How（具体怎么做）——那是 Executor 的事

### Executor（执行者）

```python
class ExecutorAgent:
    def __init__(self):
        self.execution_tools = [
            read_file, write_file, call_api,
            run_code, search_web, query_db
        ]

    async def execute(self, plan: Plan) -> ExecutionResult:
        results = []

        for step in plan.steps:
            # 选择工具
            tool = await self.select_tool(step)

            # 执行
            try:
                result = await tool.execute(step.params)
                results.append(StepResult(
                    step_id=step.id,
                    success=True,
                    output=result
                ))
            except Exception as e:
                results.append(StepResult(
                    step_id=step.id,
                    success=False,
                    error=str(e)
                ))

                # 尝试恢复
                recovered = await self.recover(step, e)
                if recovered:
                    results[-1] = recovered

        return ExecutionResult(
            step_results=results,
            summary=await self.summarize(results)
        )
```

**关注点**：How（怎么做）、Tool（用什么工具）、Result（产出了什么）
**不需要关注**：Why（为什么这么做）——那是 Planner 的职责

### Critic（审查者）

```python
class CriticAgent:
    def __init__(self):
        self.review_tools = [
            check_quality,
            check_consistency,
            check_completeness,
            check_security,
            check_performance
        ]

    async def review(
        self, plan: Plan, result: ExecutionResult
    ) -> ReviewResult:
        issues = []

        # 1. 检查是否满足计划要求
        for criterion in plan.success_criteria:
            passed = await self.check_criterion(
                criterion, result
            )
            if not passed:
                issues.append(Issue(
                    type="criterion_not_met",
                    description=f"Failed: {criterion}",
                    suggestion=self.suggest_improvement(criterion)
                ))

        # 2. 检查质量
        quality_issues = await self.check_quality(result)
        issues.extend(quality_issues)

        # 3. 检查安全性
        security_issues = await self.check_security(result)
        issues.extend(security_issues)

        return ReviewResult(
            passed=len(issues) == 0,
            issues=issues,
            score=self.calculate_score(issues)
        )
```

**关注点**：Check（检查什么）、Improve（怎么改进）、Pass/Fail（是否通过）
**不需要关注**：Do（怎么做）——那是 Executor 的事

## 优点

### 1. 关注点分离（Separation of Concerns）

每个智能体只关注自己的领域，不会被其他职责干扰。

```
Planner 不需要关心：
  - 用什么 API 实现
  - 代码性能怎么样
  - 结果是否符合安全规范

Executor 不需要关心：
  - 为什么要这样做
  - 其他步骤是什么
  - 最终目标是什么

Critic 不需要关心：
  - 怎么做出来的
  - 为什么这样设计
  - 用了什么工具
```

**收益**：每个智能体的上下文更小、更聚焦，工具选择准确率更高。

### 2. 质量反馈闭环

```
Planner → Executor → Critic → (反馈) → Executor(改进) → Critic(再审查)
                        ↓ (通过)
                      输出
```

天然的 PDCA 循环（Plan-Do-Check-Act），每次迭代质量都在提升。

```python
class QualityLoop:
    async def run_with_feedback(
        self, task: str, max_iterations: int = 3
    ) -> str:
        plan = await self.planner.create_plan(task)

        for i in range(max_iterations):
            result = await self.executor.execute(plan)
            review = await self.critic.review(plan, result)

            if review.passed:
                return result

            # 根据审查意见改进
            plan = await self.planner.revise_plan(
                plan, review.issues
            )

        # 最后一次尝试，Critic 宽松通过
        result = await self.executor.execute(plan)
        return result
```

### 3. 角色可替换

每个角色可以独立升级或替换，不影响其他角色。

```python
# 可以独立升级 Critic 而不影响 Planner 和 Executor
weak_critic = SimpleCriticAgent()
strong_critic = GPT4CriticAgent()

# 也可以使用不同的模型
fast_executor = GPT35ExecutorAgent()    # 快速但质量一般
accurate_executor = GPT4ExecutorAgent()  # 慢但质量更高
```

### 4. 职责明确，易于调试

```
问题："最终结果质量差"
排查方向：
  - 是 Planner 计划不合理？
  - 还是 Executor 执行错误？
  - 还是 Critic 没有发现关键问题？

每个角色有独立的日志和指标：
  Planner 指标：计划覆盖率、步骤完整性
  Executor 指标：工具调用成功率、错误率
  Critic 指标：问题检出率、误报率
```

## 缺点

### 1. 串行延迟

```
Planner (2s) → Executor (5s) → Critic (2s) = 9s
如果 Executor 需要迭代 3 次：
  Planner (2s) + Executor (5s) + Critic (2s) = 9s
  → 发现不通过 → Executor (5s) + Critic (2s) = 7s
  → 发现不通过 → Executor (5s) + Critic (2s) = 7s
总时间：9s + 7s + 7s = 23s
```

完全串行，无法并行。每一步都必须等上一步完成。

### 2. 信息传递损耗

```
Planner 的输出 → Executor
  "实现一个用户登录功能"
  Executor 理解："做一个登录页面"

Planner 实际想的是：
  "实现一个完整的用户认证系统，包括登录、注册、密码重置、OAuth"

信息在传递中丢失。
Executor 只能基于 Planner 的输出来理解任务。
Critic 也只能基于 Executor 的输出来判断质量。
```

```python
# 信息传递损耗示例
class InfoLossExample:
    async def run(self, task: str):
        # Planner 有完整理解
        plan = await self.planner.create_plan(task)
        # plan 中包含 Planner 所有的思考（10K tokens）

        # 传递给 Executor 的是计划摘要（2K tokens）
        # 损失了 80% 的上下文信息
        result = await self.executor.execute(
            plan.summary  # 只有摘要
        )

        # Critic 看到的是 Executor 的输出（1K tokens）
        # 加上计划摘要（2K tokens）
        # 对比原始的 10K tokens，损失了大量上下文
```

### 3. 角色间推诿

```
低质量输出时的责任归属：

Planner 说："我的计划没问题，是 Executor 执行偏差"
Executor 说："我严格按计划执行，是 Planner 计划不完善"
Critic 说："我发现了问题，但 Planner 和 Executor 没改好"

→ 没有明确的"第一责任人"
→ 改进时不知道从哪里下手
```

### 4. Critic 可能成为瓶颈

```
场景一：Critic 过于严格
  质量的 80% 已经够好，但 Critic 要求 100%
  → 需要额外 2-3 次迭代
  → 成本增加 200%，质量提升 5%

场景二：Critic 过于宽松
  质量只有 60%，但 Critic 说"通过"
  → 用户收到低质量输出
  → 失去对系统的信任

场景三：Critic 本身能力不足
  Critic 检查 SQL 注入漏洞
  但 Critic 不了解安全最佳实践
  → 放行了不安全代码
```

## 工程实践：优化 P-E-C 架构

### 1. 并行 Critic

```python
class ParallelCritic:
    def __init__(self):
        self.critics = [
            QualityCritic(),       # 质量审查
            SecurityCritic(),      # 安全审查
            ConsistencyCritic(),   # 一致性审查
            PerformanceCritic(),   # 性能审查
        ]

    async def review(
        self, plan: Plan, result: ExecutionResult
    ) -> CompositeReview:
        reviews = await asyncio.gather(
            *[critic.review(plan, result) for critic in self.critics]
        )

        return CompositeReview(
            overall_passed=all(r.passed for r in reviews),
            issues=[issue for r in reviews for issue in r.issues],
            scores={c.__class__.__name__: r.score
                    for c, r in zip(self.critics, reviews)}
        )
```

### 2. 自适应迭代次数

```python
class AdaptiveIteration:
    async def execute(
        self, task: str, quality_target: float = 0.9
    ) -> str:
        plan = await self.planner.create_plan(task)
        iteration = 0
        best_result = None
        best_score = 0

        while iteration < self.max_iterations:
            result = await self.executor.execute(plan)
            review = await self.critic.review(plan, result)

            if review.score >= quality_target:
                return result

            if review.score > best_score:
                best_score = review.score
                best_result = result

            # 如果改进幅度太小，提前终止
            if iteration > 0:
                improvement = review.score - prev_score
                if improvement < 0.02:
                    return best_result

            prev_score = review.score
            plan = await self.planner.revise_plan(
                plan, review.issues
            )
            iteration += 1

        return best_result
```

### 3. 角色职责重叠消除

```python
# 不清晰的责任边界 → 推诿
class BlurryPEC:
    async def run(self, task: str):
        plan = await self.planner.create_plan(task)
        # Executor 怀疑计划不合理 → 自己修改计划
        # → 越权了

# 清晰的责任边界 → 可追责
class ClearPEC:
    async def run(self, task: str):
        plan = await self.planner.create_plan(task)
        # Executor 发现计划不合理
        if self.executor.has_concerns(plan):
            # 不能自己改，必须反馈给 Planner
            plan = await self.planner.revise_plan(
                plan, self.executor.concerns
            )
            # 责任还在 Planner
```

## 常见误区

- 认为 Critic 必须由更强的模型担任，实际上 Critic 需要的是"检查能力"而非"生成能力"
- Planner 和 Executor 之间传递计划时只用自然语言，没有结构化格式，导致理解偏差
- 让 Critic 同时负责"发现问题"和"提出改进方案"，混淆了审查和规划职责
- 迭代没有上限，认为"多检几遍总是好的"，忽略了边际效益递减
- 三个角色使用同一个 prompt 模板，没有针对每个角色专门优化

## 追问方向

1. **"你的 Critic 怎么判断'通过'vs'不通过'？阈值怎么定的？"**

   考察候选人对质量标准的定义能力。

2. **"如果 Planner 和 Critic 都认为结果不合格，但 Executor 认为是自己能力上限了，你怎么处理？"**

   考察候选人对角色冲突的处理经验。

3. **"这种架构怎么扩展到更多角色？比如加一个 Tester？"**

   考察候选人对架构扩展的理解。

4. **"Planner 和 Critic 使用同一个模型吗？为什么？"**

   考察候选人对模型选择和角色匹配的思考。

## 学习建议

实现一个最小的 P-E-C 系统，任务定为"写一个 Python 函数并审查"。先让 Planner 写计划、Executor 写代码、Critic 检查代码质量。尝试不同的迭代策略（固定次数、自适应、单次通过），记录每种策略下的质量分数、成本和延迟。面试时能结合实验数据讨论优缺点会非常有深度。
