# 单智能体和多智能体如何选择？

## 面试问题

单智能体和多智能体如何选择？

## 考察重点

- 是否能给出明确的选型标准，而不是模糊地说"看情况"
- 是否理解单智能体适合什么场景、多智能体适合什么场景
- 是否能分析"从单智能体演进到多智能体"的触发条件
- 是否有实际切换过架构的经验，知道切换的成本和收益

## 核心结论

单智能体适合任务边界清晰、工具集固定、串行执行的场景；多智能体适合跨领域、需并行、需专业分工的场景；选择的关键是**评估复杂度和协作收益是否超过协调成本**。

---

## 选型决策矩阵

| 判断维度 | 倾向单智能体 | 倾向多智能体 |
|---------|------------|------------|
| **任务领域数** | 1-2 个 | 3 个以上 |
| **任务间依赖** | 强依赖、串行 | 弱依赖、可并行 |
| **工具集规模** | < 20 个工具 | > 20 个工具 |
| **所需专业知识** | 通用 | 多专业领域 |
| **对容错要求** | 容忍中断 | 不能单点故障 |
| **团队维护能力** | 小团队 | 有足够工程资源 |
| **成本敏感度** | 高 | 可接受较高成本 |
| **延迟要求** | 实时（< 2s） | 可接受异步（> 5s） |

---

## 决策流程

```
用户请求到达
    ↓
┌─────────────────────────────────┐
│ 任务复杂度评估                    │
│ - 需要几个领域的专业知识？         │
│ - 最多需要多少步？                │
│ - 是否有天然的子任务边界？         │
└─────────────────────────────────┘
    ↓
┌─────────────────────────────────┐
│ 工具集复杂度评估                  │
│ - 需要调用多少种外部服务？         │
│ - 工具是否需要特定领域的参数知识？  │
│ - 工具之间是否有访问权限差异？      │
└─────────────────────────────────┘
    ↓
┌─────────────────────────────────┐
│ 成本收益分析                      │
│ - 多智能体增加的延迟是否可接受？    │
│ - 多支付的 token 是否有 ROI？     │
│ - 增加的系统复杂度是否能被团队维护？│
└─────────────────────────────────┘
    ↓
    选择单智能体 / 多智能体
```

## 单智能体优先的场景

### 1. 搜索问答类任务

用户问一个问题，智能体去检索并回答。一般 3-5 步内完成。

```python
# 单智能体足够了
class SimpleSearchAgent:
    async def answer(self, question: str) -> str:
        context = await self.retriever.search(question)
        return await self.llm.generate(context)
```

**不需要多智能体的原因**：搜索 → 阅读 → 回答，这是一个线性流程，一个智能体完全可以处理。

### 2. 工具调用简单的任务

只用 1-2 个工具，参数简单。

```python
class WeatherAgent:
    tools = [get_weather, get_forecast]

    async def handle(self, query: str) -> str:
        intent = self.classify(query)
        if intent == "current":
            return await get_weather(**self.extract_params(query))
        return await get_forecast(**self.extract_params(query))
```

**不需要多智能体的原因**：工具少，调用逻辑简单，多智能体反而增加不必要的通信开销。

### 3. 用户意图明确的任务

用户一次性把需求说清楚了，不需要多轮澄清。

```
用户："帮我查一下2024年Q3的销售数据，按地区汇总"
  → 单智能体：理解意图 → 查数据库 → 汇总 → 返回
  → 如果换成多智能体：
    Agent A 理解 → 传给 Agent B 查询 → 传给 Agent C 汇总
    → 多了 3 次模型调用，质量没提升
    → 纯粹浪费成本
```

## 多智能体必选的场景

### 1. 跨领域专家协作

任务涉及多个专业领域，每个领域需要不同的知识背景。

```python
class MedicalTeamSystem:
    def __init__(self):
        self.agents = {
            "diagnosis": DiagnosisAgent(),    # 诊断专家
            "pharmacy": PharmacyAgent(),      # 药剂专家
            "surgery": SurgeryAgent(),        # 外科专家
            "radiology": RadiologyAgent(),    # 影像专家
        }

    async def consult(self, patient_case: dict) -> TreatmentPlan:
        # 每个专家从自己专业角度分析
        opinions = await asyncio.gather(
            self.agents["diagnosis"].analyze(patient_case),
            self.agents["radiology"].analyze(patient_case),
        )

        # 讨论治疗方案
        plan = await self.agents["diagnosis"].propose(opinions)
        pharmacy_review = await self.agents["pharmacy"].review(plan)
        surgery_review = await self.agents["surgery"].review(plan)

        return self.synthesize(plan, pharmacy_review, surgery_review)
```

**必须用多智能体的原因**：一个智能体不可能同时精通诊断、药理、外科和影像。

### 2. 工具集巨大且分散

工具超过 20 个，而且分属不同领域，一个智能体无法合理选择。

```
工具集:
  数据库相关的 10 个工具（SQL、NoSQL、Redis...）
  API 相关的 8 个工具（REST、GraphQL、WebSocket...）
  文件相关的 6 个工具（read、write、parse、convert...）
  外部服务 5 个工具（发送邮件、短信、推送...）

问题：一个智能体面对 29 个工具 → 选择困难
      给每个智能体 5-10 个工具 → 选择准确率大幅提升
```

```python
# 多智能体分角色管理工具
class DataAgent:
    tools = [query_sql, query_nosql, query_redis, read_csv, ...]

class APIAgent:
    tools = [call_rest, call_graphql, parse_response, ...]

class NotificationAgent:
    tools = [send_email, send_sms, send_push, ...]
```

### 3. 需要人工审核的流程

部分步骤必须有人参与审批，需要把审批逻辑封装到特定智能体中。

```python
class ApprovalAgent:
    def __init__(self):
        self.pending_approvals = asyncio.Queue()

    async def request_approval(self, action: Action) -> ApprovalResult:
        # 发送审批请求给人工
        approval = await self.notify_human(action)

        # 等待审批结果
        result = await self.pending_approvals.get()

        if result.approved:
            return ApprovalResult(
                status="approved",
                executor=action.assignee,
                approval_id=result.id
            )
        else:
            return ApprovalResult(
                status="rejected",
                reason=result.reason
            )
```

## 从单智能体演进到多智能体的触发条件

### 条件一：工具选择准确率下降

```
观测指标：工具选择的准确率
    初始：95%（10 个工具）
    当前：80%（增加到 25 个工具）
    阈值：< 85%
    → 考虑拆分工具集给不同智能体
```

### 条件二：单次任务 token 消耗过大

```
观测指标：平均每次任务的 token 消耗
    问题：上下文太长，模型注意力分散
    方案：拆成多个智能体，每个只维护自己相关的上下文
    → 总 token 可能增加，但每个智能体的上下文质量更高
```

### 条件三：需要并行执行

```
观测指标：任务执行时间
    当前：串行执行耗时 30s
    目标：要求在 10s 内返回
    → 必须拆成可并行的子任务
```

### 条件四：错误类型需要隔离

```
问题：一个智能体的错误会影响整个系统
    数据查询出错 → 后续步骤全部基于错误数据
    解决方案：查询、分析、生成分别用不同智能体
    → 查询出错时分析智能体可以请求重试
```

## 工程实践：渐进式演进

### 阶段一：单智能体

```python
class Agent:
    tools = ToolRegistry()
    context = ContextManager()
    planner = PlanningModule()

    async def run(self, task: str) -> str:
        plan = await self.planner.plan(task)
        for step in plan:
            tool = self.tools.select(step)
            context = self.context.get()
            result = await tool.run(step, context)
            self.context.add(result)
        return self.planner.synthesize(self.context)
```

### 阶段二：单智能体 + 子路由

```python
class RouterAgent(Agent):
    def __init__(self):
        super().__init__()
        self.sub_agents = {
            "data": DataAgent(),
            "api": APIAgent(),
            "file": FileAgent(),
        }

    async def run(self, task: str) -> str:
        # 先自己尝试
        plan = await self.planner.plan(task)
        for step in plan:
            if self.needs_domain_expert(step):
                # 路由给专业智能体
                sub = self.sub_agents[step.domain]
                result = await sub.execute(step)
            else:
                result = await self.execute_step(step)
        return self.synthesize()
```

### 阶段三：完整多智能体

```python
class Orchestrator:
    def __init__(self):
        self.planner = PlannerAgent()
        self.executors = {
            "data": DataAgent(),
            "api": APIAgent(),
            "code": CodeAgent(),
        }
        self.critic = CriticAgent()
        self.state = SharedState()

    async def run(self, task: str) -> str:
        plan = await self.planner.create_plan(task)
        for step in plan:
            result = await self.executors[step.type].execute(step)
            review = await self.critic.review(result)
            if not review.passed:
                result = await self.retry_with_feedback(
                    step, review.feedback
                )
            self.state.add_result(step, result)
        return await self.planner.synthesize(self.state)
```

## 成本对比分析

| 指标 | 单智能体 | 多智能体 | 差异 |
|------|---------|---------|------|
| 模型调用次数 | 5-10 次 | 10-30 次 | 2-3x |
| Token 消耗 | 10-20K | 30-60K | 2-3x |
| 延迟 | 2-5s | 5-15s | 2-3x |
| 工具选择准确率 | 85% | 95% | +10% |
| 任务完成率 | 75% | 90% | +15% |
| 系统复杂度 | 低 | 高 | 显著增加 |

**决策原则**：只有当多智能体带来的质量提升（完成率、准确率）在业务上能转化为可量化的收益，且覆盖了 2-3 倍的成本增加时，才值得切换。

## 常见误区

- 一开始就设计多智能体系统，而不知道单智能体能否先解决问题
- 认为多智能体等于"多个模型并行"导致更好，忽略了协调失败的场景
- 低估了从单智能体切换到多智能体的工程成本，包括通信、状态同步、调试等
- 没有考虑"一个超级智能体"方案：用更强的模型 + 更好的 prompt 是否已经够用
- 因为觉得"多智能体更高级"而强行使用，没有业务收益支撑

## 追问方向

1. **"你说的这个场景，如果用 GPT-4 单智能体和用 3 个 GPT-3.5 多智能体，哪个更好？"**

   考察候选人对模型能力与架构设计的权衡理解。

2. **"你的选型标准有没有量化数据支撑？比如准确率提升多少才值得增加成本？"**

   考察候选人是否做过实际的对比评测。

3. **"如果现在系统是单智能体，要演进到多智能体，你的迁移策略是什么？"**

   考察候选人的工程落地能力和渐进式思维。

4. **"有没有场景是单智能体做不到但多智能体一定能做到的？"**

   考察候选人对智能体系统能力边界的认知。

## 学习建议

找 3 个不同类型的任务（简单问答、跨领域分析、代码审查），分别用单智能体和多智能体实现，记录成本、延迟、完成率。通过实际数据来建立你对选型判断的第一手经验。面试时能说出"在我的实验中，某类任务单智能体的准确率是 X，多智能体是 Y，成本增加 Z 倍"会非常有说服力。
