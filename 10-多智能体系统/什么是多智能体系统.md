# 什么是多智能体系统？

## 面试问题

什么是多智能体系统？

## 考察重点

- 是否理解多智能体系统是"多个智能体协作解决单一智能体难以处理的复杂任务"
- 是否能说清楚多智能体系统比单智能体多了哪些关键组件（协调、通信、冲突解决）
- 是否了解主流的多智能体架构模式：集中式、分布式、混合式
- 是否有实际构建或使用过多智能体系统的经验

## 核心结论

多智能体系统是由多个自主智能体通过通信、协调和协作来共同完成复杂任务的系统，其核心挑战在于如何让多个智能体高效地共享信息、分工合作并解决冲突。

---

## 多智能体系统的定义

```
单智能体：一个智能体 + 一套工具 + 一个上下文
  → 适合：单一领域、线性流程、输入输出明确的任务

多智能体：多个智能体 × 各自工具 × 共享/独立上下文
  → 适合：跨领域、需多方决策、可并行处理的任务
```

多智能体系统的本质是**任务的分解与分配**：把一个大问题拆成多个子问题，分配给擅长各自领域的智能体，通过协作得到最终结果。

---

## 多智能体 vs 单智能体

| 维度 | 单智能体 | 多智能体 |
|------|---------|---------|
| **决策中心** | 单一决策者 | 多个决策者 |
| **上下文** | 单一上下文 | 每个智能体有自己的上下文 + 共享上下文 |
| **工具集** | 所有工具归一个智能体 | 工具按角色分配 |
| **并行度** | 串行执行 | 可并行 |
| **容错性** | 单点故障 | 单个智能体可被替换 |
| **复杂度** | 低，易于调试 | 高，协调是主要挑战 |
| **成本** | 一次模型调用 | 多次模型调用 |
| **适用场景** | 简单、明确的任务 | 复杂、跨领域的任务 |

## 三种架构模式

### 1. 集中式架构（Centralized）

```
                 ┌─────────────┐
                 │    Orchestrator     │
                 │  (协调者/规划者)    │
                 └─────────────┘
                  /     |     \
                 /      |      \
                /       |       \
          ┌──────┐  ┌──────┐  ┌──────┐
          │Agent A│  │Agent B│  │Agent C│
          │ 搜索  │  │ 分析  │  │ 生成  │
          └──────┘  └──────┘  └──────┘
```

特点：一个中央协调者负责任务分配和结果汇总。子智能体只执行，不做全局决策。

```python
class CentralizedOrchestrator:
    def __init__(self):
        self.agents = {
            "search": SearchAgent(),
            "analyze": AnalyzeAgent(),
            "generate": GenerateAgent(),
        }

    async def run(self, task: str) -> str:
        # 协调者规划任务
        plan = await self.planner.create_plan(task)

        results = {}
        for step in plan.steps:
            agent = self.agents[step.assigned_agent]
            results[step.name] = await agent.execute(
                step.input,
                context=plan.context
            )

        # 协调者汇总结果
        return await self.synthesizer.synthesize(results)
```

### 2. 分布式架构（Decentralized）

```
     Agent A ──── Agent B
        │  \        /  │
        │   \      /   │
        │    \    /    │
     Agent C ──── Agent D
```

特点：没有中央协调者，智能体之间直接通信协商。适合平等协作场景。

```python
class DecentralizedAgent:
    def __init__(self, name: str, peers: list):
        self.name = name
        self.peers = peers

    async def broadcast(self, message: Message):
        for peer in self.peers:
            await peer.receive(message)

    async def negotiate(self, task: Task) -> TaskAssignment:
        # 智能体之间通过协商决定谁做什么
        for peer in self.peers:
            response = await peer.propose(task)
            if response.confidence > 0.8:
                return TaskAssignment(
                    agent=peer.name,
                    task=task
                )
```

### 3. 混合式架构（Hybrid）

```
                 ┌──────────────┐
                 │ Supervisor Agent │
                 └──────────────┘
                         │
          ┌──────────────┼──────────────┐
          ↓              ↓              ↓
     ┌────────┐    ┌────────┐    ┌────────┐
     │ Worker │◄──►│ Worker │◄──►│ Worker │
     └────────┘    └────────┘    └────────┘
          │              │              │
          └──────────────┼──────────────┘
                         ↓
                 ┌──────────────┐
                 │   Critic Agent    │
                 └──────────────┘
```

特点：有协调者但不做所有决策，工作节点可互相通信，有专门的审查者。最大程度兼顾控制力和灵活性。

## 多智能体系统的关键机制

### 1. 通信协议

智能体之间如何交换信息：

```python
class Message:
    def __init__(self, sender: str, receiver: str,
                 msg_type: str, payload: dict):
        self.sender = sender
        self.receiver = receiver
        self.msg_type = msg_type  # request/response/broadcast
        self.payload = payload
        self.timestamp = time.now()

class CommunicationProtocol:
    def __init__(self):
        self.message_queue = asyncio.Queue()

    async def send(self, message: Message):
        await self.message_queue.put(message)

    async def receive(self, agent_name: str) -> list[Message]:
        messages = []
        while not self.message_queue.empty():
            msg = await self.message_queue.get()
            if msg.receiver == agent_name or msg.receiver == "*":
                messages.append(msg)
        return messages
```

### 2. 共识机制

多个智能体对同一问题给出不同答案时如何达成一致：

```python
class ConsensusManager:
    async def reach_consensus(
        self, agents: list[Agent],
        question: str
    ) -> ConsensusResult:
        # 收集所有智能体的意见
        opinions = []
        for agent in agents:
            response = await agent.respond(question)
            opinions.append(response)

        # 投票或加权平均
        final_answer = self.majority_vote(opinions)
        confidence = self.calculate_confidence(opinions)

        return ConsensusResult(
            answer=final_answer,
            confidence=confidence,
            disagreement=len(set(o.answer for o in opinions)) > 1
        )
```

### 3. 状态同步

智能体之间如何保持状态一致性：

```typescript
interface SharedState {
  taskId: string;
  globalProgress: number;
  artifacts: Map<string, Artifact>;
  locks: Map<string, string>;  // resource -> agent holding lock
}

class StateSynchronizer {
  private state: SharedState;

  // 乐观锁：先操作，冲突时回滚
  async updateArtifact(
    artifactId: string,
    update: Partial<Artifact>,
    agentId: string
  ): Promise<boolean> {
    const lock = this.state.locks.get(artifactId);
    if (lock && lock !== agentId) {
      return false;  // 被其他智能体锁定
    }
    this.state.artifacts.set(artifactId, {
      ...this.state.artifacts.get(artifactId),
      ...update
    });
    return true;
  }
}
```

## 多智能体系统的失败模式

### 1. 协调失败

```
Orchestrator 把任务 A 分配给 Agent X 和 Agent Y
  → 两人都去做任务 A → 重复工作
  → 或者两人都以为对方会做 → 任务遗漏
```

### 2. 通信过载

```
Agent X 每做完一步就广播给所有智能体
Agent Y 每次收到广播都重新规划自己的任务
  → 通信量 O(n²)，每个智能体频繁重规划
  → 系统效率反而低于单智能体
```

### 3. 级联失败

```
Agent X 出错 → 输出错误结果
Agent Y 依赖 Agent X 的结果 → 继续错
Agent Z 基于 Y 的结果做最终决策 → 全错
  → 错误在整个系统中传播放大
```

### 4. 目标漂移

```
Agent A 的目标：优化代码可读性
Agent B 的目标：优化代码性能
  → A 的重构破坏了 B 的性能优化
  → B 的优化让 A 的代码不可读
  → 双方在冲突中消耗大量 token
```

## 工程实践：构建多智能体系统

### 1. 系统设计检查清单

```
[ ] 是否有明确的边界？什么时候用多智能体 vs 单智能体？
[ ] 智能体之间如何通信？同步还是异步？
[ ] 如何保证状态一致性？
[ ] 如何处理智能体冲突？
[ ] 如何监控每个智能体的行为？
[ ] 单个智能体失败如何不影响全局？
[ ] 成本和延迟是否在可接受范围内？
[ ] 是否有回滚机制？
```

### 2. 最小可行多智能体实现

```python
class MultiAgentSystem:
    def __init__(self):
        self.agents: dict[str, Agent] = {}
        self.orchestrator = Orchestrator()
        self.comm_bus = MessageBus()
        self.state_store = StateStore()
        self.logger = Logger()

    async def register_agent(self, name: str, agent: Agent):
        agent.set_communicator(self.comm_bus)
        agent.set_state_store(self.state_store)
        self.agents[name] = agent

    async def execute(self, task: str) -> dict:
        # 记录执行轨迹
        trace = ExecutionTrace(task)

        # 1. 规划
        plan = await self.orchestrator.plan(task)
        trace.add_plan(plan)

        # 2. 分配并执行
        for step in plan.steps:
            agent = self.agents.get(step.assignee)
            if not agent:
                trace.add_failure(step, "Agent not found")
                continue

            result = await agent.execute(
                step, self.state_store.get_shared_context()
            )
            trace.add_step(step, result)

            # 3. 检查是否需要重新规划
            if result.needs_replan:
                plan = await self.orchestrator.replan(
                    plan, step, result
                )

        return {
            "result": await self.orchestrator.synthesize(trace),
            "trace": trace.export()
        }
```

### 3. 可观测性

```python
class AgentMonitor:
    def __init__(self):
        self.metrics = defaultdict(Counter)

    def record_event(self, event: AgentEvent):
        self.metrics[f"{event.agent}.total_calls"] += 1
        self.metrics[f"{event.agent}.total_tokens"] += event.tokens
        self.metrics[f"{event.agent}.total_latency_ms"] += event.latency_ms
        self.metrics[f"{event.agent}.errors"] += int(event.is_error)
        self.metrics["system.conflicts"] += int(event.is_conflict)
        self.metrics["system.replans"] += int(event.is_replan)

    def get_report(self) -> dict:
        return {
            "total_cost": self.metrics["system.total_tokens"] * 0.01,
            "avg_latency": self.metrics["system.total_latency_ms"],
            "error_rate": (
                self.metrics["system.errors"] /
                max(self.metrics["system.total_calls"], 1)
            ),
            "conflict_rate": (
                self.metrics["system.conflicts"] /
                max(self.metrics["system.total_calls"], 1)
            )
        }
```

## 常见误区

- 以为多智能体就是多个模型并行调用，忽略了协调和通信的开销
- 认为多智能体一定比单智能体好，不考虑"一个优秀智能体 + 精心设计的工具链"是否已经足够
- 低估了上下文同步的复杂度，以为每个智能体只需要自己的上下文
- 没有考虑智能体之间的竞争条件和对共享资源的锁定
- 在简单任务上强行使用多智能体，导致不必要的复杂度和成本

## 学习建议

建议动手搭建一个简单的多智能体系统，比如"代码审查系统"：一个智能体写代码、一个审核、一个测试。先用集中式架构跑通，再尝试加入分布式协商和冲突解决。在搭建过程中重点关注：通信机制的设计、状态同步、冲突检测，以及最终的评测指标对比（成本、延迟、质量）。面试时能讲清楚你踩过的坑和为什么这样设计，会比背概念有说服力得多。
