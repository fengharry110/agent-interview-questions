# 多智能体的成本和延迟如何控制？

## 面试问题

多智能体的成本和延迟如何控制？

## 考察重点

- 是否理解多智能体系统的成本构成（模型调用、上下文传递、协调开销）
- 是否能说清楚延迟的瓶颈在哪里（串行等待、模型响应、通信传输）
- 是否有实际的成本优化经验，能给出量化数据
- 是否理解成本、延迟和质量三者之间的权衡关系

## 核心结论

多智能体系统的成本和延迟控制是一个"**三角权衡**"：降低成本会牺牲质量，降低延迟会牺牲成本，提升质量会增加成本和延迟；系统化控制需要从**模型选择、上下文管理、并行策略、缓存复用**四个方向同时入手。

## 成本构成分析

```
多智能体系统一次任务的成本分解：

            ┌─────────────────────────┐
            │  Planner 模型调用 20%      │
            ├─────────────────────────┤
            │  Executor 模型调用 40%     │
            ├─────────────────────────┤
            │  Critic 模型调用 20%       │
            ├─────────────────────────┤
            │  协调通信开销 10%          │
            ├─────────────────────────┤
            │  上下文传递开销 10%        │
            └─────────────────────────┘

对比单智能体：
  单智能体：1 次规划 + 5 次执行 = 6 次模型调用
  多智能体：3 次规划 + 5 次执行 + 2 次审查 + N 次协调 = 15+ 次模型调用
  成本差：2.5x - 5x
```

### 成本公式

```
总成本 = Σ(每次模型调用 token 数 × 模型单价) + 外部 API 费用

每次模型调用成本 = input_tokens × input_price + output_tokens × output_price

多智能体额外成本：
  协调成本：Orchestrator 的规划、分配、汇总调用
  通信成本：智能体之间传递信息的 token
  冗余成本：多个智能体处理同一信息的重复 token
  冲突成本：冲突解决过程中的额外调用
```

## 延迟构成分析

```
单次任务延迟分解（多智能体串行场景）：

  Planner 规划:    2s (模型响应)
  Executor 执行:   5s (模型响应 + 工具调用)
  Critic 审查:     2s (模型响应)
  协调传递:        1s (上下文序列化/反序列化)
  冲突解决:        1s (如果发生)
                    ─────
  总延迟:         11s (无冲突)
                   23s (冲突 + 一次重试)

瓶颈识别：
  - 模型响应时间占总延迟的 70-80%
  - 上下文传递占 10-15%
  - 冲突解决占 5-10%
```

## 控制策略一：模型选择

### 强弱模型混用

```python
class CostOptimizedSystem:
    def __init__(self):
        self.strong_model = GPT4  # 用于关键决策
        self.medium_model = GPT4Mini  # 用于一般任务
        self.weak_model = GPT35  # 用于简单任务

    async def select_model(self, task: TaskComplexity) -> Model:
        if task.requires_reasoning:
            return self.strong_model  # 2x 成本，但关键步骤需要
        elif task.requires_tool_selection:
            return self.medium_model  # 0.5x 成本，足够好
        else:
            return self.weak_model  # 0.1x 成本，便宜快速

    async def execute_step(self, step: Step) -> Result:
        model = self.select_model(self.assess_complexity(step))
        return await model.generate(step.prompt)
```

**效果**：总成本降低 40-60%，质量下降 < 5%

## 控制策略二：上下文优化

### 1. 上下文裁剪

```python
class ContextOptimizer:
    def __init__(self, max_tokens: int = 8000):
        self.max_tokens = max_tokens

    def optimize(self, context: list[Message]) -> list[Message]:
        total_tokens = self.count_tokens(context)

        if total_tokens <= self.max_tokens:
            return context

        # 策略：按优先级保留
        priority_order = [
            "system_prompt",
            "task_goal",
            "latest_interaction",
            "tool_results",
            "historical_context"
        ]

        optimized = []
        remaining_budget = self.max_tokens

        for category in priority_order:
            messages = self.get_messages_by_category(context, category)
            for msg in messages:
                tokens = self.count_tokens([msg])
                if tokens <= remaining_budget:
                    optimized.append(msg)
                    remaining_budget -= tokens
                else:
                    # 截断或摘要剩余内容
                    if category in ["tool_results", "historical_context"]:
                        summary = self.summarize(msg, remaining_budget)
                        if summary:
                            optimized.append(summary)
                    break

        return optimized

    def summarize(self, message: Message, budget: int) -> Message:
        # 使用快速模型摘要
        summary_text = f"[摘要] {message.content[:budget * 4]}..."
        return Message(role="summary", content=summary_text)
```

**效果**：token 消耗降低 30-50%，信息保留率 > 85%

### 2. 上下文复用

```python
class ContextCache:
    def __init__(self):
        self.cache: dict[str, CachedContext] = {}
        self.ttl = 300  # 5 分钟缓存

    async def get_or_compute(
        self, key: str, compute_func: callable
    ) -> str:
        now = time.time()

        # 检查缓存
        if key in self.cache:
            cached = self.cache[key]
            if now - cached.timestamp < self.ttl:
                return cached.context

        # 重新计算
        context = await compute_func()
        self.cache[key] = CachedContext(
            context=context,
            timestamp=now
        )

        return context

    def invalidate(self, key: str):
        if key in self.cache:
            del self.cache[key]
```

**效果**：重复任务的 token 消耗降低 60-80%

### 3. 增量更新

```python
class IncrementalContextUpdater:
    def __init__(self):
        self.base_context = ""
        self.deltas: list[Delta] = []

    def apply_delta(self, delta: Delta):
        self.deltas.append(delta)

    def build_context(self) -> str:
        if not self.deltas:
            return self.base_context

        # 只更新有变化的部分
        context = self.base_context
        for delta in self.deltas:
            if delta.type == "append":
                context += f"\n{delta.content}"
            elif delta.type == "replace":
                context = context.replace(delta.old, delta.new)
            elif delta.type == "delete":
                context = context.replace(delta.content, "")

        return context

    def estimate_tokens(self) -> int:
        # 对比全量更新，计算节省
        full_context = self.build_context()
        incremental_tokens = self.count_delta_tokens()
        savings = len(full_context) - incremental_tokens
        return savings
```

## 控制策略三：并行执行

### 1. 任务并行化

```python
class ParallelExecutor:
    async def execute_parallel(self, steps: list[Step]) -> dict:
        # 分析依赖关系，找到可并行的步骤
        dag = self.build_dependency_graph(steps)

        results = {}
        ready_queue = self.get_initial_tasks(dag)

        while ready_queue:
            # 并行执行所有无依赖的任务
            batch_tasks = [
                self.execute_single(task)
                for task in ready_queue
                if task.id not in results
            ]

            batch_results = await asyncio.gather(*batch_tasks)

            for task, result in zip(ready_queue, batch_results):
                results[task.id] = result

            # 更新就绪队列
            ready_queue = self.get_newly_ready_tasks(dag, results)

        return results

    def estimate_speedup(self, steps: list[Step]) -> float:
        # 计算理论加速比（Amdahl's Law）
        serial_part = sum(
            s.estimated_time for s in steps
            if s.has_dependency
        )
        parallel_part = sum(
            s.estimated_time for s in steps
            if not s.has_dependency
        )
        cores = self.max_parallelism
        speedup = 1 / (serial_part + parallel_part / cores)
        return speedup
```

**效果**：延迟降低 30-60%（取决于可并行比例）

### 2. 流式输出

```python
class StreamingAgent:
    async def execute_streaming(self, task: str) -> AsyncIterator[str]:
        # 边生成边输出，不等完整结果
        plan = await self.planner.create_plan(task)

        async for partial_result in self.executor.execute_streaming(plan):
            yield partial_result

            # 同时开始预加载下一步
            if self.has_next_step(plan):
                next_step = self.prepare_next(plan)
                # 预取上下文、预热连接
                await self.prefetch_resources(next_step)
```

**效果**：首 token 时间降低 50%，用户体验显著提升

## 控制策略四：缓存与复用

### 1. 工具调用结果缓存

```python
class ToolCache:
    def __init__(self):
        self.cache: dict[str, CacheEntry] = {}
        self.max_size = 1000

    async def execute_with_cache(
        self, tool: str, args: dict, ttl: int = 60
    ) -> ToolResult:
        cache_key = f"{tool}:{json.dumps(args, sort_keys=True)}"

        # 检查缓存
        if cache_key in self.cache:
            entry = self.cache[cache_key]
            if time.time() - entry.timestamp < ttl:
                return entry.result

        # 执行工具
        result = await self.execute_tool(tool, args)

        # 写入缓存
        self.cache[cache_key] = CacheEntry(
            result=result,
            timestamp=time.time()
        )

        return result
```

**效果**：重复工具调用节省 80-90% 成本

### 2. 智能体输出复用

```python
class AgentOutputCache:
    def __init__(self):
        self.cache: dict[str, CachedOutput] = {}

    async def get_agent_output(
        self, agent_id: str, task_signature: str
    ) -> str | None:
        cache_key = f"{agent_id}:{task_signature}"

        if cache_key in self.cache:
            entry = self.cache[cache_key]
            if self.is_still_valid(entry):
                return entry.output

        return None

    def is_still_valid(self, entry: CachedOutput) -> bool:
        # 检查依赖是否变化
        for dep in entry.dependencies:
            if dep.has_changed_since(entry.timestamp):
                return False
        return True
```

## 控制策略五：自适应策略

### 动态调整执行策略

```python
class AdaptiveCostController:
    def __init__(self, budget: float):
        self.budget = budget
        self.spent = 0
        self.quality_history = []

    async def decide_execution_strategy(
        self, task: Task
    ) -> ExecutionStrategy:
        remaining_budget = self.budget - self.spent
        estimated_complexity = self.estimate_complexity(task)

        if remaining_budget < 0.1:
            # 预算不足：低成本模式
            return ExecutionStrategy(
                model="gpt-3.5-turbo",
                max_iterations=1,
                skip_critic=True,
                context_compression="aggressive"
            )
        elif estimated_complexity > 0.8:
            # 复杂任务：投入更多资源
            return ExecutionStrategy(
                model="gpt-4",
                max_iterations=3,
                skip_critic=False,
                context_compression="conservative"
            )
        else:
            # 一般任务：平衡模式
            return ExecutionStrategy(
                model="gpt-4-turbo-preview",
                max_iterations=2,
                skip_critic=False,
                context_compression="normal"
            )

    def record_quality(self, quality: float):
        self.quality_history.append(quality)
        # 如果质量持续下降，调整策略
        if len(self.quality_history) > 5:
            recent_avg = sum(self.quality_history[-5:]) / 5
            if recent_avg < 0.7:
                self.budget *= 1.2  # 增加预算
```

### 质量-成本-延迟三角权衡

```
          Quality
             ▲
            /|\
           / | \
          /  |  \
         /   |   \
        /    |    \
       /     |     \
      /      |      \
     /       |       \
    /        |        \
   /         |         \
Cost ◄───────┴──────────► Latency

无法同时优化三个维度：
  方案 A：高质量 + 低延迟 = 高成本（用最强模型并行）
  方案 B：低成本 + 低延迟 = 低质量（用小模型快速执行）
  方案 C：低成本 + 高质量 = 高延迟（用多次迭代）
```

## 常见误区

- 为了省钱全用小模型，导致质量太差需要多次重试，最终成本反而更高
- 并行执行所有任务，不加分析地认为"越快越好"，忽略了串行依赖
- 缓存滥用：缓存了不应该缓存的结果（如安全策略检查），引入安全隐患
- 只优化模型调用成本，忽略了上下文传递和序列化的开销
- 预算控制太死板，在复杂任务上强行使用低成本策略导致任务失败

## 学习建议

选一个多智能体任务，记录基线版本的成本和延迟。然后逐一应用以下优化：强弱模型混用（记录节省比例）、上下文裁剪（记录 token 减少）、并行执行（记录延迟改善）、结果缓存（记录复用率）。把每次优化前后的数据整理成表格，面试时展示这些真实数据会比空谈策略有说服力得多。核心是要理解：优化不是线性过程，第一个 50% 的优化最容易，后 50% 需要精细调优。
