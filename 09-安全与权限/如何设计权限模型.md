# 如何设计权限模型？

## 面试问题

如何设计权限模型？

## 考察重点

- 是否理解智能体系统的权限模型和传统 RBAC 模型的区别
- 是否能说清楚"用户权限"和"工具权限"和"数据权限"三个维度
- 是否了解最小权限原则在智能体系统中的具体落地方式
- 是否有设计智能体权限模型的经验

## 核心结论

智能体系统的权限模型有三个维度：**用户维度**（谁在用）、**工具维度**（能做什么）、**数据维度**（能访问什么）。核心原则是最小权限——智能体默认没有任何权限，每次需要权限时动态申请。权限模型的设计目标是让智能体刚好能完成用户的任务，不多也不少。

### 工具维度

```python
class ToolPermissionManager:
    """
    工具级别的权限管理
    """
    
    def __init__(self):
        self.permissions: Dict[str, ToolPermission] = {}
    
    def register_tool_permission(self, 
                                  tool_name: str,
                                  permission: ToolPermission):
        """注册工具的权限配置"""
        self.permissions[tool_name] = permission
    
    def check_tool_access(self, 
                          user: User,
                          tool_name: str,
                          context: Context) -> AccessResult:
        """
        检查用户是否有权在上下文中使用工具
        """
        if tool_name not in self.permissions:
            return AccessResult(denied=True, reason="工具不存在")
        
        perm = self.permissions[tool_name]
        
        # 1. 检查角色
        if user.role not in perm.allowed_roles:
            return AccessResult(
                denied=True,
                reason=f"{user.role} 没有使用 {tool_name} 的权限"
            )
        
        # 2. 检查上下文限制
        for constraint in perm.context_constraints:
            if not constraint.check(context):
                return AccessResult(
                    denied=True,
                    reason=constraint.failure_reason
                )
        
        # 3. 是否需要确认
        if perm.requires_confirmation:
            return AccessResult(
                denied=False,
                requires_confirmation=True
            )
        
        return AccessResult(denied=False)

class ToolPermission:
    """工具权限配置"""
    
    def __init__(self,
                 allowed_roles: List[str],
                 requires_confirmation: bool = False,
                 context_constraints: List = None):
        self.allowed_roles = allowed_roles
        self.requires_confirmation = requires_confirmation
        self.context_constraints = context_constraints or []
```

### 数据维度

```python
class DataScopeManager:
    """
    数据范围管理
    """
    
    def __init__(self):
        self.path_rules: List[PathRule] = []
    
    def add_path_rule(self, rule: PathRule):
        """添加路径访问规则"""
        self.path_rules.append(rule)
    
    def check_path_access(self, 
                          user: User,
                          path: str,
                          mode: str) -> AccessResult:
        """
        检查用户是否有权以指定模式访问路径
        """
        allowed = False
        matched_rule = None
        
        for rule in self.path_rules:
            if fnmatch.fnmatch(path, rule.pattern):
                matched_rule = rule
                if user.role in rule.allowed_roles:
                    if mode in rule.allowed_modes:
                        allowed = True
                        break
        
        if not allowed:
            return AccessResult(
                denied=True,
                reason=f"无权以 {mode} 模式访问 {path}"
            )
        
        return AccessResult(denied=False)
```

## 权限确认等级对比

| 等级 | 行为 | 适用场景 | 风险 | 用户体验 |
|------|------|---------|------|---------|
| relaxed | 所有操作直接执行 | 管理员、可信任环境 | 高 | 最好 |
| balanced | 安全操作直行，危险操作确认 | 普通开发者 | 中 | 良好 |
| strict | 所有工具调用都需确认 | 访客、敏感操作 | 低 | 较差 |
| read_only | 只允许读操作 | 只读用户 | 极低 | 受限 |

## 深度展开

### 动态权限申请

智能体不应预加载所有权限，而应在需要时动态申请：

```python
class DynamicPermissionRequest:
    """
    动态权限申请流程
    """
    
    async def request_permission(
        self,
        agent: Agent,
        required_permission: Permission
    ) -> PermissionResult:
        """
        智能体在需要权限时动态申请
        """
        # 1. 检查缓存中是否已有此权限
        cached = self._check_cache(
            agent.session_id, required_permission
        )
        if cached:
            return PermissionResult(granted=True, source='cache')
        
        # 2. 向权限管理器申请
        decision = await self.permission_manager.evaluate(
            agent=agent,
            requested=required_permission,
            context=agent.current_context
        )
        
        if decision == 'auto_grant':
            # 自动授权（低风险）
            self._cache_permission(agent.session_id, required_permission)
            return PermissionResult(
                granted=True,
                source='auto',
                valid_duration=300
            )
        
        elif decision == 'require_confirmation':
            # 需要用户确认
            confirmed = await self._ask_user(required_permission)
            if confirmed:
                self._cache_permission(agent.session_id, required_permission)
                return PermissionResult(
                    granted=True,
                    source='user_confirmed'
                )
            return PermissionResult(granted=False, reason='用户拒绝')
        
        else:
            return PermissionResult(
                granted=False,
                reason='权限管理器拒绝'
            )
```

### 最小权限策略

```python
class LeastPrivilegeStrategy:
    """
    最小权限策略实现
    """
    
    def __init__(self):
        self.session_permissions: Dict[str, Set] = {}
    
    def init_session(self, session_id: str, user_role: str):
        """会话开始时授予最小必要权限"""
        base_permissions = {
            "admin": {"read_file", "search", "list_directory"},
            "developer": {"read_file", "search", "list_directory"},
            "viewer": {"search", "list_directory"}
        }
        self.session_permissions[session_id] = set(
            base_permissions.get(user_role, set())
        )
    
    def expand_permission(self, session_id: str, permission: str):
        """按需扩展权限"""
        if session_id in self.session_permissions:
            self.session_permissions[session_id].add(permission)
    
    def revoke_permission(self, session_id: str, permission: str):
        """完成任务后回收权限"""
        if session_id in self.session_permissions:
            self.session_permissions[session_id].discard(permission)
    
    def cleanup_session(self, session_id: str):
        """会话结束清理所有权限"""
        self.session_permissions.pop(session_id, None)
```

### 权限决策链

```
用户请求
    ↓
┌──────────────────┐
│ 1. 用户身份认证   │  ← 从 token/session 获取用户角色
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 2. 工具权限检查   │  ← 该角色是否允许使用该工具
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 3. 上下文约束检查  │  ← 当前上下文是否满足限制条件
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 4. 数据范围检查   │  ← 目标资源是否在允许范围内
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 5. 动态风险评估   │  ← 结合历史行为做风险评分
└────────┬─────────┘
         ↓
    [通过] → 执行操作 + 记录审计日志
    [拒绝] → 返回拒绝原因 + 记录审计日志
```

### 权限缓存策略

```python
class PermissionCache:
    """
    会话级权限缓存，减少重复检查
    """
    
    def __init__(self, ttl: int = 300):
        self.cache: Dict[str, CacheEntry] = {}
        self.ttl = ttl
    
    def get(self, session_id: str, permission_key: str) -> Optional[PermissionResult]:
        entry = self.cache.get(f"{session_id}:{permission_key}")
        if entry and time.time() - entry.timestamp < self.ttl:
            return entry.result
        return None
    
    def set(self, session_id: str, permission_key: str, result: PermissionResult):
        self.cache[f"{session_id}:{permission_key}"] = CacheEntry(
            result=result,
            timestamp=time.time()
        )
    
    def invalidate_session(self, session_id: str):
        keys = [k for k in self.cache if k.startswith(f"{session_id}:")]
        for k in keys:
            del self.cache[k]
```

## 常见误区

- **权限粒度太粗**：要么全部允许要么全部禁止，缺少精细控制。需要分级、分上下文、分数据的细粒度权限
- **权限静态分配**：会话一开始就把所有权限给智能体，违反了最小权限原则，容易导致越权操作
- **忽略用户身份**：不同用户的能力和需求不同，统一的权限策略不适合所有人
- **权限和上下文脱钩**：同样的用户同样的工具，在不同任务中可能需要不同的权限。权限判断需要结合上下文
- **权限缓存时间过长**：缓存了权限后用户角色变更或权限撤销无法及时生效
- **只检查工具不检查参数**：允许使用 read_file 但参数传入 /etc/passwd 应被拒绝
- **忽略权限链组合风险**：智能体通过组合多个低风险工具完成一个高风险操作

## 追问方向

- 你的权限模型支持动态权限升级吗？比如开发者临时需要管理员权限来修复紧急问题
- 如何防止智能体通过合法的工具组合实现越权？（比如用 read_file 读到了本不该读的文件）
- 权限检查失败时应该返回什么给模型？暴露了太多信息可能帮助攻击者，暴露太少模型不知道怎么修正
- 权限和审计日志如何联动？权限检查通过和拒绝的记录应该保留多久？
- 多租户场景下，权限模型如何隔离不同租户的数据和操作？
- 如果权限引擎本身出现故障（如不可用），系统应该采取什么降级策略？

## 学习建议

从最粗粒度的权限模型开始——只有"管理员"和"普通用户"两个角色。运行一段时间后，根据实际需要增加精细度。权限模型的关键不是一开始就设计完美，而是能够灵活演进。优先确保"不该执行的绝对不能执行"（安全），再优化"该执行的尽量减少确认"（体验）。在面试中，重点强调三维权限模型（用户/工具/数据）和最小权限的动态授予这两点。
