# 用户需求模糊时智能体应该怎么做？

## 面试问题

用户需求模糊时智能体应该怎么做？

## 考察重点

- 是否理解模糊需求的类型（参数模糊、意图模糊、目标模糊）
- 是否能设计有效的澄清策略
- 是否了解何时该主动推断、何时该询问
- 是否有处理多次澄清失败的经验

## 核心结论

处理模糊需求的核心是**主动澄清 + 智能推断**。不要假设、不要猜测用户意图，要主动询问关键信息。澄清时提供具体选项而非开放式问题。

---

## 模糊需求的类型

### 类型一：参数缺失

```
用户："帮我查一下" (缺少查什么)
用户："帮我订个位" (缺少时间、人数)
用户："帮我取消" (缺少取消什么)
```

**特点**：意图明确，但缺少必要参数

### 类型二：意图模糊

```
用户："帮我看看" (不知道看什么)
用户："这个怎么样" (不知道问什么)
用户："有什么推荐的吗" (没有明确偏好)
```

**特点**：用户自己也不确定想要什么

### 类型三：表达不清晰

```
用户："帮我处理一下" (处理什么？怎么处理？)
用户："上次那个" (需要从历史推断)
用户："那个问题" (指代不明)
```

**特点**：有信息但表达不明确

## 处理策略

### 策略一：主动澄清

```python
def handle_fuzzy_request(user_input: str, intent: str) -> str:
    missing_slots = get_missing_slots(intent)

    if missing_slots:
        return generate_clarification_prompt(intent, missing_slots)

    return "抱歉，我需要更多信息才能帮助您。"
```

### 策略二：提供选项

```python
CLARIFICATION_OPTIONS = {
    "query_something": [
        "查询订单状态",
        "查询账户余额",
        "查询物流信息",
        "查询其他"
    ],
    "order_type": [
        "外卖订餐",
        "酒店预订",
        "机票预订",
        "其他服务"
    ]
}

def generate_clarification_with_options(intent: str) -> str:
    options = CLARIFICATION_OPTIONS.get(intent, [])

    if options:
        options_text = "\n".join([
            f"{i+1}. {opt}" for i, opt in enumerate(options)
        ])
        return f"请问您想了解哪个？\n{options_text}"

    return None
```

### 策略三：上下文推断

```python
def infer_from_context(user_input: str, context: ConversationContext) -> Option:
    # 从历史中推断
    last_intent = context.get_last_intent()
    if last_intent:
        # 用户可能在延续上一个意图
        return Option(
            inferred_intent=last_intent,
            confidence=0.7,
            message="您是想继续上次的操作吗？"
        )

    # 从当前环境中推断
    current_page = context.get_current_page()
    if current_page:
        # 用户可能想问当前页面相关的问题
        return infer_from_page_context(user_input, current_page)

    return None
```

## 澄清话术设计

### 好话术 vs 坏话术

```
❌ 坏："请问您想做什么？"
✅ 好："您是想(A)查询订单 (B)取消订单 (C)其他？"

❌ 坏："您说的那个是什么？"
✅ 好："您是问订单号12345的物流信息吗？"

❌ 坏："请提供更多信息"
✅ 好："请问您的订单号是多少？可以在订单详情页找到。"
```

### 渐进式澄清

```python
def progressive_clarification(user_input: str, context: ConversationContext) -> str:
    # 第一层：最可能的解释
    most_likely = infer_most_likely_intent(user_input)

    if most_likely.confidence > 0.8:
        return f"您是想{most_likely.description}吗？"

    # 第二层：提供选项
    if most_likely.confidence > 0.5:
        return generate_options_question(most_likely)

    # 第三层：开放式询问
    return "抱歉我没理解清楚。请问您具体想查询什么呢？"
```

## 多轮澄清机制

```python
class ClarificationManager:
    def __init__(self):
        self.max_rounds = 3
        self.current_round = 0
        self.history = []

    def should_continue(self) -> bool:
        return self.current_round < self.max_rounds

    def add_exchange(self, user_input: str, system_question: str):
        self.history.append((user_input, system_question))
        self.current_round += 1

    def get_clarification_strategy(self) -> str:
        if self.current_round == 0:
            return "specific_options"
        elif self.current_round == 1:
            return "general_options"
        else:
            return "human_handoff"
```

## 特殊情况处理

### 情况一：用户拒绝澄清

```
用户："别问了，直接帮我弄"
```

**策略**：
```python
def handle_user_refusal(context: ClarificationManager) -> str:
    if context.current_round > 0:
        return """好的，我将尝试根据已有信息处理。
        如果结果不符合预期，您可以随时告诉我需要调整。"""

    return """抱歉，我需要一些信息才能更好地帮助您。
    您可以提供订单号或简短描述您的问题吗？"""
```

### 情况二：多次澄清后仍模糊

```python
def handle_clarification_failure(context: ClarificationManager) -> str:
    # 记录失败日志
    log_clarification_failure(context.history)

    # 转人工
    return """我理解您可能遇到了问题。
    为了更快地帮助您，我将为您转接到人工客服。"""

    # 或者尝试默认行为
    return """我将为您查询最近的订单信息，
    如果这不是您需要的，请告诉我具体要查询的内容。"
```

## 从 Claude Code 看模糊需求处理

Claude Code 在编程场景中处理模糊需求的方式：

```typescript
async function handleFuzzyCodeRequest(
    userInput: string,
    context: CodeContext
): Promise<CodeAction> {
    // 1. 检查用户是否在指代之前的代码
    const reference = resolveCodeReference(userInput, context);

    if (reference) {
        // 2. 确认指代是否正确
        return {
            type: "clarification",
            message: `您是指这个 ${reference.type} 吗？`,
            options: ["是的", "不是", "我要修改"]
        };
    }

    // 3. 检查是否在当前文件中
    const currentSymbols = getCurrentFileSymbols(context);
    const matching = fuzzyMatch(userInput, currentSymbols);

    if (matching.length === 1) {
        // 4. 高置信度推断
        return {
            type: "implicit_confirm",
            action: matching[0],
            message: `您是想修改 ${matching[0].name} 吗？`
        };
    }

    // 5. 提供选项
    if (matching.length > 1) {
        return {
            type: "disambiguate",
            options: matching.map(m => m.name),
            message: "您是指哪个？"
        };
    }

    // 6. 无法处理，返回帮助
    return {
        type: "help",
        message: "我需要更多信息。请提供函数名或描述您想做的修改。"
    };
}
```

## 评估标准

| 指标 | 计算方式 | 目标 |
|-----|---------|------|
| **澄清成功率** | 成功推断意图的次数 / 总澄清次数 | > 80% |
| **平均澄清轮次** | 总澄清轮次 / 澄清次数 | < 2 |
| **用户满意度** | 模糊请求处理后的满意度评分 | > 4/5 |
| **升级人工率** | 需要转人工的模糊请求比例 | < 10% |

## 面试追问方向

1. **"用户说'帮我弄一下'，你怎么处理？"**

   考察候选人对极端模糊输入的处理策略。

2. **"如果用户在澄清过程中变得不耐烦，你怎么调整策略？"**

   考察候选人对用户体验的敏感度。

3. **"你怎么平衡'多问几句'和'直接猜测'之间的度？"**

   考察候选人对置信度和用户体验的权衡。

4. **"模糊请求处理的效果怎么评估？"**

   考察候选人对指标设计的理解。
