# 意图识别应该用规则和小模型还是大模型？

## 面试问题

意图识别应该用规则和小模型还是大模型？

## 考察重点

- 是否理解不同技术方案的优缺点和适用场景
- 是否能根据业务场景做出合理的技术选型
- 是否了解混合使用多种方案的最佳实践
- 是否有实际的意图识别系统设计经验

## 核心结论

没有万能方案，只有合适的方案。规则最快、成本最低但覆盖率低；小模型（如 BERT）平衡了准确率和成本；大模型（如 GPT-4）最准确但成本高、延迟大。最佳实践是**分层架构**：规则做快速过滤，模型做精确分类，大模型处理边缘 case。

---

## 技术方案对比

| 方案 | 延迟 | 成本 | 准确率 | 覆盖率 | 适用场景 |
|-----|-----|-----|-------|-------|---------|
| **规则** | < 10ms | $0 | 中 | 低 | 固定模式、高频意图 |
| **小模型** | 50-100ms | $0.001 | 高 | 中 | 有训练数据、中等复杂度 |
| **大模型** | 1-5s | $0.01-0.1 | 最高 | 高 | 复杂意图、边缘 case |

## 何时用规则

### 适用场景

1. **意图表达固定**：用户只能用预设的方式表达
   ```
   用户只能说："查询余额"、"取消订单"
   不能说其他变体
   ```

2. **意图数量少**：只有 5-10 种意图
   ```
   意图列表：
   - help（帮助）
   - quit（退出）
   - restart（重启）
   - status（状态查询）
   - confirm（确认）
   ```

3. **高频意图**：占总流量的 80% 以上
   ```
   场景：客服机器人
   - 60% 是查询订单状态
   - 20% 是咨询退款
   - 20% 是其他问题
   ```

### 规则示例

```python
# 高频意图用规则
HIGH_FREQ_INTENTS = {
    "查.*余额": "query_balance",
    "余额.*查询": "query_balance",
    "余额多少": "query_balance",
    "取消.*订单": "cancel_order",
    "撤.*订单": "cancel_order",
}

def classify_high_freq(text: str) -> str | None:
    text = text.lower()
    for pattern, intent in HIGH_FREQ_INTENTS.items():
        if re.search(pattern, text):
            return intent
    return None
```

## 何时用小模型

### 适用场景

1. **意图表达多样**：用户可以用多种方式表达同一意图
   ```
   "帮我看看余额"
   "余额还有多少"
   "还剩多少钱"
   "账户情况如何"
   ```

2. **意图数量中等**：10-100 种意图
   ```
   需要分类的意图：
   - 账户管理（余额查询、明细查询、密码修改）
   - 订单管理（下单、取消、退款、评价）
   - 物流查询（状态、地址修改、签收）
   ```

3. **有一定训练数据**：每种意图有 100+ 条标注样本

### 小模型选择

| 模型 | 参数量 | 准确率 | 延迟 | 适用场景 |
|-----|-------|-------|-----|---------|
| **BERT-base** | 110M | 高 | 100ms | 通用场景 |
| **DistilBERT** | 66M | 中高 | 50ms | 追求延迟 |
| **ALBERT** | 12M | 中 | 30ms | 资源受限 |
| **TextCNN** | < 1M | 中 | 10ms | 超低延迟 |

### 小模型训练示例

```python
from transformers import BertForSequenceClassification, Trainer

model = BertForSequenceClassification.from_pretrained(
    "bert-base-chinese",
    num_labels=len(INTENT_LABELS)
)

trainer = Trainer(
    model=model,
    train_dataset=train_dataset,
    eval_dataset=eval_dataset,
)

trainer.train()

# 推理
def classify_intent(text: str) -> str:
    inputs = tokenizer(text, return_tensors="pt")
    outputs = model(**inputs)
    predicted_class = outputs.logits.argmax(-1)
    return INTENT_LABELS[predicted_class]
```

## 何时用大模型

### 适用场景

1. **意图复杂**：需要理解上下文、隐含意图
   ```
   用户："帮我看看那个上次的订单，就是上次说有问题那个"
   → 需要理解"上次说有问题那个"指的是什么
   ```

2. **边缘 case**：规则和模型都无法处理的情况
   ```
   用户："帮我处理一下我表弟的那个订单"
   → 这种奇怪的表达只有大模型能理解
   ```

3. **冷启动**：没有任何训练数据的新业务

### 大模型使用示例

```python
def classify_with_llm(text: str, context: list[str]) -> dict:
    prompt = f"""你是意图识别专家。根据用户输入和对话历史，识别用户意图。

    可选意图：query_balance, cancel_order, refund, complaint, other

    对话历史：
    {chr(10).join(context)}

    用户输入：{text}

    输出格式（JSON）：
    {{
        "intent": "意图名称",
        "confidence": 0.0-1.0,
        "slots": {{"参数名": "参数值"}},
        "reasoning": "识别理由"
    }}"""

    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}]
    )

    return json.loads(response.choices[0].message.content)
```

## 最佳实践：分层架构

### 三层分离

```
第一层：规则（< 10ms）
  ↓ 不匹配
第二层：小模型（50-100ms）
  ↓ 低置信度
第三层：大模型（1-5s）
```

```python
def classify_intent(text: str, context: list[str] = None) -> IntentResult:
    # 第一层：规则
    rule_result = classify_by_rules(text)
    if rule_result:
        return rule_result

    # 第二层：小模型
    small_model_result = classify_by_small_model(text)
    if small_model_result.confidence > 0.8:
        return small_model_result

    # 第三层：大模型（只处理低置信度的case）
    if small_model_result.confidence < 0.5:
        llm_result = classify_with_llm(text, context or [])
        return llm_result

    return small_model_result
```

### 成本优化

假设流量分布：
- 规则覆盖：60%（免费，<10ms）
- 小模型覆盖：30%（$0.001/次，50ms）
- 大模型兜底：10%（$0.01/次，1s）

加权成本：
```
60% * $0 + 30% * $0.001 + 10% * $0.01
= $0.0003 + $0.0001
= $0.0004/次

vs 全部用大模型：$0.01/次
节省：96%
```

## 工程实践建议

### 1. 明确评估标准

```python
# 业务指标
BUSINESS_THRESHOLDS = {
    "high_freq_intents": {  # 高频意图
        "accuracy": 0.99,  # 必须非常准确
        "latency": 20ms    # 必须快
    },
    "medium_freq_intents": {  # 中频意图
        "accuracy": 0.95,
        "latency": 200ms
    },
    "low_freq_intents": {  # 低频意图
        "accuracy": 0.90,  # 可以有误差
        "latency": 5000ms
    }
}
```

### 2. 建立回流机制

```python
def handle_low_confidence(result: IntentResult, original_text: str):
    """低置信度结果回流分析"""
    log_low_confidence(
        text=original_text,
        predicted_intent=result.intent,
        confidence=result.confidence,
        final_intent=None  # 等用户反馈后补充
    )
```

### 3. 定期优化

```
每周分析：低置信度case
    ↓
找出规则可覆盖的模式 → 加入规则
找出小模型可覆盖的模式 → 补充训练数据
找出大模型特有的能力 → 考虑fine-tune小模型
```
