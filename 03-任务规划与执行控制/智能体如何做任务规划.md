# 智能体如何做任务规划？

## 面试问题

智能体如何做任务规划？

## 考察重点

- 是否理解规划在智能体系统中的两种实现模式：ReAct 的边做边想 vs Plan-and-Execute 的先想后做
- 是否能说清楚状态机在规划中的具体作用，以及为什么 Claude Code 选择 while(true)+状态机而非递归
- 是否了解实际生产系统中规划失败的具体模式，以及如何检测和恢复
- 是否能从 Codex 和 Claude Code 源码中找到规划机制的具体实现

## 核心结论

任务规划不是"让模型生成一个计划"，而是设计一个**控制结构**来协调模型和工具的交互。主流实现有两种：ReAct（边做边想）和 Plan-and-Execute（先想后做），两者各有优劣。生产系统中，规划模块的核心挑战是**目标漂移检测**和**规划失败后的恢复**，而不是生成漂亮的计划文本。

## 规划的两条技术路线

### 路线一：ReAct（Reasoning + Acting）

模型在执行每个动作时都先输出思考过程，然后执行动作，再根据结果继续。思考和行动交织在一起。

**代表系统**：LangChain Agent、早期 GPT-4 插件

**优点**：
- 对简单任务反应快，不需要先生成完整计划
- 能根据上一步结果调整下一步行动

**缺点**：
- 复杂任务容易陷入"试错循环"
- 规划过程不透明，难以调试
- 模型可能在一个错误方向上走很远才回头

### 路线二：Plan-and-Execute

模型先完整分析任务，生成一个行动计划（通常是步骤列表），然后按顺序执行每一步，执行过程中可以调整计划。

**代表系统**：Claude Code、Codex

**优点**：
- 计划文本可记录、可审计
- 允许在执行前让用户确认计划
- 更容易实现"目标漂移检测"

**缺点**：
- 对简单任务有额外开销
- 计划可能因为信息不足而失效

## Codex 的规划实现

Codex 实现了一个内置的 `update_plan` 工具，定义如下：

```json
{
  "type": "function",
  "name": "update_plan",
  "description": "Updates the task plan...",
  "strict": false,
  "parameters": {
    "type": "object",
    "properties": {
      "plan": {...},
      "explanation":...
    },
    "required": ["plan"]
  }
}
```

模型可以调用这个工具来更新当前计划。Codex 的设计哲学是：**计划是模型的输出，不是系统的输入**。

具体流程：
1. 模型接收用户任务
2. 模型分析任务，调用 `update_plan` 产生初始计划
3. 模型开始执行计划的第一个步骤
4. 每执行完一个步骤，模型判断是否需要更新计划
5. 如果需要，调用 `update_plan` 修改计划
6. 重复直到任务完成

关键实现细节在 [codex-rs/core/src/tools/handlers/plan.rs](https://github.com/openai/codex/blob/99f47d6e9a3546c14c43af99c7a58fa6bd130548/codex-rs/core/src/tools/handlers/plan.rs#L20-L60)。

## Claude Code 的规划实现

Claude Code 使用了**状态机 + while(true) 循环**来实现规划控制，而不是递归。

### 为什么不用递归？

Claude Code 早期版本使用递归——query 函数调用自己。但递归有一个致命问题：当对话很长、有数百次工具调用时，调用栈会不断增长直到溢出。

```typescript
// query.ts — lines 207-215 (State type, partial)
interface State {
  autoCompactTracking: AutoCompactTrackingState | undefined;
  maxOutputTokensRecoveryCount: number;
  hasAttemptedReactiveCompact: boolean;
  stopHookActive: boolean | undefined;
  turnCount: number;
  transition: { reason: string } | undefined;
}
```

### 核心循环结构

```typescript
// query.ts — line 307
while (true) {
  let { toolUseContext } = state;
  const { ... } = state;
  // ... 1,421 lines of state machine logic ...
  state = next;
}
```

循环一次迭代包含 **10 个步骤**：

| 步骤 | 名称 | 作用 |
|-----|------|-----|
| 1 | Context Compression | 4 阶段上下文压缩，防止 token 超出限制 |
| 2 | Token Budget Check | 检查是否还有足够的 token 预算 |
| 3 | Call Model API | 调用模型 API |
| 4 | Streaming Tool Execution | 流式执行工具调用 |
| 5 | Error Recovery | 错误恢复 |
| 6 | Stop Hooks | 检查停止钩子 |
| 7 | Token Budget Check (Again) | 再次检查 token 预算 |
| 8 | Tool Execution | 执行工具 |
| 9 | Attachment Injection | 注入附件 |
| 10 | Assemble and Loop | 组装状态并继续循环 |

关键点：**9 个不同的 continue 点**，每个代表一次状态转换。这使得执行路径可以被追踪和调试。

## 目标漂移检测

规划失败最常见的原因是**目标漂移**（goal drift）——模型在执行过程中逐渐偏离了原始目标。

### 检测方法

1. **周期性检查**：每隔 N 步或 N 秒，检查当前状态是否还在原目标范围内
2. **用户确认**：执行关键步骤前请求用户确认
3. **失败计数**：连续失败超过阈值时触发恢复或放弃

Claude Code 使用 **reactive compact** 作为目标漂移的 circuit breaker：

```typescript
hasAttemptedReactiveCompact: boolean  // 是否已经尝试过响应式压缩
```

### 恢复策略

| 策略 | 适用场景 |
|-----|---------|
| 回退到上一个检查点 | 有周期性快照的系统 |
| 重新规划 | 上下文还有足够空间 |
| 压缩上下文后重试 | token 耗尽但还有机会 |
| 放弃并报告 | 超过最大重试次数 |

## 停止条件设计

停止条件是规划模块最关键的部分之一。设计不当会导致无限循环。

### 常见停止条件

1. **任务完成**：模型输出最终回复，不再调用工具
2. **Token 耗尽**：上下文窗口达到上限
3. **时间超限**：任务执行超过预定时间
4. **步骤超限**：执行的步骤数超过上限（如 max_steps=100）
5. **用户中断**：用户主动取消任务

### Codex 的停止条件

Codex 通过 Responses API 的 `previous_response_id` 机制追踪对话状态。当模型输出 `response.output_text.done` 而非 `function_call` 时，认为任务完成。

### Claude Code 的停止条件

Claude Code 在每次循环迭代结束时检查：
- `turnCount` 是否超过限制
- token 预算是否耗尽
- 用户是否发送了停止信号

## 工程实践：如何设计一个可调试的规划模块

### 1. 结构化日志

每次状态转换都要记录：
```json
{
  "turn": 42,
  "transition_reason": "tool_call_completed",
  "tool_name": "update_plan",
  "tool_args": {...},
  "tool_result": "success",
  "current_plan_steps_remaining": 3,
  "token_budget_used_pct": 67.2
}
```

### 2. 可视化执行轨迹

规划执行过程应该能生成类似以下的轨迹图：
```
[User] "帮我重构 auth 模块"
  │
  ├─► [Model] 分析任务，生成初始计划
  │     Plan: [
  │       "Step 1: 阅读 auth.go",
  │       "Step 2: 识别需要重构的函数",
  │       "Step 3: 重构并测试"
  │     ]
  │
  ├─► [Tool: ReadFile] auth.go
  │     Result: 读取成功，250 行
  │
  ├─► [Model] 更新计划
  │     Plan: [
  │       "Step 1: 重构 validateToken (80 行)"
  │       "Step 2: 重构 refreshToken (60 行)"
  │       "Step 3: 重构 hashPassword (40 行)"
  │       "Step 4: 写测试"
  │     ]
  │
  └─► ... 继续执行
```

### 3. 降级策略

当规划失败时，系统不应该直接崩溃，而是尝试降级：

| 失败类型 | 降级策略 |
|---------|---------|
| 计划生成失败 | 切换到 ReAct 模式，边做边想 |
| 工具执行失败 | 重试 3 次，失败则跳过该步骤 |
| Token 超限 | 压缩上下文，保留关键信息重试 |
| 连续失败 | 询问用户是否继续 |

## 面试追问方向

1. **"你的规划模块如何检测目标漂移？如果模型在第三步突然转向一个不相关的任务，系统会怎么做？"**

   考察候选人对目标保持（goal maintenance）的理解。

2. **"为什么 Claude Code 选择 while(true)+状态机而不是递归？有没有其他选择？"**

   考察候选人对系统设计和递归局限性的理解。

3. **"如果模型的计划在执行到一半时因为上下文耗尽而失败，你会怎么设计恢复机制？"**

   考察候选人对容错设计和降级策略的理解。

4. **"ReAct 和 Plan-and-Execute 各适合什么场景？你如何在系统中同时支持两种模式？"**

   考察候选人对不同规划范式的适用场景理解。

5. **"Codex 的 update_plan 工具允许模型随时修改计划，这可能导致计划不断变化。如何防止模型'计划通胀'（每一步都更新计划导致计划失去意义）？"**

   考察候选人对工具设计副作用的理解。

## 真实踩坑案例

### 案例：上下文窗口耗尽导致的"幽灵计划"

某团队实现 Plan-and-Execute 时发现：当上下文快耗尽时，模型会生成一个包含 20+ 步骤的"完整计划"，但实际只执行了前 5 步就因 token 限制被中断。更糟糕的是，系统没有记录计划，只记录了执行过程，导致无法追溯模型"想做什么"。

**修复方案**：
1. 强制要求 update_plan 调用必须记录到日志
2. 引入 token 预算的预警机制（75% 时警告）
3. 计划执行前先检查剩余 token 是否足够
