# 智能体是否应该暴露计划给用户？

## 面试问题

智能体是否应该暴露计划给用户？

## 考察重点

- 是否能把问题放到「任务规划与执行控制」的工程语境中分析
- 是否能说清楚关键概念背后的系统边界和失败模式
- 是否能给出可落地的设计方法，而不是只背定义
- 是否关注可评测、可调试、可上线和可回滚

## 核心结论

规划的核心是控制任务可靠推进，而不是生成一段看起来完整的计划。回答这个问题时，要同时覆盖核心机制、工程取舍、失败处理和评测方式。

## 参考回答

这个问题不能只从概念层面回答。以「智能体是否应该暴露计划给用户」为例，面试官真正想听的是你是否理解智能体系统如何在真实环境里运行：用户输入进入系统后，系统需要判断目标、组织上下文、选择工具或流程、执行动作、观察结果，并在必要时继续迭代。

在「任务规划与执行控制」这个主题下，重点可以围绕 计划粒度、状态机、重试、回滚、停止条件、目标保持 展开。一个成熟回答通常先说明定义，再说明为什么重要，然后讲如果设计不好会出现什么问题，最后给出工程方案和评测方法。

如果把它放到生产系统里，还需要考虑权限、日志、异常、成本和用户体验。比如系统不能默认相信模型的每一次判断，也不能把所有信息无脑塞进上下文，更不能在没有审计的情况下执行高风险工具。

## 深度展开

### 一、为什么计划暴露是个设计问题

计划暴露不是简单的是/否问题，而是涉及**透明度**、**用户体验**、**系统可控性**三个维度的权衡。

| 维度 | 暴露计划 | 不暴露计划 |
|-----|---------|-----------|
| 用户体验 | 用户知道进展，有预期 | 无负担，但可能困惑 |
| 可控性 | 用户可干预、取消、修改 | 难以中途介入 |
| 实现复杂度 | 需要计划生成、格式化、版本管理 | 相对简单 |
| 典型场景 | 复杂任务、长周期任务 | 简单问答、快速操作 |

**结论**：问题不在于"是否暴露"，而在于"暴露什么"、"何时暴露"、"如何暴露"。

### 二、计划暴露的三种策略

#### 策略一：完全不暴露（隐式规划）

典型系统：纯 ReAct 模式的聊天机器人。

用户发送"帮我重构 auth 模块"，模型直接开始执行，用户只能看到最终结果或错误。

**优点**：
- 实现简单，不需要额外的计划生成逻辑
- 用户体验流畅，没有信息噪音
- 模型可以灵活调整行动

**缺点**：
- 用户无法中途干预
- 复杂任务时用户不知道进展
- 失败时用户无法提供上下文帮助

```python
# 隐式规划的执行循环
class ImplicitPlanner:
    async def execute(self, task: UserTask) -> Result:
        while not task.is_complete():
            action = await self.model.decide(task, self.get_context())
            result = await self.executor.run(action)
            self.update_context(action, result)
        return self.final_result
```

#### 策略二：完全暴露（显式规划）

典型系统：Claude Code、Devin、Coding Agent。

系统会显示完整的计划供用户确认或修改。

**优点**：
- 用户可以中途干预、取消、修改目标
- 复杂任务时用户能保持对系统的信任
- 计划可审计、可回溯

**缺点**：
- 生成计划有延迟（即使是流式）
- 计划可能不准确，需要维护"计划与执行同步"
- 用户可能过度干预反而降低效率

#### 策略三：选择性暴露（混合策略）

根据任务类型、风险等级、用户偏好动态决定是否暴露计划。

```python
class SelectivePlanExposer:
    def should_expose_plan(self, task: UserTask, context: ExecutionContext) -> ExposureDecision:
        # 高风险操作必须暴露计划
        if task.requires_confirmation():
            return ExposureDecision(expose=True, timing="before_execution", detail="full")
        # 简单任务按需暴露
        if task.is_simple():
            return ExposureDecision(expose=False)
        # 复杂任务暴露摘要
        if task.is_complex() and context.user_prefers_transparency:
            return ExposureDecision(expose=True, timing="milestone", detail="summary")
        return ExposureDecision(expose=False)
```

### 三、Claude Code 的计划暴露实现

Claude Code 在 `query.ts` 中实现了完整的计划暴露机制：

```typescript
// query.ts — 显示给用户的计划状态
interface PlanDisplayState {
  currentPlan: string[];      // 当前计划的步骤列表
  completedSteps: string[];   // 已完成的步骤
  nextStepHint: string;      // 即将执行的步骤
  userConfirmed: boolean;     // 用户是否已确认
}
```

Claude Code 的计划暴露时机：

1. **开始执行前**：显示初始计划，等待用户确认（如果有 `claude.md` 中的明确步骤）
2. **关键里程碑**：执行完一个大的步骤后更新显示
3. **用户询问**：用户输入 "what's the plan?" 时返回计划摘要
4. **需要干预时**：检测到问题，询问用户如何处理

```typescript
// 计划暴露的核心逻辑
function formatPlanForUser(plan: Plan, state: ExecutionState): PlanDisplay {
  const completed = plan.steps.filter(s => s.status === 'completed');
  const remaining = plan.steps.filter(s => s.status !== 'completed');

  return {
    summary: `${completed.length}/${plan.steps.length} 步骤完成`,
    currentStep: remaining[0]?.description,
    nextSteps: remaining.slice(1, 4).map(s => s.description),
    estimatedTime: estimateTime(remaining)
  };
}
```

### 四、计划版本管理与冲突处理

当模型可以随时修改计划时，会遇到"计划通胀"问题——计划频繁变化导致用户无法追踪。

#### 方案一：计划快照与 Diff

```python
class PlanVersionManager:
    def __init__(self):
        self.versions: list[PlanSnapshot] = []
        self.max_versions = 10

    def add_version(self, plan: Plan, reason: str):
        snapshot = PlanSnapshot(
            plan=plan,
            timestamp=datetime.now(),
            reason=reason,
            diff=self.compute_diff(self.versions[-1] if self.versions else None, plan)
        )
        self.versions.append(snapshot)
        if len(self.versions) > self.max_versions:
            self.versions.pop(0)

    def format_change_for_user(self, old: Plan, new: Plan) -> str:
        diff = self.compute_diff(old, new)
        lines = ["计划更新:"]
        for step in diff.added:
            lines.append(f"  + {step}")
        for step in diff.removed:
            lines.append(f"  - {step}")
        for step in diff.modified:
            lines.append(f"  ~ {step}")
        return "\n".join(lines)
```

#### 方案二：计划变更频率限制

```python
class PlanChangeThrottler:
    def __init__(self, min_interval_seconds: int = 30, max_changes_per_task: int = 5):
        self.min_interval = min_interval_seconds
        self.max_changes = max_changes_per_task
        self.change_timestamps = []

    def can_change_plan(self, task_id: str) -> bool:
        now = datetime.now()
        # 清理过期记录
        self.change_timestamps = [
            t for t in self.change_timestamps
            if (now - t).seconds < self.min_interval * 2
        ]
        # 检查频率限制
        if len(self.change_timestamps) >= self.max_changes:
            return False
        # 检查时间间隔
        if self.change_timestamps and (now - self.change_timestamps[-1]).seconds < self.min_interval:
            return False
        return True
```

#### 2. 进度展示

```python
class ProgressDisplay:
    def render(self, plan: Plan, state: ExecutionState) -> str:
        completed = sum(1 for s in plan.steps if s.is_completed())
        total = len(plan.steps)
        pct = completed / total * 100

        lines = [
            f"进度: {completed}/{total} ({pct:.0f}%)",
            f"当前: {plan.steps[completed].description if completed < total else '完成'}",
            ""
        ]

        # 显示最近完成的几步
        recent = [s for s in plan.steps[:completed]][-3:]
        if recent:
            lines.append("已完成:")
            for s in recent:
                lines.append(f"  ✓ {s.description}")

        return "\n".join(lines)
```

### 六、工程实践 Checklist

设计计划暴露机制时的 Checklist：

- [ ] **粒度设计**：计划步骤应该多细？太细增加噪音，太粗失去意义
- [ ] **时机选择**：何时显示计划？执行前/中/后？还是按需？
- [ ] **变更同步**：模型修改计划时，用户界面如何同步？
- [ ] **版本历史**：是否需要保留计划变更历史？
- [ ] **确认流程**：高风险操作是否需要用户确认？
- [ ] **进度展示**：如何展示当前进度和剩余时间估算？
- [ ] **干预接口**：用户可以中途干预哪些操作？
- [ ] **降级策略**：当计划系统失败时如何降级？
- [ ] **日志记录**：计划变更是否需要审计？

### 七、真实踩坑案例

#### 案例一：Devin 的"计划幻觉"

某团队实现类似 Devin 的系统时，发现模型会在没有实际执行的情况下生成一个"看起来完整"的计划，并显示给用户。用户以为任务在按计划执行，实际上模型只是在"描述计划"而不是执行计划。

**问题根源**：计划生成和计划执行是分离的，没有强制同步机制。

**修复方案**：
1. 计划必须基于实际执行状态更新
2. 未执行的计划步骤显示为"pending"，执行后变为"completed"
3. 计划执行和计划显示使用同一个状态源

#### 案例二：Claude Code 的响应式压缩导致计划丢失

Claude Code 在上下文压缩时可能会删除历史计划内容，导致用户询问"当前计划是什么"时模型无法准确回答。

**修复方案**：
1. 将计划核心信息（目标、关键步骤）放入压缩免疫区
2. 实现 `plan_summary` 字段，每次计划变更时更新
3. 压缩后从 `plan_summary` 重建计划显示

## 工程实践

- 先定义输入、输出、边界和失败条件，再选择模型或框架
- 对关键决策保留结构化记录，方便调试和评测
- 对高风险动作加入权限校验、人工确认和审计日志
- 用固定测试集验证改动效果，避免只凭单次演示判断质量
- 将失败案例沉淀为回归测试，持续改进系统稳定性

## 常见误区

- 只解释概念，不讲系统如何落地
- 把模型能力当成系统可靠性，忽略工具、权限和评测
- 只关注成功路径，不考虑异常、攻击和成本
- 用一个漂亮演示代替长期可复现的评测结果

## 追问方向

- 如果模型判断错了，系统如何发现并恢复？
- 这个模块的输入输出应该如何结构化？
- 如何设计测试集来验证这个能力？
- 线上失败案例如何回流到评测体系？
- 如果成本、延迟和质量冲突，你会如何取舍？

## 学习建议

准备这个问题时，不要只背定义。建议结合一个具体项目来讲：用户提出任务后，系统如何理解、检索、调用工具、记录轨迹、处理失败并最终评估效果。面试中能把抽象概念落到一条完整执行链路上，会明显更有说服力。
