# 智能体系统为什么特别需要可观测性？

## 面试问题

智能体系统为什么特别需要可观测性？

## 考察重点

- 是否理解智能体系统的非确定性输出导致传统监控手段失效
- 是否能说清楚可观测性与普通日志监控在智能体场景下的本质区别
- 是否能从"调试"、"评测"、"安全"三个维度解释可观测性的价值
- 是否有过实际排查智能体线上问题的经验

## 核心结论

传统软件系统的输出是确定性的——输入相同，输出就相同。智能体系统不同：同样的用户请求，可能因为模型版本、上下文状态、工具返回结果的微小差异而走完全不同的执行路径。没有可观测性，你根本不知道智能体在线上"做了什么"和"为什么这么做"。

## 参考回答

智能体系统本质上是一个**非确定性决策系统**。它每一轮都在做决策——选什么工具、用什么参数、要不要继续——这些决策受模型能力、上下文内容、工具返回结果甚至 prompt 格式的影响，每次都可能不同。

这种非确定性带来了三个传统监控无法解决的问题：

**第一，你不知道智能体做了什么。** 传统的 HTTP 状态码和请求耗时只能告诉你"请求成功了"，但无法告诉你智能体是否走了弯路、是否选错了工具、是否在同一个步骤上循环了三次。

**第二，你不知道智能体为什么这么做。** 智能体的每个决策都基于上下文，而上下文是动态拼装的。如果不记录完整的上下文和决策过程，事故发生后你根本无法定位是模型问题、工具问题还是 prompt 问题。

**第三，你无法预防同类问题。** 传统系统的 bug 修复后通常不会复现，但智能体系统即使修复了 prompt，同样的错误模式可能在下个版本以不同的形式再次出现。没有可观测性数据沉淀，就没办法把线上事故转化为评测用例。

所以智能体系统的可观测性不是"锦上添花"，而是**基础设施**。它同时服务于三个目标：
- **调试**：定位单次执行中的问题
- **评测**：收集线上数据用于离线评估
- **安全**：审计智能体的行为是否符合预期

## 深度展开

### 智能体 vs 传统软件的可观测性需求对比

| 维度 | 传统 Web 服务 | 智能体系统 |
|------|-------------|-----------|
| **输出确定性** | 输入相同，输出必然相同 | 相同输入可能走不同路径 |
| **决策过程** | 代码逻辑固定，可本地复现 | LLM 决策不可复现，每次都可能不同 |
| **失败原因** | 异常栈明确 | 可能是模型、工具、上下文任意环节 |
| **调试方式** | 断点 + 日志 | 需要完整执行轨迹回放 |
| **测试覆盖** | 单元测试可覆盖大部分路径 | 路径空间无限，需要统计评估 |
| **关键指标** | 延迟、错误率、吞吐量 | 任务完成率、工具正确率、循环次数 |

### 为什么传统日志监控不够

传统日志监控假设系统行为是**可枚举的**：

```
2026-05-30 10:00:01 ERROR - Database connection timeout
```

你可以统计这种错误的次数，设置告警阈值。但智能体系统的"错误"不是这种形式：

```
2026-05-30 10:00:01 Agent step 3 - 选择了 analyze_data 工具，参数为 {query: "..."}
2026-05-30 10:00:05 Agent step 4 - 工具返回空结果
2026-05-30 10:00:06 Agent step 5 - 没有重试，直接给出"没有数据"的回答
```

单看每一行都不是"错误"，但合在一起就是一个**漏报数据的问题**。而智能体系统不需要真正的编程错误就会给出有问题的结果。

### 可观测性的三个层次

```
┌─────────────────────────────────────────────┐
│ 第三层：行为审计                              │
│ 谁在什么时候让智能体做了什么操作                │
│ 目标：安全合规、操作追溯                       │
├─────────────────────────────────────────────┤
│ 第二层：聚合分析                              │
│ 在过去1000次执行中，工具调用成功率多少           │
│ 目标：发现趋势问题、评估系统质量                 │
├─────────────────────────────────────────────┤
│ 第一层：单次执行回放                           │
│ 逐步骤回放一次完整的智能体决策过程               │
│ 目标：定位具体问题、理解模型行为                 │
└─────────────────────────────────────────────┘
```

#### 第一层：单次执行回放

这是最基础的需求。当用户反馈"智能体回答错了"，你需要能回答：

1. 用户最终看到了什么？
2. 智能体做了哪些步骤？
3. 每步的输入输出是什么？
4. 模型每一步的决策依据是什么？

#### 第二层：聚合分析

单次回放只能定位具体问题，聚合分析才能发现系统性缺陷：

```python
class AggregatedMetrics:
```

#### 第三层：行为审计

智能体会调用工具、操作文件、发送请求。这些操作需要有完整的审计记录：

```json

    "session_id": "session_xxx",

    "arguments": {"command": "rm -rf /tmp/cache"},
    "approved_by": "auto_policy",
    "result_status": "success",

```

### Claude Code 的可观测性设计

Claude Code 的轨迹记录是一个很好的参考：

```python
class ClaudeCodeTracing:
    """
    Claude Code 中的可观测性实现
    """
    
    def __init__(self):
        self.traces: List[TraceEntry] = []
    
    def record_decision(self, decision: ModelDecision):
        """
        记录模型的每次决策
        """
        entry = TraceEntry(
            step=len(self.traces) + 1,
            input_tokens=decision.input_tokens,
            output_tokens=decision.output_tokens,
            thinking=decision.thinking_content,  # Claude 的思考过程
            tool_call=decision.tool_call,
            timestamp=now()
        )
        self.traces.append(entry)
    
    def record_tool_result(self, result: ToolResult):
        """
        记录工具执行结果，包含截断和摘要
        """
        entry = self.traces[-1]
        entry.tool_result = {
            "status": result.status,
            "stdout_truncated": len(result.stdout) > 4096,
            "summary": self.summarize(result),
            "execution_time_ms": result.duration_ms,
            "exit_code": result.exit_code
        }
    
    def export_trace(self) -> TraceExport:
        """
        导出完整执行轨迹，用于调试和分享
        """
        return TraceExport(
            session_id=self.session_id,
            user_goal=self.initial_goal,
            steps=self.traces,
            final_response=self.final_response,
            total_cost=self.calculate_cost(),
            total_duration_ms=self.calculate_duration()
        )
```

## 工程实践

### 1. 建立执行轨迹标准格式

```typescript
interface TraceEntry {
  step: number;
  timestamp: string;
  type: 'decision' | 'tool_call' | 'observation' | 'response';
  
  // 输入
  model_input?: string;        // 发送给模型的完整 prompt
  tool_args?: Record<string, any>;
  
  // 输出
  model_output?: string;       // 模型返回的原始内容
  tool_result?: ToolResult;
  
  // 元数据
  tokens_in: number;
  tokens_out: number;
  model: string;
  latency_ms: number;
  
  // 决策过程
  reasoning?: string;          // 模型的思考过程（如可用）
  confidence?: number;         // 模型的自信心（如可用）
}
```

### 3. 可观测性系统集成

```python
class ObservabilityPipeline:
    """
    可观测性数据的采集、存储、分析流水线
    """
    
    def __init__(self):
        self.storage = TraceStorage()
        self.analyzer = TraceAnalyzer()
        self.alert_manager = AlertManager()
    
    async def on_agent_step(self, entry: TraceEntry):
        # 1. 实时写入
        await self.storage.append(entry)
        
        # 2. 实时分析（检测循环、异常等）
        alerts = await self.analyzer.realtime_check(entry)
        
        # 3. 触发告警
        for alert in alerts:
            await self.alert_manager.dispatch(alert)
    
    async def on_session_end(self, session: SessionTrace):
        # 1. 持久化完整轨迹
        await self.storage.save(session)
        
        # 2. 异步聚合分析
        asyncio.create_task(
            self.analyzer.aggregate(session)
        )
        
        # 3. 更新监控面板
        await self.dashboard.update(session.metrics)
```

## 常见误区

- **把可观测性和日志监控等同**：日志只记录"发生了什么"，可观测性要回答"为什么发生"和"这是否正常"
- **只关注成功路径**：失败路径的轨迹才是最有价值的调试资料
- **过度记录导致成本失控**：每个 token 都记会导致存储爆炸，需要分级记录和自动清理
- **忽略敏感信息脱敏**：执行轨迹可能包含用户隐私、API key、内部数据，记录时必须脱敏

## 追问方向

- 你的可观测性系统每天产生多少数据？如何管理存储成本？
- 如何区分"模型决策错误"和"上下文信息不足"导致的失败？
- 可观测性数据本身可能成为攻击面（比如注入攻击），你怎么防御？
- 当你发现某个智能体行为异常时，排查的第一步是什么？

## 学习建议

建议动手在自己的智能体项目中加入执行轨迹记录：从最简单的 `console.log` 开始，逐步演进到结构化的 JSON trace 格式。先关注单次执行的回放能力，再逐步建设聚合分析和告警。不要一开始就追求全链路可观测性，从"能复现问题"开始就够了。
