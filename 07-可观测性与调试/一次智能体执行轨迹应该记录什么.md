# 一次智能体执行轨迹应该记录什么？

## 面试问题

一次智能体执行轨迹应该记录什么？

## 考察重点

- 是否能列出执行轨迹必须包含的核心字段，并解释每个字段的价值
- 是否理解不同的受众（开发者、安全审计、产品经理）关心的轨迹数据不同
- 是否在"记录全面"和"控制成本"之间有清晰的取舍策略
- 是否有过实际设计轨迹结构并用于排查问题的经验

## 核心结论

一次执行轨迹应该记录"谁、在什么上下文中、做了哪些决策、每步决策的输入输出是什么、最终结果如何"。最少需要包含：会话元信息、用户的原始目标、每一步的模型输入/输出/决策/工具调用/结果、以及最终的汇总指标。关键原则是：**记录足够让你事后完整复现和理解这次执行的信息，但不记录冗余信息**。

## 参考回答

执行轨迹的内容可以按"层级"来组织，每一层服务于不同的目的。

### 第一层：会话元信息（谁、什么时间、什么模型）

```json
{
  "session_id": "sess_20260530_abc123",
  "user_id": "user_encrypted_hash",
  "started_at": "2026-05-30T10:00:00Z",
  "finished_at": "2026-05-30T10:02:35Z",
  "model": "claude-sonnet-4-20250514",
  "model_config": {
    "temperature": 0,
    "max_tokens": 8192
  },
  "system_prompt_version": "v2.3.1",
  "agent_type": "code_assistant"
}
```

这一层回答：这次执行是谁触发的，用了什么配置。

### 第二层：用户目标（执行的目标是什么）

```json
{
  "user_goal": "帮我重构 user_service.go 中的 GetUser 方法，改用 context 传参",
  "intent_classification": "code_refactoring",
  "intent_confidence": 0.95,
  "attached_files": ["src/services/user_service.go"]
}
```

这一层回答：用户想让智能体做什么。这是后续所有步骤的"评价基准"。

### 第三层：执行步骤序列（核心内容）

每一步包含三个子部分：

```json
{
  "step": {
    "number": 3,
    "type": "tool_call",
    "timestamp": "2026-05-30T10:00:45Z",
    
    "model_input": {
      "messages": [...],        // 完整的 messages 数组（可选，成本高时可省略）
      "tools": [...],           // 此时可用的工具列表
      "tool_choice": "auto"
    },
    "model_output": {
      "content": "...",          // 模型的文本输出
      "tool_calls": [{           // 模型的工具调用请求
        "id": "call_xxx",
        "type": "function",
        "function": {
          "name": "read_file",
          "arguments": "{\"path\": \"src/services/user_service.go\"}"
        }
      }],
      "stop_reason": "tool_use"
    },
    
    "reasoning": "需要先查看 GetUser 方法的当前实现，才能决定如何重构",  // Claude 的思考过程
    
    "execution": {
      "tool_name": "read_file",
      "tool_args": {"path": "src/services/user_service.go"},
      "result": {
        "status": "success",
        "stdout": "package services\n\nfunc GetUser(ctx context.Context, id string)...",
        "truncated": false,
        "size_bytes": 2340
      },
      "duration_ms": 12
    },
    
    "cost": {
      "tokens_in": 4521,
      "tokens_out": 89,
      "cost_usd": 0.0023
    },
    
    "context_changes": {
      "files_read": ["src/services/user_service.go"],
      "files_written": [],
      "commands_run": []
    }
  }
}
```

这一层回答：每一步模型看到了什么、做了什么决策、调用了什么工具、结果如何。

#### 工具调用结果的特殊处理

工具返回结果可能很大（比如 grep 返回 10000 行），需要做分层记录：

```json

    "result_summary": {
      "status": "success",
      "stdout_preview": "src/services/user_service.go:10: func GetUser...\nsrc/services/user_service.go:25: ...",
```

## 深度展开

### 不同角色的关注点不同

| 角色 | 主要关注 | 需要哪些数据 |
|------|---------|------------|
| **开发者** | 定位问题、改进 prompt | 完整轨迹、模型输入输出、工具调用详情 |
| **安全审计** | 是否有越权、数据泄露 | 工具调用参数、访问的文件、执行的命令 |
| **产品经理** | 功能是否正常、用户满意度 | 完成率、步骤数、用户反馈 |
| **运维** | 成本、稳定性 | Token 用量、延迟、错误率 |

### 记录的粒度选择

不是所有轨迹都需要记录所有信息。可以按执行类型分级：

```python
class TraceLevel(Enum):
    """轨迹记录级别"""
    
    MINIMAL = "minimal"
    """只记录元信息 + 汇总指标，用于大规模成本监控"""
    
    NORMAL = "normal"
    """记录步骤序列 + 工具调用摘要，用于日常调试"""
    
    DEBUG = "debug"
    """记录完整模型输入输出 + 思考过程，用于深度排查"""
    
    FULL = "full"
    """记录一切，包括隐式重试、并行调用等，用于模型行为研究"""


def get_trace_level(session: Session) -> TraceLevel:
    """根据会话类型决定记录级别"""
    if session.user_is_internal:
        return TraceLevel.DEBUG
    if session.has_error:
        return TraceLevel.FULL
    if random.random() < 0.05:  # 5% 采样
        return TraceLevel.DEBUG
    return TraceLevel.NORMAL
```

### 轨迹数据的去标识化

记录轨迹时必须考虑隐私。以下几种信息必须处理：

```python
def sanitize_trace(trace: Trace) -> Trace:
    """去除轨迹中的敏感信息"""
    
    # 1. 用户标识：替换为哈希
    trace.user_id = hash(trace.user_id)
    
    # 2. API Key / Token：在工具参数中检测并替换
    for step in trace.steps:
        if "token" in step.tool_args:
            step.tool_args["token"] = "***redacted***"
        if "password" in step.tool_args:
            step.tool_args["password"] = "***redacted***"
    
    # 3. 个人身份信息：文件名、路径中的用户名
    trace.user_goal = remove_pii(trace.user_goal)
    
    # 4. 模型输出中可能包含的敏感信息
    for step in trace.steps:
        step.model_output = sanitize_output(step.model_output)
        if step.tool_result:
            step.tool_result.stdout = sanitize_output(
                step.tool_result.stdout
            )
    
    return trace
```

## 工程实践

### 轨迹记录的核心接口

```typescript
interface Tracer {
  // 创建新轨迹
  startTrace(session: Session): TraceContext;
  
  // 记录步骤开始
  startSpan(context: TraceContext, name: string): SpanContext;
  
  // 记录模型决策
  recordDecision(span: SpanContext, decision: ModelDecision): void;
  
  // 记录工具调用
  recordToolCall(span: SpanContext, call: ToolCall): void;
  
  // 记录工具结果
  recordToolResult(span: SpanContext, result: ToolResult): void;
  
  // 结束步骤
  endSpan(span: SpanContext): void;
  
  // 结束轨迹并输出
  endTrace(context: TraceContext): Trace;
}
```

### 最小可行轨迹实现

```python
@dataclass
class MinimalTrace:
    """最小的可用轨迹结构"""
    session_id: str
    user_goal: str
    steps: List[Dict]
    
    @classmethod
    def from_execution(cls, session):
        """从执行引擎中构建轨迹"""
        trace = cls(
            session_id=session.id,
            user_goal=session.initial_request,
            steps=[]
        )
        return trace
    
    def add_step(self, 
                 action: str,
                 input_data: Dict,
                 output_data: Dict,
                 duration_ms: int,
                 tokens: int = 0):
        self.steps.append({
            "step": len(self.steps) + 1,
            "action": action,
            "input_summary": self._summarize(input_data),
            "output_summary": self._summarize(output_data),
            "duration_ms": duration_ms,
            "tokens": tokens
        })
    
    def _summarize(self, data: Dict) -> str:
        """对输入输出做摘要，避免存太多数据"""
        text = json.dumps(data, ensure_ascii=False)
        if len(text) > 500:
            return text[:250] + "...(truncated)" + text[-250:]
        return text
```

## 常见误区

- **追求"记全"而记了太多无用信息**：模型输入的完整 prompt 可能几百 KB，全部记录会导致存储爆炸。分级记录才是正确的做法
- **只记模型输出不记模型输入**：只知道模型"说/做了什么"但不知道"当时看到了什么"，等于没法复现
- **忽略工具执行的上下文依赖**：工具调用的结果会影响后续的模型输入，这个因果链必须保留
- **不在轨迹中记录版本信息**：没有 prompt 版本号和模型版本号，你无法知道哪个版本的改动导致了行为变化
- **忘记脱敏**：轨迹中可能包含 API key、用户隐私、内部业务数据，不脱敏就是安全事件

## 追问方向

- 如果模型输入的 prompt 有 100K token，你如何决定是否要完整记录？
- 一次 30 步的执行会产生多少轨迹数据？压缩方案是什么？
- 你的轨迹数据存储在什么数据库？如何支持按用户、按时间范围、按结果的快速检索？
- 轨迹数据本身可能包含用户的敏感信息，你如何设计权限控制让不同角色只能看到自己该看的部分？

## 学习建议

从最简单的 JSON 数组开始实践。在自己写智能体项目时，在 agent loop 里加一个 `traces.append(step_data)`，session 结束时输出。先跑几个任务看看每条轨迹包含什么数据、数据量有多大，然后再设计正式的轨迹存储方案。
