# 如何记录词元和成本和延迟？

## 面试问题

如何记录词元和成本和延迟？

## 考察重点

- 是否理解 token 计数、成本计算、延迟测量的基本方法
- 是否能区分"模型推理延迟"和"工具执行延迟"并分别记录
- 是否了解不同模型的计价方式不同，需要不同的成本计算方法
- 是否有通过成本数据分析优化智能体系统的经验

## 核心结论

记录词元、成本和延迟不只是"算算花了多少钱"，而是优化智能体系统的基础。你需要分别记录**模型推理**（token 用量和推理时间）和**工具执行**（调用时间和返回时间）的指标，并且关联到具体的执行步骤。只有这样，你才能回答"哪个环节最花钱"、"哪个步骤最慢"以及"哪里可以优化"。

### 如何计算成本

不同模型的定价不同，需要维护一个定价表：

```python
class CostCalculator:
    """
    模型成本计算器
    """
    
    PRICING = {
        'gpt-4o': {
            'input': 2.50 / 1_000_000,    # $2.50 per 1M input tokens
            'output': 10.00 / 1_000_000,   # $10.00 per 1M output tokens
        },
        'claude-sonnet-4-20250514': {
            'input': 3.00 / 1_000_000,
            'output': 15.00 / 1_000_000,
            'cache_write': 3.75 / 1_000_000,
            'cache_read': 0.30 / 1_000_000,
        },
        'deepseek-chat': {
            'input': 0.27 / 1_000_000,
            'output': 1.10 / 1_000_000,
        },
    }
    
    def calculate(self, model: str, usage: TokenUsage) -> Cost:
        pricing = self.PRICING.get(model)
        if not pricing:
            raise ValueError(f"Unknown model: {model}")
        
        input_cost = usage.prompt_tokens * pricing['input']
        output_cost = usage.completion_tokens * pricing['output']
        
        # 缓存成本（如果模型支持）
        cache_cost = 0
        if hasattr(usage, 'cache_read_tokens') and 'cache_read' in pricing:
            cache_cost = usage.cache_read_tokens * pricing['cache_read']
        if hasattr(usage, 'cache_creation_tokens') and 'cache_write' in pricing:
            cache_cost += usage.cache_creation_tokens * pricing['cache_write']
        
        return Cost(
            input_cost=round(input_cost, 6),
            output_cost=round(output_cost, 6),
            cache_cost=round(cache_cost, 6),
            total_cost=round(input_cost + output_cost + cache_cost, 6)
        )
```

### 如何测量延迟

智能体系统的延迟需要分层测量：

```python
class LatencyTracker:
    """
    分层延迟追踪
    """
    
    def __init__(self):
        self.marks: Dict[str, float] = {}
        self.spans: List[Span] = []
    
    def mark(self, name: str):
        """打点"""
        self.marks[name] = time.time()
    
    def span(self, name: str) -> Span:
        """创建一个时间跨度"""
        span = Span(name=name, start=time.time())
        self.spans.append(span)
        return span
    
    def get_summary(self) -> Dict:
        """获取延迟汇总"""
        if 'start' not in self.marks or 'end' not in self.marks:
            return {}
        
        total = self.marks['end'] - self.marks['start']
        
        return {
            'total_duration_ms': round(total * 1000),
            'breakdown': {
                span.name: round(span.duration_ms)
                for span in self.spans
                if span.duration_ms > 0
            },
            'model_inference_ms': self._sum_span_type('llm_call'),
            'tool_execution_ms': self._sum_span_type('tool_call'),
            'overhead_ms': self._sum_span_type('overhead'),
        }
    
    def _sum_span_type(self, span_type: str) -> float:
        return sum(
            s.duration_ms
            for s in self.spans
            if s.type == span_type
        )
```

## 深度展开

### 智能体系统的成本构成

```
一次 Agent 执行的成本 = 
  Σ(每次模型调用的 token 成本) + 
  Σ(每次工具调用的计算成本) + 
  存储成本（轨迹记录）

其中模型调用成本占 90%+，是最主要的优化对象。
```

#### 成本分析案例

```python
class CostAnalysisReport:
    """
    智能体执行的成本分析报告
    """
    
    @staticmethod
    def analyze_session(trace: Trace) -> CostReport:
        report = CostReport(session_id=trace.session_id)
        
        for step in trace.steps:
            if step.type == 'llm_call':
                report.add_llm_cost(
                    model=step.model,
                    tokens_in=step.tokens_in,
                    tokens_out=step.tokens_out,
                    duration_ms=step.latency_ms
                )
            elif step.type == 'tool_call':
                report.add_tool_cost(
                    tool=step.tool_name,
                    duration_ms=step.execution_time_ms
                )
        
        report.summarize()
        return report
    
    def identify_optimization_opportunities(self) -> List[Optimization]:
        """识别优化机会"""
        opportunities = []
        
        # 检查是否存在大量低价值工具调用
        if self.tool_call_count > 20:
            opportunities.append(
                Optimization(
                    area="tool_efficiency",
                    suggestion="减少不必要的工具调用",
                    expected_savings="30-50%"
                )
            )
        
        # 检查上下文是否过长
        avg_context_tokens = self.total_prompt_tokens / self.llm_call_count
        if avg_context_tokens > 50000:
            opportunities.append(
                Optimization(
                    area="context_compression",
                    suggestion="压缩上下文以降低 token 消耗",
                    expected_savings="20-40%"
                )
            )
        
        return opportunities
```

### 成本/延迟/质量的三角权衡

```python
class AgentOptimizer:
    """
    成本、延迟、质量之间的平衡
    """
    
    @staticmethod
    def balance_strategy(metrics: SystemMetrics):
        """根据当前指标推荐优化策略"""
        
        strategies = []
        
        # 如果成本过高
        if metrics.cost_per_task > 0.50:
            strategies.append("使用更便宜的模型处理简单步骤")
            strategies.append("增加上下文压缩以减少 token")
            strategies.append("对简单任务使用小模型路由")
        
        # 如果延迟过高
        if metrics.p95_latency_ms > 30000:
            strategies.append("并行化独立的工具调用")
            strategies.append("使用 streaming 提升首 token 体验")
            strategies.append("对非关键步骤异步执行")
        
        # 如果质量下降
        if metrics.task_completion_rate < 0.8:
            strategies.append("关键路径使用更强的模型")
            strategies.append("增加验证步骤")
        
        return strategies
```

## 常见误区

- **只看总成本不看分解**：不知道钱花在哪一步，就无法针对性优化。必须按步骤、按工具、按模型分解
- **token 计数不准**：不同平台的 tokenizer 不同（OpenAI 的 tiktoken、Anthropic 的 claude tokenizer），要使用对应的计数方法
- **延迟只记总时间**：模型推理延迟和工具执行延迟混在一起，无法找出瓶颈
- **忽略缓存成本**：部分模型支持 prompt 缓存，缓存读写有专门的计价。不记录缓存命中率，成本分析就不准

## 追问方向

- 如果用户一次会话触发了 20 次模型调用，成本 $2，你如何判断这 $2 花得值不值？
- 你的系统如何实时追踪成本？是按 token 累加还是等模型返回后统一算？
- 当你发现某个工具调用次数特别多但价值不高时，你会怎么做？
- 如何让 product 团队也理解 token 消耗的含义，而不只是给一个"$0.5/次"的数字？

## 学习建议

先在开发环境跑一个简单的成本记录器：每次模型调用后打印 token 数和预估成本，跑一天看看总量。然后逐步增加延迟分解和工具调用成本。不要一开始就搭建复杂的监控系统，先从"我到底花了多少钱"这个最基础的问题开始。
