# 如何定位是规划问题和检索问题还是工具问题？

## 面试问题

如何定位是规划问题和检索问题还是工具问题？

## 考察重点

- 是否有系统化的"问题分类"方法，而不是靠猜测
- 是否能通过执行轨迹中的特征信号来区分三类问题
- 是否理解三类问题的修复手段完全不同，定位错误会导致完全无效的修复
- 是否有实际排查经验

## 核心结论

规划问题、检索问题和工具问题是智能体系统的三类核心故障，它们的表现相似（智能体给出了错误结果），但根因完全不同。定位的方法不是"读代码"，而是**分析执行轨迹中的特征模式**：规划问题通常在决策层表现为"选择了错误的路径"，检索问题表现为"模型看到的上下文信息不足以做决策"，工具问题表现为"模型正确决策但执行失败"。区分这三类问题后，修复手段完全不同——改 prompt、改检索策略、改工具实现——搞错方向就是白费功夫。

## 参考回答

### 三类问题的本质区别

| 维度 | 规划问题 | 检索问题 | 工具问题 |
|------|---------|---------|---------|
| **本质** | 模型决策错了 | 模型信息不够 | 执行环节失败 |
| **表现** | 走了错误的路径 | 基于错误信息做正确决策 | 正确的决策但执行不了 |
| **根因** | prompt / 模型能力 | 文档切分 / 检索策略 / 排序 | 工具实现 / 参数解析 / 权限 |
| **修复手段** | 优化 prompt / 换模型 | 改进检索链路 | 修工具代码 / 改权限 |
| **检测时机** | 决策时 | 决策后 | 执行时 |

### 定位方法论：三段式排查

#### 第一段：从执行轨迹提取特征

```python
def classify_problem(trace: Trace) -> ProblemType:
    """
    通过执行轨迹的特征来判断问题类型
    """
    features = extract_features(trace)
    
    # 特征1：工具调用是否成功
    tool_failure_rate = features['tool_failure_rate']
    
    # 特征2：模型是否在关键决策点做出了合理选择
    decision_quality = features['decision_quality']
    
    # 特征3：最终结果中是否引用了上下文中的信息
    citation_quality = features['citation_quality']
    
    if tool_failure_rate > 0.3:
        return ProblemType.TOOL
    elif decision_quality < 0.5:
        return ProblemType.PLANNING
    elif citation_quality < 0.5:
        return ProblemType.RETRIEVAL
    else:
        return ProblemType.UNKNOWN
```

#### 第二段：逐个验证

**验证是否工具问题**：

```python
def is_tool_problem(trace: Trace) -> bool:
    """工具问题的判断逻辑"""
    for step in trace.steps:
        if step.type == 'tool_call':
            # 特征1：工具返回错误
            if step.tool_result.status == 'error':
                return True
            
            # 特征2：工具返回空或异常值
            if is_unexpected_empty_result(step.tool_result):
                return True
            
            # 特征3：相同的工具调用在不同执行中结果不一致
            # （需要跨轨迹对比）
            
            # 特征4：参数解析失败
            if has_param_parse_error(step):
                return True
    
    return False
```

**验证是否检索问题**：

```python
def is_retrieval_problem(trace: Trace) -> bool:
    """检索问题的判断逻辑"""
    for step in trace.steps:
        # 特征1：模型声称"信息不足"
        if "not enough information" in step.model_output.lower():
            return True
        
        # 特征2：模型引用了不存在或错误的信息
        if has_false_citation(step):
            return True
        
        # 特征3：检索结果和用户问题明显不相关
        if has_irrelevant_retrieval(step):
            return True
        
        # 特征4：模型忽略了检索结果中的重要信息
        if ignored_key_retrieval_result(step):
            return True
    
    return False
```

**验证是否规划问题**：

```python
def is_planning_problem(trace: Trace) -> bool:
    """规划问题的判断逻辑"""
    
    # 特征1：选择了错误的工具
    for step in trace.steps:
        if is_wrong_tool_selection(step):
            return True
    
    # 特征2：步骤顺序不合理
    if has_illogical_step_order(trace):
        return True
    
    # 特征3：没有必要的步骤
    if missing_necessary_steps(trace):
        return True
    
    # 特征4：走了不必要的弯路
    if has_unnecessary_detour(trace):
        return True
    
    # 特征5：循环或放弃
    if detect_looping(trace) or detect_early_abandon(trace):
        return True
    
    return False
```

#### 第三段：隔离验证

当你初步判断是哪类问题后，用隔离验证来确认：

```python
def isolate_and_verify(trace: Trace, hypothesis: ProblemType) -> bool:
    """通过隔离实验验证问题分类"""
    
    if hypothesis == ProblemType.TOOL:
        # 手动调用工具，传入同样的参数，看是否失败
        return manual_test_tool(trace.failed_tool_call)
    
    elif hypothesis == ProblemType.RETRIEVAL:
        # 手动测试检索，传同样的查询，看能召回什么
        retrieved = manual_test_retrieval(trace.retrieval_query)
        return not is_relevant(retrieved, trace.user_goal)
    
    elif hypothesis == ProblemType.PLANNING:
        # 在理想上下文中（提供足够信息），让同一个模型重新决策
        ideal_context = build_ideal_context(trace)
        new_decision = model.decide(ideal_context, trace.available_tools)
        return not is_reasonable(new_decision)
    
    return False
```

## 深度展开

### 三类问题的典型轨迹模式

#### 工具问题的典型轨迹

```
用户："查一下用户张三的余额"
Step 1: search_user("张三") → 成功，返回 {id: 123, name: "张三"}
Step 2: get_balance(user_id: "张三") → 错误！"参数 user_id 应为数字"
Step 3: 重试 get_balance(user_id: 123) → 成功
```

**特征**：工具调用的参数格式错误是最明显的信号。通常是工具定义的参数规范和模型的理解不一致。

**修复**：修复工具的参数 schema，或者在 description 中给出明确的参数示例。

#### 检索问题的典型轨迹

```
用户："我们公司休假政策是什么"
Step 1: search_kb("休假政策") → 返回了 3 个文档
  - 文档1：2025 年年假规定（已过期）
  - 文档2：2026 年考勤制度更新
  - 文档3：产假相关规定
Step 2: 根据文档1回答"年假为 15 天" → ❌ 实际应该是 10 天（2026 年新政策）
```

**特征**：模型引用了上下文中的信息，但信息本身是错的或过时的。模型"正确决策但基于错误信息"。

**修复**：
1. 改进检索策略（关键词搜到了过时文档）
2. 增加时效性过滤
3. 在元数据中标注文档版本/有效期

#### 规划问题的典型轨迹

```
用户："帮我写一个 Python 脚本处理 CSV"
Step 1: read_csv("data.csv") → 成功
Step 2: write_python_script("process.py") → 成功
Step 3: run_shell("python process.py") → 失败
Step 4: search_web("python csv error") → 返回结果
Step 5: edit_file("process.py") → 修复了一个无关的 bug
Step 6: run_shell("python process.py") → 还是失败
...
Step 15: "我无法完成这个任务"
```

**特征**：模型的目标逐渐偏移——从"写脚本"到"修 bug"到"搜索解决方案"，但没有一个完整的规划。问题的根因是模型没有在开始时做充分规划，而是走一步看一步导致迷失。

**修复**：
1. 引入显式的 Plan-and-Execute 模式
2. 在上下文中保持初始计划可见
3. 在关键路径上加入"停下来检查是否偏离目标"的步骤

### 三类问题修复方向的对比

| 问题类型 | 立即修复 | 系统改进 | 长期方案 |
|---------|---------|---------|---------|
| **规划** | 优化 prompt 中的规划指导 | 引入 Plan-and-Execute 模式 | 换用规划能力更强的模型 |
| **检索** | 修改检索查询词、调整 chunk 大小 | 改进切分策略和元数据 | 增加重排和混合检索 |
| **工具** | 修复工具参数 schema、完善 description | 增加参数校验和友好错误提示 | 增加自动重试和参数修正 |

### 一个真实的排查案例

```
用户反馈：智能体查询用户订单时总返回空结果

排查过程：
1. 查看轨迹 → 看到工具调用 search_orders(user_id="张三") 返回了空数组
2. 初步判断：可能是工具问题
3. 隔离测试：手动调用 search_orders(user_id=123) → 返回了订单数据 ✓
4. 再看轨迹 → 模型传的是用户名而不是用户 ID
5. 根因分析：
   - 工具 search_orders 的参数要求 user_id（数字）
   - 模型从对话中拿到的是"张三"（名字）
   - 模型没有先调用 getUserID 再查订单
6. 结论：规划问题（步骤缺失），不是工具问题
7. 修复：在 prompt 中明确要求"查用户数据前先确认用户标识类型"
```

## 工程实践

### 自动化问题分类器

```python
class AutoProblemClassifier:
    """
    自动化智能体故障分类系统
    """
    
    CLASSIFICATION_RULES = [
        # 工具问题规则
        {
            "name": "high_tool_failure",
            "condition": lambda t: t.tool_failure_rate > 0.3,
            "result": ProblemType.TOOL,
            "confidence": 0.8
        },
        {
            "name": "param_errors",
            "condition": lambda t: t.has_param_errors(),
            "result": ProblemType.TOOL,
            "confidence": 0.9
        },
        
        # 检索问题规则
        {
            "name": "irrelevant_retrieval",
            "condition": lambda t: t.has_irrelevant_content(),
            "result": ProblemType.RETRIEVAL,
            "confidence": 0.7
        },
        {
            "name": "outdated_information",
            "condition": lambda t: t.used_outdated_docs(),
            "result": ProblemType.RETRIEVAL,
            "confidence": 0.85
        },
        
        # 规划问题规则
        {
            "name": "wrong_tool_choice",
            "condition": lambda t: t.has_wrong_tool_selection(),
            "result": ProblemType.PLANNING,
            "confidence": 0.75
        },
        {
            "name": "missing_steps",
            "condition": lambda t: t.has_missing_prerequisite_steps(),
            "result": ProblemType.PLANNING,
            "confidence": 0.8
        },
        {
            "name": "looping",
            "condition": lambda t: detect_looping(t),
            "result": ProblemType.PLANNING,
            "confidence": 0.95
        }
    ]
    
    def classify(self, trace: Trace) -> ClassificationResult:
        scores = {
            ProblemType.PLANNING: 0,
            ProblemType.RETRIEVAL: 0,
            ProblemType.TOOL: 0
        }
        
        for rule in self.CLASSIFICATION_RULES:
            if rule["condition"](trace):
                scores[rule["result"]] += rule["confidence"]
        
        if max(scores.values()) == 0:
            return ClassificationResult(
                type=ProblemType.UNKNOWN,
                confidence=0,
                message="No clear pattern detected"
            )
        
        best = max(scores, key=scores.get)
        return ClassificationResult(
            type=best,
            confidence=scores[best],
            message=f"Classified as {best.value} based on {sum(1 for r in self.CLASSIFICATION_RULES if r['condition'](trace))} rules"
        )
```

## 常见误区

- **看到一个错误就认为是工具问题**：工具执行失败往往只是表象，根因可能是模型传了错误的参数（规划问题）
- **检索不到认为就是检索问题**：有时模型根本就没去检索，而是凭自己的知识回答了（规划问题中的步骤缺失）
- **同一个问题反复出现但每次都当新问题修**：积累典型轨迹模式，建立问题分类知识库，让同类问题首次出现就被自动识别
- **定位错误导致修复方向完全反了**：把检索问题当规划问题修，改 prompt 改一百遍也没用

## 学习建议

找一个记录了你智能体执行轨迹的实际日志（如果没有，先加轨迹记录）。随机挑 5 条失败的轨迹，用本节的方法逐一分类——先不急着修，而是练习"分类"这个动作。等你积累了 20-30 条分类经验后，你会发现大多数问题属于少数几种模式，这时候再系统性地修复会更高效。
