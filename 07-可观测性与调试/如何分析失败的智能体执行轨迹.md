# 如何分析失败的智能体执行轨迹？

## 面试问题

如何分析失败的智能体执行轨迹？

## 考察重点

- 是否有系统的故障排查方法论，而不是凭感觉乱试
- 是否能区分"模型能力不足"、"工具设计缺陷"、"上下文信息缺失"这三种不同类型的失败
- 是否了解常见的失败模式（循环、过早放弃、上下文漂移）及其识别方法
- 是否有把失败轨迹转化为测试用例的经验

## 核心结论

分析失败轨迹不是"读日志找异常"，而是有一套方法论的：先确认失败的表现是什么→再沿着执行链回溯找到第一个"不对劲"的决策点→判断这个决策错误的根因（模型、工具、上下文、权限）→最后修复并沉淀为测试用例。关键是**不要跳过"定义失败"这一步**：很多时候智能体其实完成了任务，只是输出格式不符合预期。

## 参考回答

### 第一步：定义"失败"

在执行轨迹分析之前，先明确什么算失败：

```
┌─ 失败类型 ─────────────────────────────────────────────────┐
│                                                             │
│  1. 任务未完成（explicit failure）                            │
│     智能体直接说"我做不到"或进入死循环后超时被终止             │
│                                                             │
│  2. 结果不正确（quality failure）                             │
│     智能体给出了一个答案，但答案是错的                        │
│     需要人工判断，通常通过评测集发现                          │
│                                                             │
│  3. 结果低质量（suboptimal）                                  │
│     答案是对的，但步骤绕路、工具调用不合理、成本过高           │
│     需要通过对比分析发现                                     │
│                                                             │
│  4. 安全违规（safety failure）                                │
│     智能体执行了不应该执行的操作（删文件、越权查询等）          │
│     通常通过审计发现                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 第二步：回溯定位第一个异常决策点

拿到一条失败轨迹后，从最终结果开始逆推：

```python
def analyze_failure_trace(trace: Trace):
    """从最终结果回溯到第一个异常决策"""
    
    # 1. 从最后一步看
    last_step = trace.steps[-1]
    if last_step.type == 'response':
        # 终点是文本回复，检查这个回复是否正确
        check_response_quality(last_step)
    elif last_step.type == 'timeout':
        # 被超时终止，说明最后几步进入了无效循环
        analyze_looping(trace)
    elif last_step.type == 'error':
        # 工具执行报错
        analyze_tool_failure(last_step)
    
    # 2. 从终点回溯，找到"第一个错误决策"
    for i in range(len(trace.steps) - 1, -1, -1):
        step = trace.steps[i]
        
        # 检查当时的决策是否合理
        if not is_reasonable_decision(step):
            return {
                "root_cause_step": i,
                "type": "bad_decision",
                "description": f"第{i+1}步的决策不合理",
                "decision": step.model_output,
                "available_context": step.model_input
            }
        
        # 检查工具调用是否正确
        if step.type == 'tool_call':
            if step.tool_result.status == 'error':
                return {
                    "root_cause_step": i,
                    "type": "tool_error",
                    "description": f"工具 {step.tool_name} 执行失败",
                    "error": step.tool_result.error
                }
            if not is_tool_args_correct(step):
                return {
                    "root_cause_step": i,
                    "type": "bad_args",
                    "description": f"工具 {step.tool_name} 参数错误"
                }
    
    return {"type": "unknown", "needs_deeper_analysis": True}
```

### 第三步：分类根因

找到异常决策点后，判断根因属于哪一类：

```python
def classify_root_cause(step: TraceStep) -> RootCause:
    """判断失败根因的类型"""
    
    # Check 1: 模型问题
    if has_model_reasoning_error(step):
        return RootCause(
            type="model_capability",
            evidence=step.reasoning,
            fix="需要优化 prompt 或尝试更强的模型"
        )
    
    # Check 2: 上下文问题
    if has_context_deficiency(step):
        return RootCause(
            type="context_insufficient",
            evidence=compare_context_and_need(step),
            fix="需要补充上下文或改进检索策略"
        )
    
    # Check 3: 工具设计问题
    if has_tool_design_flaw(step):
        return RootCause(
            type="tool_design",
            evidence=analyze_tool_definition(step.tool_name),
            fix="需要优化工具描述或参数结构"
        )
    
    # Check 4: 权限问题
    if has_permission_issue(step):
        return RootCause(
            type="permission",
            evidence=step.tool_result.error,
            fix="需要调整权限配置或添加用户确认"
        )
    
    # Check 5: 模型对工具的理解问题
    if has_tool_understanding_issue(step):
        return RootCause(
            type="tool_understanding",
            evidence=compare_tool_schema_and_usage(step),
            fix="需要改进工具的描述文档或添加示例"
        )
    
    return RootCause(type="unclassified", needs_manual_review=True)
```

## 深度展开

### 常见的失败模式

#### 模式一：无限循环

识别方法：轨迹中出现重复的"工具调用→结果→再调用"模式。

```python
def detect_looping(trace: Trace) -> bool:
    """检测执行轨迹中是否存在循环"""
    
    tool_call_sequence = [
        (step.tool_name, step.tool_args)
        for step in trace.steps
        if step.type == 'tool_call'
    ]
    
    # 检查最后 N 步是否存在重复模式
    for window_size in range(2, len(tool_call_sequence) // 2):
        windows = [
            tool_call_sequence[-i-window_size:-i]
            for i in range(window_size)
        ]
        if len(set(str(w) for w in windows)) == 1:
            return True  # 发现循环
    
    return False
```

**典型场景**：
```
Step 5: search_web("error X solution")
Step 6: 尝试修复方案 A → 失败
Step 7: search_web("error X solution alternative")
Step 8: 尝试修复方案 B → 失败
Step 9: search_web("error X solution")  ← 回到 Step 5 的搜索
```

**根因**：模型没有"记住"已经尝试过的方案，或者在失败时失去了目标。

**修复**：
1. 在上下文中维护"已尝试方案"列表
2. 增加最大重试次数上限
3. 如果所有已知方案都失败，主动升级用户

#### 模式二：过早放弃

识别方法：轨迹很短（1-3 步），智能体快速给出了"我做不到"的回答。

```python
def detect_early_abandon(trace: Trace) -> bool:
    """检测是否过早放弃"""
    if len(trace.steps) < 3:
        for step in trace.steps:
            if "cannot" in step.model_output.lower() or \
               "unable" in step.model_output.lower() or \
               "sorry" in step.model_output.lower():
                return True
    return False
```

**典型场景**：
```
用户："帮我分析这份 100 页 PDF 中的财务数据"
Step 1: 读取 PDF → 返回 "文件太长（100页），超过上下文限制"
Step 2: "我无法处理这个任务"
```

**根因**：模型遇到了第一个障碍就放弃了，没有尝试替代方案（按章节读取、先抽取目录等）。

**修复**：
1. 在 system prompt 中明确要求"遇到困难时尝试至少 3 种不同的方案"
2. 提供"分块处理"的工具引导
3. 在轨迹中标记"放弃"前是否尝试了多方案

#### 模式三：上下文漂移

识别方法：轨迹前半段和后半段的目标不一致。

```python
def detect_context_drift(trace: Trace) -> bool:
    """检测执行过程中目标是否出现偏移"""
    
    # 用户的原始目标
    original_goal = trace.user_goal
    
    # 分析每个步骤的意图是否偏离原始目标
    for i, step in enumerate(trace.steps):
        current_intent = extract_intent(step.model_output)
        similarity = semantic_similarity(original_goal, current_intent)
        
        # 如果相似度持续下降，可能存在上下文漂移
        if similarity < 0.4 and i > len(trace.steps) * 0.5:
            return True
    
    return False
```

**典型场景**：
```
用户："帮我优化这个 SQL 查询"
Step 1: 查看 SQL → "发现这个表没有索引"
Step 2: 查看表结构 → "可以加索引"
Step 3: 执行 ALTER TABLE ADD INDEX  ← 已经从"优化查询"变成了"改表结构"
```

**根因**：模型在执行过程中被中间结果"带偏"了，忘记了原始目标。

#### 模式四：工具参数错误

识别方法：工具调用轨迹中参数明显不符合预期。

```python
def analyze_bad_params(trace: Trace) -> List[ParamError]:
    """分析工具参数错误"""
    errors = []
    for step in trace.steps:
        if step.type != 'tool_call':
            continue
        
        schema = get_tool_schema(step.tool_name)
        args = step.tool_args
        
        # 检查必填参数是否缺失
        for required in schema.required:
            if required not in args:
                errors.append(ParamError(
                    step=step.number,
                    tool=step.tool_name,
                    issue=f"Missing required param: {required}"
                ))
        
        # 检查参数类型是否正确
        for param_name, param_value in args.items():
            expected_type = schema.properties.get(param_name, {}).get('type')
            if expected_type and not type_matches(param_value, expected_type):
                errors.append(ParamError(
                    step=step.number,
                    tool=step.tool_name,
                    issue=f"Param {param_name}: expected {expected_type}, got {type(param_value)}"
                ))
    
    return errors
```

### 失败分析检查清单

```
□ 1. 确认失败类型：未完成 / 结果错误 / 低质量 / 安全违规
□ 2. 回溯定位第一个异常步骤
□ 3. 判断根因类型：模型 / 上下文 / 工具 / 权限 / 其他
□ 4. 检查是否已知问题（搜索历史轨迹中的相似模式）
□ 5. 修复并验证
□ 6. 将样例加入回归测试集
```

## 工程实践

### 自动失败分类器

```python
class FailureClassifier:
    """
    自动对失败轨迹进行分类和分析
    """
    
    def classify(self, trace: Trace) -> FailureReport:
        report = FailureReport(trace_id=trace.session_id)
        
        # 1. 循环检测
        if detect_looping(trace):
            report.add_finding(
                type="looping",
                severity="high",
                detail=f"Detected cycle of {find_loop_length(trace)} steps",
                suggestion="Add max retry limit or diversify strategy"
            )
        
        # 2. 过早放弃检测
        if detect_early_abandon(trace):
            report.add_finding(
                type="early_abandon",
                severity="medium",
                detail="Agent gave up within 3 steps",
                suggestion="Encourage multiple approaches in prompt"
            )
        
        # 3. 上下文漂移检测
        if detect_context_drift(trace):
            report.add_finding(
                type="context_drift",
                severity="high",
                detail="Goal shifted during execution",
                suggestion="Add goal reminder in context"
            )
        
        # 4. 工具调用分析
        for step in trace.steps:
            if step.type == 'tool_call' and step.tool_result.status == 'error':
                report.add_finding(
                    type="tool_error",
                    severity="medium" if step.tool_result.recoverable else "high",
                    detail=f"Tool {step.tool_name} failed: {step.tool_result.error}",
                    suggestion=generate_tool_fix_suggestion(step)
                )
        
        return report
```

### 轨迹对比分析

```python
def compare_good_and_bad_trace(good: Trace, bad: Trace) -> ComparisonReport:
    """
    对比成功和失败轨迹，定位关键差异
    """
    report = ComparisonReport()
    
    # 1. 步骤数对比
    report.add_diff("total_steps", len(good.steps), len(bad.steps))
    
    # 2. 工具选择对比
    good_tools = Counter(s.tool_name for s in good.steps)
    bad_tools = Counter(s.tool_name for s in bad.steps)
    if good_tools != bad_tools:
        report.add_finding(
            "tool_choice_diff",
            f"Good used {dict(good_tools)}, Bad used {dict(bad_tools)}"
        )
    
    # 3. 关键决策点对比——在同一上下文中，模型的选择是否不同
    for i in range(min(len(good.steps), len(bad.steps))):
        if good.steps[i].model_input == bad.steps[i].model_input:
            if good.steps[i].model_output != bad.steps[i].model_output:
                report.add_finding(
                    "decision_diff",
                    f"Step {i+1}: Same input, different output"
                )
    
    return report
```

## 常见误区

- **只关注"是否完成"不问"完成质量"**：任务完成了但走了 3 倍的路，同样是"失败"，只是表现不明显
- **跳过根因分类直接改 prompt**：盲目改 prompt 可能解决表象问题，但真正的原因可能是工具设计缺陷
- **只分析失败轨迹，不分析成功轨迹**：对比成功和失败的轨迹，往往比单看失败的更有洞察
- **修复一个失败就"算完了"**：没有把失败案例加入回归测试集，同样的错误模式可能以不同形式再次出现

## 学习建议

找一个实际的失败案例，按本节的方法论逐步分析：定义失败类型→回溯定位异常步骤→分类根因→修复→沉淀为测试用例。走完一次完整流程比读十篇文章更有用。多积累几种失败模式后，你会发现很多问题本质上是同一类原因的不同表现。
