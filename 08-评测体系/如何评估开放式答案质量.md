# 如何评估开放式答案质量？

## 面试问题

如何评估开放式答案质量？

## 考察重点

- 是否理解开放式答案的评估不同于选择题（没有唯一正确答案）
- 是否能说清楚 LLM-as-Judge 的优劣势和适用条件
- 是否了解评估开放式答案需要多维度评分而非单一分数
- 是否有设计过开放式答案评估方案的经验

## 核心结论

评估开放式答案没有"标答"可对比，核心方法是 **LLM-as-Judge + 多维度评分**。从正确性、完整性、相关性、清晰度、安全性五个维度打分，并定期用人工评分校准自动评分。关键原则是：评估标准必须先于答案出现——在知道智能体怎么回答之前就定好"什么样的答案算好"。

```python
class OpenEndedEvaluator:
    """
    开放式答案评估器
    """
    
    DIMENSIONS = {
        "correctness": {
            "description": "正确性",
            "weight": 0.35,
            "criteria": "答案是否准确，没有事实错误"
        },
        "completeness": {
            "description": "完整性",
            "weight": 0.25,
            "criteria": "是否覆盖了用户问题的所有方面"
        },
        "relevance": {
            "description": "相关性",
            "weight": 0.15,
            "criteria": "答案是否紧扣问题，没有离题"
        },
        "clarity": {
            "description": "清晰度",
            "weight": 0.10,
            "criteria": "表达是否清晰、结构化、易于理解"
        },
        "actionability": {
            "description": "可操作性",
            "weight": 0.15,
            "criteria": "答案是否给出了可执行的建议或方案"
        }
    }
    
    def evaluate(self, question: str, answer: str) -> Score:
        """
        用 LLM-as-Judge 评估
        """
        prompt = f"""
        你是一个专业的答案评估员。请评估以下智能体的回答。

        用户问题：{question}
        
        智能体回答：{answer}

        请从以下维度评分（1-5分）：
        {self._format_dimensions()}
        
        请输出 JSON：
        {{
            "correctness": <1-5>,
            "completeness": <1-5>,
            "relevance": <1-5>,
            "clarity": <1-5>,
            "actionability": <1-5>,
            "overall": <加权平均>,
            "strengths": ["优点1", "优点2"],
            "weaknesses": ["缺点1", "缺点2"],
            "explanation": "简短评分理由"
        }}
        """
        
        response = self.judge_llm.complete(prompt)
        result = json.loads(response)
        
        # 加权计算总分
        result['weighted_score'] = sum(
            result[dim] * info['weight']
            for dim, info in self.DIMENSIONS.items()
        )
        
        return result
```

### 评分校准

LLM-as-Judge 有偏差，需要定期校准：

```python
class JudgeCalibration:
    """
    LLM-as-Judge 的评分校准
    """
    
    def __init__(self):
        self.human_scores: List[Dict] = []
        self.llm_scores: List[Dict] = []
    
    def add_pair(self, human_score: Dict, llm_score: Dict):
        """记录一次人工评分和 LLM 评分的对比"""
        self.human_scores.append(human_score)
        self.llm_scores.append(llm_score)
    
    def calculate_bias(self) -> BiasReport:
        """计算 LLM 评分偏差"""
        if len(self.human_scores) < 10:
            return None
        
        differences = []
        for h, l in zip(self.human_scores, self.llm_scores):
            for dim in h:
                if dim in l:
                    differences.append({
                        "dimension": dim,
                        "human": h[dim],
                        "llm": l[dim],
                        "diff": l[dim] - h[dim]
                    })
        
        avg_bias = sum(d['diff'] for d in differences) / len(differences)
        
        return BiasReport(
            sample_size=len(self.human_scores),
            avg_bias=avg_bias,
            by_dimension=self._by_dimension(differences),
            recommendation=self._get_recommendation(avg_bias)
        )
```

### 评估结果分析

```python
class EvaluationAnalysis:
    """
    评估结果的分析方法
    """
    
    @staticmethod
    def analyze_results(results: List[Score]) -> Analysis:
        """分析评估结果"""
        scores_by_dim = defaultdict(list)
        for r in results:
            for dim in ['correctness', 'completeness', 'relevance']:
                scores_by_dim[dim].append(r[dim])
        
        return Analysis(
            mean_scores={
                dim: mean(scores)
                for dim, scores in scores_by_dim.items()
            },
            weakness_distribution=self._find_common_weaknesses(results),
            improvement_suggestions=self._suggest_improvements(results)
        )
    
    @staticmethod
    def _find_common_weaknesses(results):
        """找出最常见的弱点"""
        weaknesses = Counter()
        for r in results:
            for w in r.get('weaknesses', []):
                weaknesses[w] += 1
        return weaknesses.most_common(5)
```

## 工程实践

### 混合评估流水线

```python
class HybridEvaluationPipeline:
    """
    自动 + 人工混合评估流水线
    """
    
    def __init__(self):
        self.auto_judge = OpenEndedEvaluator()
        self.human_review_queue = Queue()
    
    async def evaluate(self, test_cases: List[TestCase]):
        results = []
        
        for case in test_cases:
            # 1. 自动评分
            auto_score = self.auto_judge.evaluate(
                case.input, case.response
            )
            
            # 2. 低分或边界情况送人工
            needs_review = (
                auto_score['weighted_score'] < 3.0 or
                3.0 <= auto_score['weighted_score'] <= 3.5
            )
            
            if needs_review:
                await self.human_review_queue.put({
                    "case": case,
                    "auto_score": auto_score,
                    "priority": "high" if auto_score['weighted_score'] < 2.0 else "normal"
                })
                results.append({
                    **case.to_dict(),
                    "status": "pending_human_review",
                    "auto_score": auto_score
                })
            else:
                results.append({
                    **case.to_dict(),
                    "status": "auto_approved",
                    "final_score": auto_score
                })
        
        return results
```

## 常见误区

- **只用 LLM-as-Judge 不用人工校准**：LLM 对自己的评分结果往往过于自信，但对某些类型错误（如事实错误）可能漏判
- **评分维度太少**：一个"总体分"信息量太少，无法指导改进方向。至少需要 3-5 个维度
- **评判标准不一致**：同一批答案让同一个 LLM 评两次可能得到不同分数。需要用温度=0 和固定 prompt
- **忽略评分者的偏差**：LLM 可能偏好格式漂亮但内容空洞的答案，或者偏好和自己观点一致的答案

## 追问方向

- 你如何判断 LLM-as-Judge 本身的评分质量？谁来评估评估者？
- 当 LLM-as-Judge 的评分和用户满意度不一致时，以哪个为准？
- 开放式答案的评估成本比封闭式高多少？你愿意为准确度付出多少成本？
- 如果答案很短但要点都说到了（简洁）和答案很长但有点啰嗦（全面），哪个评分应该更高？

## 学习建议

先定义"什么是好的回答"——针对你的业务场景，列出 3-5 个评估维度。然后找 20 个回答样本，让三个人各评一遍，看评分一致性。一致性差就说明维度定义不够清晰。迭代几次后再引入 LLM-as-Judge。关键不是评分多精确，而是评分能驱动改进。
