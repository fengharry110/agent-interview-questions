# 如何设计工具结构定义？

## 面试问题

如何设计工具结构定义？

## 考察重点

- 是否理解工具定义中各字段（name、description、parameters、required）的作用和工程意义
- 是否能从源码级别理解 Codex 和 Claude Code 的工具定义实现差异
- 是否能设计出能帮助模型正确选择和调用工具的定义结构
- 是否理解 JSON Schema 在工具定义中的应用和局限性

## 核心结论

工具结构定义是模型与外部世界交互的接口契约。设计良好的工具定义能让模型准确理解工具用途、正确填充参数、预估执行结果。核心要素包括：`name`（唯一标识）、`description`（用途说明）、`parameters`（JSON Schema 参数定义）、`strict`（严格校验）以及 `examples`（示例）。设计时需要平衡表达力和模型理解能力，避免过度复杂化。

## 工具定义的核心要素

### 一、完整工具定义结构

```json
{
  "type": "function",
  "name": "database_query",
  "description": "Executes a read-only SQL query on the analytics database. Use this when you need to retrieve data for analysis, generate reports, or answer questions about business metrics. Returns up to 1000 rows.",
  "strict": true,
  "parameters": {
    "type": "object",
    "properties": {
      "sql": {
        "type": "string",
        "description": "A read-only SELECT query. Do NOT include INSERT, UPDATE, DELETE, DROP, or ALTER statements.",
        "examples": ["SELECT * FROM orders WHERE created_at > '2024-01-01'", "SELECT product_id, COUNT(*) as cnt FROM sales GROUP BY product_id"]
      },
      "params": {
        "type": "object",
        "description": "Optional named parameters for the query",
        "additionalProperties": { "type": "string" }
      }
    },
    "required": ["sql"]
  }
}
```

### 二、各字段的工程意义

| 字段 | 作用 | 设计注意点 |
|-----|------|-----------|
| `type` | 固定为 `function`，告诉模型这是可调用函数 | 必须有，部分系统据此路由 |
| `name` | 全局唯一标识，避免命名冲突 | 使用 snake_case 或 camelCase |
| `description` | **最关键字段**，直接影响模型是否选对工具 | 需要清晰说明用途、适用场景、限制 |
| `strict` | 是否严格校验参数类型 | `true` 更安全但模型可能出错，`false` 更宽容 |
| `parameters` | JSON Schema 定义参数类型和约束 | 影响模型参数填充准确性 |
| `required` | 必填参数列表 | 避免模型漏填关键参数 |
| `examples` | 参数示例 | 帮助模型理解正确格式 |
| `default` | 默认值 | 模型可能忘记传这个参数 |

## Description 的设计艺术

### 好的 description 应该包含

1. **工具用途**：这个工具做什么
2. **适用场景**：什么时候应该用这个工具
3. **返回内容**：调用后会得到什么
4. **限制条件**：什么情况下不能用
5. **避免误用**：常见错误用法及正确方式

### Claude Code 的 description 设计

Claude Code 的 `Read` 工具定义：

```json
{
  "name": "Read",
  "description": "Read file from the filesystem. Use this tool when you need to examine the contents of a file. Returns the file contents as a string. For large files, use Grep or Read to scan for specific content."
}
```

关键设计点：
- 明确"做什么"（读取文件）
- 明确"什么时候用"（需要检查文件内容时）
- 说明"返回什么"（文件内容作为字符串）
- 提到相关工具（Grep），暗示分工

### Codex 的 description 设计

Codex 的 `update_plan` 工具定义：

```json
{
  "name": "update_plan",
  "description": "Updates the task plan. Call this when you need to modify the plan based on new information or completed steps. Always provide an explanation for why the plan is changing."
}
```

关键设计点：
- 明确触发条件（基于新信息或完成步骤时）
- 要求解释（防止模型随意修改计划）

## Parameters 的设计

### 基础参数类型

```json
{
  "type": "object",
  "properties": {
    "filename": {
      "type": "string",
      "description": "Name of the file to operate on"
    },
    "content": {
      "type": "string",
      "description": "Content to write or search for"
    },
    "line_count": {
      "type": "integer",
      "description": "Maximum number of lines to return",
      "minimum": 1,
      "maximum": 1000
    },
    "enabled": {
      "type": "boolean",
      "description": "Whether this feature is active"
    }
  }
}
```

### 数组参数

```json
{
  "type": "object",
  "properties": {
    "files": {
      "type": "array",
      "description": "List of file paths to process",
      "items": {
        "type": "string"
      },
      "minItems": 1,
      "maxItems": 100
    },
    "options": {
      "type": "array",
      "description": "Configuration options",
      "items": {
        "type": "string",
        "enum": ["verbose", "recursive", "ignore_errors"]
      }
    }
  }
}
```

### 嵌套对象参数

```json
{
  "type": "object",
  "properties": {
    "query": {
      "type": "object",
      "description": "Search query configuration",
      "properties": {
        "text": {
          "type": "string",
          "description": "Text to search for"
        },
        "case_sensitive": {
          "type": "boolean",
          "default": false
        },
        "use_regex": {
          "type": "boolean",
          "default": false
        }
      },
      "required": ["text"]
    }
  }
}
```

### 常见错误

1. **参数过于复杂**：嵌套超过 3 层，模型难以理解
2. **缺少 default**：可选参数没有默认值，模型可能传 `null`
3. **类型模糊**：使用 `any` 或不指定 type
4. **约束不足**：没有 min/max/length 限制，收到无效大对象

## 如何处理工具数量过多

当系统有 20+ 工具时，模型容易选错。解决方案：

### 方案一：工具分组

```json
{
  "name": "file_operations",
  "description": "Group of file operations. Use Read for viewing files, Write for creating/modifying files, Grep for searching within files.",
  "subtools": ["Read", "Write", "Grep", "Edit"]
}
```

模型先选择组，再选择具体工具。

### 方案二：必选 vs 可选工具

只向模型暴露当前任务相关的工具：

```python
def get_tools_for_task(task: Task, context: ExecutionContext) -> list[Tool]:
    base_tools = ["Read", "Write", "Shell"]  # 始终可用

    if task.requires_code_execution():
        base_tools.append("Bash")

    if task.requires_web_access():
        base_tools.append("WebSearch")

    return [tool_registry.get(name) for name in base_tools]
```

### 方案三：分层层级

```
Level 1: 任务类型分类
  - coding
  - data_analysis
  - web_browsing
  - document_processing

Level 2: 领域工具
  - coding: [Read, Write, Edit, Bash, Git]
  - data_analysis: [SQLQuery, DataViz, Export]

Level 3: 具体操作
  - Read: [ReadFile, ReadLines, Glob]
```

## 工具定义的版本管理

### 变更影响评估

修改工具定义时需要评估对现有智能体的影响：

```python
class ToolDefinitionVersionManager:
    def assess_impact(self, old_def: ToolDef, new_def: ToolDef) -> ImpactReport:
        impacts = []

        if old_def.name != new_def.name:
            impacts.append(Impact(type="breaking", detail="Tool name changed"))

        if self.param_changed(old_def, new_def, "required"):
            impacts.append(Impact(type="breaking", detail="Required params changed"))

        if self.param_added(new_def):
            impacts.append(Impact(type="minor", detail="New optional param added"))

        if self.description_significantly_changed(old_def, new_def):
            impacts.append(Impact(type="medium", detail="Description changed may affect model behavior"))

        return ImpactReport(impacts=impacts, migration_required=any(i.type == "breaking" for i in impacts))
```

### 向后兼容策略

1. **添加可选参数**：始终安全
2. **添加必填参数**：需要版本升级流程
3. **修改参数名**：需要别名支持
4. **修改 description**：灰度测试，监控召回率

## 评测工具定义质量

### 评测指标

```python
@dataclass
class ToolDefinitionMetrics:
    selection_accuracy: float    # 模型选对工具的比例
    param_fill_accuracy: float   # 模型填对参数的比例
    invalid_call_rate: float     # 无效调用（选错工具或参数）比例
    description_clarity: float  # 人工评估 description 清晰度

def evaluate_tool_definition(tool: Tool, test_cases: list[TestCase]) -> ToolDefinitionMetrics:
    results = []
    for case in test_cases:
        # 让模型选择工具并填充参数
        model_output = model.predict(case.user_input, available_tools=[tool])
        expected_tool, expected_params = case.expected

        results.append(CallResult(
            tool_correct=model_output.selected_tool == expected_tool,
            params_correct=model_output.params == expected_params
        ))

    return ToolDefinitionMetrics(
        selection_accuracy=sum(r.tool_correct for r in results) / len(results),
        param_fill_accuracy=sum(r.params_correct for r in results) / len(results),
        invalid_call_rate=1 - (sum(r.tool_correct for r in results) / len(results))
    )
```

### A/B 测试工具定义

当不确定哪个 description 更好时：

```python
def ab_test_tool_definition(
    variant_a: ToolDef,
    variant_b: ToolDef,
    traffic_split: float = 0.5
) -> VariantResult:
    test_tasks = load_balanced_test_set()

    # A 组使用 variant_a
    results_a = run_evaluation(variant_a, test_tasks[:len(test_tasks)//2])

    # B 组使用 variant_b
    results_b = run_evaluation(variant_b, test_tasks[len(test_tasks)//2:])

    return VariantResult(
        winner="a" if results_a.selection_accuracy > results_b.selection_accuracy else "b",
        improvement=abs(results_a.selection_accuracy - results_b.selection_accuracy)
    )
```

## 工程实践 Checklist

设计工具定义时的 Checklist：

- [ ] **name 唯一性**：工具名是否全局唯一，避免歧义
- [ ] **description 完整性**：是否包含用途、场景、限制、返回
- [ ] **description 简洁性**：是否避免过长的 description（模型会截断）
- [ ] **parameters 类型明确**：是否每个参数都有明确的 type
- [ ] **parameters 有约束**：是否有 min/max/length 防止大对象
- [ **required 合理**：必填参数是否真正必要
- [ ] **examples 有代表性**：示例是否覆盖典型用法
- [ ] **工具数量控制**：是否控制在模型能处理的范围内（<20）
- [ ] **版本兼容性**：修改是否考虑了向后兼容

## 真实踩坑案例

### 案例：description 过长导致模型选错工具

某团队发现模型频繁选错 `Read` 和 `Grep` 工具。检查发现 `Read` 的 description 长达 500 字，描述了各种边界情况，导致模型无法快速理解其核心用途。

**修复方案**：
1. 将 description 压缩到 100 字以内
2. 将边界情况移到参数的 description 中
3. 保持工具之间 description 长度相近

### 案例：参数类型过于宽泛

某团队定义的 `query` 参数 type 为 `string`，但实际只需要 SQL 查询语句。模型有时传入 "Show me all users" 这样的自然语言。

**修复方案**：
1. 将 type 改为特定格式字符串
2. 在 description 中明确说明期望的格式
3. 添加正则约束 `pattern: "^SELECT\\s+.*"`
