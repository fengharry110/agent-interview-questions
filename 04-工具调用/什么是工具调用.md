# 什么是工具调用？

## 面试问题

什么是工具调用？

## 考察重点

- 是否理解工具调用本质上是"让模型生成符合 schema 的结构化数据"，而不是"模型在执行代码"
- 是否能说清楚工具定义（JSON Schema）各字段的作用，以及如何避免模型选错工具
- 是否了解 Codex 和 Claude Code 的工具实现差异，以及 Tool Use vs Tool Call 的区别
- 是否能从源码级别理解工具调用的执行流程

## 核心结论

工具调用不是让模型"执行代码"，而是让模型生成一个**符合预定义 JSON Schema 的结构化请求**。模型输出的工具调用（如 `{"name": "shell", "arguments": {"command": "ls"}}`）只是数据，真正的执行由智能体框架完成。理解这一点，才能理解为什么工具调用会失败，以及如何设计可靠的工具。

## 工具调用的本质

### 不是"模型执行代码"，而是"模型生成结构化数据"

当模型输出：
```
{
  "name": "shell",
  "arguments": {"command": "ls -la"}
}
```

这**不是**模型在执行 `ls` 命令。这只是模型生成的一个 JSON 对象。真正的执行流程是：

1. **生成阶段**：模型根据上下文生成工具调用请求（JSON）
2. **解析阶段**：智能体框架解析这个 JSON，提取工具名和参数
3. **执行阶段**：框架调用实际的函数/命令
4. **返回阶段**：执行结果被转回文本，追加到上下文

### Tool Call vs Tool Use

| 概念 | 含义 |
|-----|------|
| **Tool Call** | 模型生成调用请求的动作（模型视角） |
| **Tool Use** | 框架执行工具并返回结果（系统视角） |
| **Function Calling** | OpenAI API 中的具体实现机制 |

有些系统（如 Anthropic）用 "Tool Use"，有些（如 OpenAI）用 "Function Calling"，但本质相同。

## 工具定义：JSON Schema 的工程细节

### Codex 的 shell 工具定义

以下是一个完整可用的工具定义，来自 Codex 源码：

```json
{
  "type": "function",
  "name": "shell",
  "description": "Runs a shell command and returns its output...",
  "strict": false,
  "parameters": {
    "type": "object",
    "properties": {
      "command": {
        "type": "array",
        "description": "The command to execute",
        "items": {"type": "string"}
      },
      "workdir": {
        "description": "The working directory...",
        "default": "当前工作目录"
      },
      "timeout_ms": {
        "description": "The timeout for the command...",
        "default": 60000
      }
    },
    "required": ["command"]
  }
}
```

关键字段解析：

| 字段 | 作用 | 设计注意点 |
|-----|------|-----------|
| `type` | 固定为 `function` | 告诉模型这是一个可调用的函数 |
| `name` | 工具唯一标识 | 必须全局唯一，避免命名冲突 |
| `description` | 工具用途说明 | **最重要**的字段，直接影响模型选择 |
| `strict` | 是否严格校验参数 | `false` 更宽容但可能收到垃圾参数 |
| `parameters` | JSON Schema | 定义参数类型和约束 |
| `required` | 必填参数列表 | 避免模型漏填关键参数 |
| `default` | 默认值 | 模型可能"忘记"传这个参数 |

### description 的艺术

`description` 是决定模型是否选对工具的关键。好的 description 应该：

1. **明确工具的边界**：这个工具能做什么，不能做什么
2. **给出使用示例**：模型能从示例中学到何时该用
3. **标注危险操作**：如果工具有副作用，必须明确说明

**反面例子**：
```json
"description": "执行命令"
```

**正面例子**：
```json
"description": "在本地 shell 中执行命令。用于文件操作、运行脚本、查看进程等。\n\n危险：此命令会真正执行，具有副作用。请确认命令不会破坏系统。\n\n示例：\n- \"ls -la\" 查看目录内容\n- \"git status\" 查看 Git 状态\n- \"npm run build\" 构建项目"
```

## Codex 的工具生态

### 内置工具类型

Codex 的工具来自三个来源：

```json
[
  // 1. Codex 内置工具
  {
    "type": "function",
    "name": "shell",
    "description": "Runs a shell command..."
  },
  {
    "type": "function",
    "name": "update_plan",
    "description": "Updates the task plan..."
  },

  // 2. Responses API 提供的工具
  {
    "type": "web_search",
    "external_web_access": false
  },

  // 3. MCP 服务器提供的工具
  {
    "type": "function",
    "name": "mcp__weather__get-forecast",
    "description": "Get weather alerts for a US state"
  }
]
```

### MCP 工具的命名规范

MCP（Model Context Protocol）工具的命名采用 `mcp__<server>__<method>` 格式：
- `mcp__weather__get-forecast` = MCP 服务器 `weather` 的 `get-forecast` 方法

这种命名方式：
1. 避免不同来源工具的命名冲突
2. 明确标识工具来源（本地 MCP vs 远程 API）
3. 便于权限分级和管理

## 工具调用的执行流程

### Step by Step

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 模型推理                                                 │
│    输入: "列出当前目录的文件"                                │
│    输出: ToolCall { name: "shell", arguments: { cmd: ["ls"] } }
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. 框架解析                                                 │
│    解析 JSON，提取 name="shell" 和 arguments                │
│    验证 arguments 符合 shell 工具的 JSON Schema            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. 权限检查                                                 │
│    - shell 命令是否在白名单内？                              │
│    - 命令是否有危险关键词（如 rm -rf）？                      │
│    - 是否需要用户确认？                                      │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 4. 执行                                                     │
│    调用真正的 shell（forkexec），捕获 stdout/stderr         │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 5. 结果转换                                                 │
│    将 stdout/stderr 转换为文本，追加到上下文                  │
│    格式: "Command executed successfully.\nOutput:\n<output>"
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 6. 再次推理                                                 │
│    模型基于新的上下文决定下一步                               │
└─────────────────────────────────────────────────────────────┘
```

### Claude Code 的权限分级

Claude Code 实现了 **7 种权限模式**，核心是区分"安全操作"和"危险操作"：

```typescript
// 权限级别示例
enum PermissionMode {
  ACCEPT_ALL,        // 所有操作直接执行
  ACCEPT_SAFE,      // 安全操作直接执行，危险操作需确认
  DENY_ALL,          // 所有操作都需确认
  READ_ONLY,         // 只读操作，无危险
  // ... 更多模式
}
```

危险操作（如删除文件、执行外部命令）会触发：
1. **ML 分类器**：判断操作是否危险
2. **用户确认**：暂停执行，等待用户输入
3. **审计日志**：记录所有操作及其上下文

## 如何减少模型选错工具

### 问题原因

模型选错工具通常因为：
1. **工具描述不清**：模型无法区分两个相似工具
2. **工具数量过多**：选择疲劳，模型随便选一个
3. **参数设计缺陷**：模型不知道如何正确传参

### 解决方案

#### 1. 工具数量控制

人类一次能在 Working Memory 中处理 5-7 个选项。模型类似。

| 工具数量 | 建议策略 |
|---------|---------|
| ≤ 5 | 直接全部暴露给模型 |
| 5-15 | 按场景分组，用 `oneOf` 让模型先选组 |
| > 15 | 实现**工具路由**，模型先描述意图，路由再选工具 |

#### 2. 提升 description 质量

```json
{
  "name": "read_file",
  "description": "读取文件内容。\n\n使用场景：\n- 当你需要查看代码、日志、配置文件时\n- 当你需要了解某个模块的实现时\n\n限制：\n- 不适合读取大文件（>1MB），大文件请用 search_files\n- 二进制文件无法读取"
}
```

#### 3. 参数设计原则

| 原则 | 说明 |
|-----|------|
| 参数尽量少 | 必填参数 ≤ 3 个 |
| 使用 default | 可选参数提供默认值 |
| 类型明确 | 使用 enum 而非 string 当有固定选项 |
| 避免嵌套 | 嵌套参数让模型容易出错 |

#### 4. 添加 tool_choice 约束

在 API 层面可以强制要求模型使用某个工具：

```python
# OpenAI API 示例
response = client.chat.completions.create(
    model="gpt-4",
    messages=[...],
    tools=[...],
    tool_choice={"type": "function", "function": {"name": "my_tool"}}
)
```

## 工具返回值的结构化设计

### 问题：原始返回值太大

Shell 工具可能返回大量输出，直接塞进上下文会：
1. 消耗大量 token
2. 淹没关键信息
3. 让模型"迷失"在细节中

### 解决方案：截断 + 摘要 + 存储

```json
{
  "tool": "shell",
  "input": {
    "command": "ls -la /var/log",
    "workdir": "/home/user"
  },
  "output": {
    "status": "success",
    "stdout": "total 128\ndrwxr-xr-x  12 root root  4096 May 29 10:00 app.log\ndrwxr-xr-x   2 root root  4096 May 29 10:00 error.log",
    "stdout_truncated": true,
    "truncation_reason": "output_exceeded_4kb",
    "summary": "列出了 /var/log 目录，共 2 个日志文件"
  },
  "metadata": {
    "execution_time_ms": 23,
    "exit_code": 0
  }
}
```

关键设计：
1. **截断**：超过阈值就截断，保留首尾各一部分
2. **摘要**：自动生成一句话总结
3. **元数据**：返回执行时间、退出码等，便于调试

## 面试追问方向

1. **"如果模型输出 `{"name": "delete_file", "arguments": {"path": "/"}}`，你的系统会怎么处理？"**

   考察候选人对危险的认知和防御措施。

2. **"工具 A 和工具 B 都能完成某个任务，但 A 返回的是结构化 JSON，B 返回的是自然语言。你如何决定用哪个？"**

   考察候选人对工具设计的权衡理解。

3. **"Codex 用 `command: ["string"]` 而你用 `command: "string"`，为什么数组更好？"**

   考察候选人对参数设计的细节理解。

4. **"模型说它要调用工具，但实际上不需要（比如只是查询信息）。你如何避免过度工具调用？"**

   考察候选人对工具调用成本和效率的考虑。

5. **"如果模型的工具调用一直失败（参数格式错误、权限不足等），你如何定位问题？是模型的问题还是工具设计的问题？"**

   考察候选人对问题定位和系统调试的理解。

## 真实踩坑案例

### 案例：工具描述引发的"删库"危机

某团队的删除文件工具有这样的描述：
```json
"description": "删除指定路径的文件"
```

模型在执行"清理项目中的冗余文件"时，误判了哪些是"冗余"的，结果调用删除工具把整个 `node_modules` 删了。

**修复后的描述**：
```json
"description": "删除指定路径的文件。\n\n警告：此操作不可恢复！\n\n安全检查：\n1. 只删除项目根目录下的临时文件\n2. 不会删除 node_modules、.git 等关键目录\n3. 单次最多删除 10 个文件\n\n恢复方式：需要手动从 git 恢复"
```
