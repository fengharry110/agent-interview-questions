# 如何避免被认为是大模型套壳？

## 面试问题

你的项目怎么证明不是大模型套壳？如果面试官觉得你只是调了个 API，你怎么解释？

## 考察重点

- 是否能清晰区分"套壳"和"做系统"的边界
- 是否能从技术维度证明项目的工程深度
- 是否可以举出具体的非模型相关的工作量
- 是否理解面试官质疑套壳背后的真实顾虑
- 是否能在简历和面试中主动消除这种印象

## 核心结论

面试官怕的不是你调 API，怕的是你**只会调 API**。证明你不是套壳，核心是展示你在 Agent Loop、工具工程、评测体系、安全控制和系统架构五个方面有独立的工程贡献。

## 参考回答

"套壳"在行业内的定义是：直接调用模型 API，不做任何系统工程。换句话说，套壳项目 = 模型能力 = 项目能力。证明你不是套壳，关键是要展示你的项目中存在大量**与模型无关的工程工作**。

### 面试官判断套壳的三个信号

```
信号一：简历上只有 "基于 GPT-4"，没有架构设计
信号二：项目难点都是模型自身的问题（幻觉、遗忘、跑题）
信号三：回答问题时只能重复模型的能力，说不出系统设计
```

### 反套壳的六大技术维度

| 维度 | 套壳项目 | 工程化项目 | 证明方式 |
|------|---------|-----------|---------|
| Agent Loop | 用框架默认循环 | 自研控制逻辑，自定义停止条件 | 展示代码 |
| 工具系统 | 写死工具调用 | 动态注册、权限分级、参数校验 | 架构图 |
| 上下文管理 | 无脑塞全部历史 | 智能压缩、摘要、选择性保留 | 压缩策略 |
| 评测体系 | 手工测试几个 Case | 自动化评测流水线、回归用例 | 评测报告 |
| 安全控制 | 无权限校验 | 工具权限、数据隔离、审计日志 | 设计文档 |
| 可观测性 | 只有 print 日志 | 链路追踪、指标监控、错误分类 | 监控面板 |

### 面试表述示例

```
面试官：你这个项目用了什么模型？
候选A：用的 GPT-4，效果很好。（套壳信号）
候选B：主力模型是 GPT-4，但我们做了三层工程优化：

  第二层是工具描述的自动生成和 A/B 测试，把工具选择准确率从 82% 提升到 96%；
  第三层是上下文的 4 阶段压缩策略，支撑 50+ 轮长对话。
  模型是我们的组件之一，但不是全部。（反套壳）
```

## 深度展开

### 套壳的三种层次

**第一层：纯套壳（Less Than API）**
直接调用 API，不做任何处理。典型特征：
- 项目 = 一个 OpenAPI 调用 + 一个 Web 界面
- 所有"智能"来自模型
- 模型换掉，项目归零

**第二层：半套壳（API + 简单编排）**
使用了 LangChain / AutoGPT 等框架，但仅限于框架的基本用法。典型特征：
- 项目 = 框架预设流程 + Prompt 调优
- 难点集中在 Prompt 工程
- 框架换掉，系统归零

**第三层：工程化（API + 系统工程）**
把模型当作组件之一，围绕模型构建完整的工程体系。典型特征：
- 自研系统架构，框架只是辅助
- 有独立的工程价值（工具系统、评测、安全）
- 模型换掉，大部分系统仍然有效

### 面试官的真实顾虑

面试官质疑"套壳"，背后是五个更本质的问题：

1. **可迁移性**：如果换了模型，你的系统还剩什么？
2. **深度**：你理解底层原理，还是只停留在调用层？
3. **工程质量**：你的系统经得起生产环境的考验吗？
4. **解决问题**：当模型能力不够时，你能用工程手段弥补吗？
5. **成长潜力**：你未来能解决更难的问题，还是只能调 API？

  const closing = "您想深入了解哪一块的具体实现？";

## 工程实践

### 项目去套壳化的检查清单

```python
def anti_wrapping_audit(project: dict) -> dict:
    """审计项目是否有套壳风险"""
    result = {}
    
    # 1. 系统架构独立性
    result["has_custom_agent_loop"] = project.get("agent_loop") == "custom"
    result["has_tool_framework"] = "tool_registry" in project.get("components", [])
    result["has_eval_system"] = project.get("eval_system") is not None
    
    # 2. 与模型无关的代码量
    non_model_code = project.get("total_code_lines", 0) - project.get("prompt_lines", 0)
    result["non_model_code_ratio"] = non_model_code / max(project.get("total_code_lines", 1), 1)
    
    # 3. 工程设施
    result["has_test_suite"] = project.get("test_count", 0) > 50
    result["has_monitoring"] = project.get("has_monitoring", False)
    result["has_cicd"] = project.get("has_cicd", False)
    
    # 4. 容错与边界处理
    result["has_retry"] = project.get("has_retry", False)
    result["has_circuit_breaker"] = project.get("has_circuit_breaker", False)
    result["has_fallback"] = project.get("has_fallback", False)
    
    # 评分
    score = sum(1 for v in result.values() if v) / len(result) * 100
    result["anti_wrapping_score"] = f"{score:.0f}%"
    
    return result
```

## 常见误区

- **过度强调 Prompt 工程**：面试官不认为写 Prompt 是高价值工作，不要把它作为项目的主要难点
- **贬低框架来抬高自己**：说"LangChain 太烂所以我自己写"，面试官会认为你缺乏判断力
- **只展示技术不展示业务**：纯技术项目容易被当成 Demo，要结合业务场景说明价值
- **忽略"为什么不用框架"的解释**：说自研但不说理由，面试官会认为你重复造轮子
- **模型选型不合理**：用 GPT-4 做简单的分类任务，面试官会觉得你缺乏成本意识
- **摆很多代码但不解释思路**：代码不能直接证明工程能力，面试官要听的是设计思路

## 追问方向

- **"如果你换一个模型（比如从 GPT-4 换到 Claude 3），你的系统哪些需要改？"**
  考察系统对模型的耦合度。回答好的人会说出哪些是模型无关层、哪些是模型相关层。

- **"你说自己实现了 Agent Loop，和 LangChain 的实现有什么区别？"**
  考察对框架底层原理的理解。需要有具体的差异点，而不是"我的更灵活"。

- **"如果模型有一天不需要 Tool Calling 了（原生支持），你的系统还有价值吗？"**
  考察工程价值的独立性。好回答是说工具注册、权限、评测这些非模型部分仍然有效。

- **"你的系统在模型能力不够时如何兜底？"**
  考察工程容错设计，比如降级到规则引擎、人工接管、明确告知用户能力边界。

- **"你们项目团队几个人？你负责哪部分？"**
  考察个人贡献。要清晰说明自己的角色和交付物。

## 学习建议

准备这个问题最好的方式是在自己的项目中刻意增加"非模型"的工程工作。加一个工具注册框架、搭一套评测流水线、设计权限体系——这些都是低投入高回报的工程化动作。面试时主动引导到"工程"而非"模型"的话题上，用"我们做了 X，解决了 Y，达到了 Z"的句式，而不是"模型可以做到 X"。如果简历上能体现出 60% 以上是非模型工程代码量，套壳质疑基本不会出现。
