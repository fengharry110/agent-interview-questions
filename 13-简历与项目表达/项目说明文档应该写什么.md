# 项目说明文档应该写什么？

## 面试问题

你的智能体项目的说明文档里应该写什么内容？怎么让读者快速理解你的项目？

## 考察重点

- 是否有撰写技术文档的习惯和能力
- 是否能区分面向不同读者的文档内容
- 是否能结构化地组织项目信息
- 是否理解文档质量对项目影响力的作用
- 是否在文档中体现了工程思维和设计决策

## 核心结论

好的项目说明文档不是功能的罗列，而是**面向不同读者提供不同层次的信息**：一句话让面试官觉得有价值，五分钟让工程师理解架构，半小时让贡献者能上手开发。

## 参考回答

文档是智能体项目的"销售员"。面试官在看你的 GitHub 项目时，前 30 秒决定是否继续，前 5 分钟决定是否约面试。好的文档在 30 秒内传递核心价值，5 分钟内展示技术深度。

### 文档的四个层次和读者

```
Layer 1: 项目简介（READEM.md 顶部）
  读者：面试官、技术管理者
  内容：一句话定义 + 核心亮点 + 演示链接
  时间：30 秒看完

Layer 2: 快速入门（README.md 主体）
  读者：开发者、想尝试的用户
  内容：安装 + 配置 + 运行 + 基本使用
  时间：5 分钟看完

Layer 3: 技术文档（docs/ 目录）
  读者：贡献者、技术评估者
  内容：架构设计 + API 文档 + 设计决策
  时间：30 分钟看完

Layer 4: 深入文档（docs/deep-dive/ 目录）
  读者：维护者、研究者
  内容：实现细节 + 评测方法 + 扩展指南
  时间：按需阅读
```

### 架构文档的写法

```markdown
# 系统架构

## 架构概览
[用文字描述整体架构，附架构图（ASCII 或图片）]

## 核心模块

### 1. Agent Loop
**职责**：控制智能体的执行流程

**核心逻辑**：
1. 接收用户输入
2. 判断意图
3. 选择工具（多轮可能）
4. 执行工具
5. 处理结果
6. 判断完成或继续

**关键设计**：
- 停止条件：maxTurns(10), tokenBudget(8K), userInterrupt
- 状态持久化：支持中断恢复
- Circuit Breaker：检测循环调用

**接口**：
\`\`\`python
class AgentLoop:
    async def run(self, input: str, context: Context) -> Response: ...
    async def step(self, state: State) -> StepResult: ...
    def should_stop(self, state: State) -> bool: ...
\`\`\`

### 2. Tool Registry
[类似结构：职责 + 逻辑 + 设计 + 接口]

### 3. Context Manager
[类似结构]

### 4. Permission System
[类似结构]

## 数据流
[描述用户请求的完整数据流]

## 设计决策记录

### ADR-001: 为什么自研 Agent Loop 而不是用 LangChain？
- **上下文**：项目初期使用了 LangChain
- **决策**：自研 Agent Loop
- **理由**：
  - LangChain 回调机制在复杂工具链下调试困难
  - 需要支持 ReAct 和 Plan-and-Execute 混合模式
  - 自研后工具调用成功率从 82% 提升到 97%
- **后果**：维护成本增加，但可控性大幅提升

### ADR-002: 为什么选择三级权限体系？
- **上下文**：需要支持不同风险级别的工具调用
- **决策**：READ / WRITE / ADMIN 三级
- **理由**：覆盖了主要的使用场景，实现复杂度可控
- **后果**：需要用户配置角色，增加了部署复杂度

### ADR-003: 为什么评测使用 LLM-as-Judge？
- **上下文**：需要评估开放域回答的质量
- **决策**：使用 GPT-4 作为评测 Judge
- **理由**：精确匹配无法处理语义等价问题
- **后果**：评测成本增加，但评估准确性大幅提升
```

### 工具文档的写法

```markdown
# Tool: get_user_info

## 功能
根据用户 ID 或用户名查询用户基本信息

## 使用场景
- 身份验证：查询用户是否存在
- 权限判断：获取用户的角色和部门
- 信息展示：获取用户的联系方式

## 参数
| 参数名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| user_id | string | 否 | - | 用户 ID（优先） |
| username | string | 否 | - | 用户名 |
| include_dept | boolean | 否 | true | 是否包含部门信息 |

注意：user_id 和 username 至少提供一个

## 返回
\`\`\`json
{
  "user_id": "u001",
  "username": "张三",
  "department": "研发部",
  "role": "工程师",
  "email": "zhangsan@company.com"
}
\`\`\`

## 权限要求
- 最低权限：READ
- 数据范围：当前用户只能查询自己的信息，管理员可查询全部

## 错误场景
| 错误 | 原因 | 处理方式 |
|------|------|---------|
| USER_NOT_FOUND | 用户不存在 | 返回错误信息 |
| PERMISSION_DENIED | 权限不足 | 记录审计日志 |
| DB_TIMEOUT | 数据库超时 | 重试 2 次后返回错误 |

## 安全注意事项
- 返回结果中脱敏手机号和密码字段
- 日志中不记录完整的用户信息
- 查询频率限制：30 次/分钟/用户
```

### 评测文档的写法

```markdown
# 评测体系

## 评测维度
| 维度 | 说明 | 指标 | 当前分数 | 目标 |
|------|------|------|---------|------|
| 功能正确性 | 任务是否正确完成 | 完成率 | 93.7% | >95% |
| 工具选择 | 是否选择了正确的工具 | 准确率 | 96.2% | >98% |
| 延迟 | 端到端响应时间 | P95 | 3.2s | <3s |
| 成本 | 单次任务费用 | 平均 | ¥0.032 | <¥0.03 |
| 安全 | 安全事件率 | 事件数/月 | 2 | <1 |
| 一致性 | 同一输入是否一致输出 | 一致率 | 91.7% | >95% |

## 数据集
- 总用例数：200
- 来源：合成数据 + 人工构建
- 覆盖：20 种任务类型，10 种边缘情况
- 分布：简单 40%，中等 35%，困难 25%

## 评测方法
1. 精确匹配：结构化输出（JSON Schema 校验）
2. LLM-as-Judge：开放域语义等价判断
3. 轨迹评测：工具调用序列的合理性
4. 人工抽样：每月随机抽样 50 个评测，人工审核

## 质量门禁
- 通过率 > 90%
- 关键场景（安全类）100% 通过
- P95 延迟 < 5s
- 单次成本 < ¥0.1

## 运行方式
\`\`\`bash
# 运行完整评测
python eval/run.py --suite full --report eval-report.json

# 运行快速评测
python eval/run.py --suite quick

# 查看评测报告
python eval/view.py eval-report.json
\`\`\`

## CI/CD 集成
每次 PR 自动触发快速评测，合并到 main 触发完整评测。
评测失败自动在 PR 中评论报告。
```

### 文档的量化标准

好的项目文档可以通过以下标准来量化评估：

```typescript
interface DocumentationQuality {
  // 覆盖度
  coverage: {
    hasReadme: boolean;           // 有 README.md
    hasArchitecture: boolean;     // 有架构文档
    hasApiDoc: boolean;           // 有 API 文档
    hasEvalDoc: boolean;          // 有评测文档
    hasSecurityDoc: boolean;      // 有安全文档
    hasContributing: boolean;     // 有贡献指南
  };

  // 完整性
  completeness: {
    hasInstallGuide: boolean;     // 安装指南
    hasConfigGuide: boolean;      // 配置说明
    hasUsageExample: boolean;     // 使用示例
    hasDemoLink: boolean;         // 演示链接
    hasTestResult: boolean;       // 评测结果
    hasComparison: boolean;       // 对比测试
  };

  // 可读性
  readability: {
    oneLinerScore: number;        // 一句话简介清晰度 1-5
    structureScore: number;       // 结构清晰度 1-5
    codeExampleCount: number;     // 代码示例数量
    diagramCount: number;         // 图表数量
  };
}
```

## 深度展开

### 面试场景的文档策略

面试官看你的项目文档时，通常在验证以下问题：

```
1. 这个项目解决了什么问题？（30 秒找到答案）
2. 这个项目有多大的技术深度？（2 分钟找到证据）
3. 这个项目是真实的还是 Demo？（5 分钟找到工程痕迹）
```

所以文档设计要针对这三个问题做优化：

```markdown
# 在 README 顶部提供 "TL;DR" 区块

## TL;DR
本项目是一个面向 IT 工单处理的智能体系统，核心创新点：
1. 自研 Agent Loop 替代 LangChain，工具调用准确率从 82% 提升到 97%
2. 三级权限体系，月均拦截 200+ 异常工具调用
3. 四级上下文压缩策略，支撑 50+ 轮长对话
4. 自动化评测流水线，200+ 回归用例

[演示视频] | [在线 Demo] | [评测报告]
```

### 文档中的工程信号

面试官在文档中寻找以下"工程信号"：

| 信号 | 好的体现 | 差的体现 |
|------|---------|---------|
| 版本管理 | 有 CHANGELOG.md，版本语义化 | 没有版本号 |
| 引用规范 | 引用了相关论文和工具 | 闭门造车 |
| 一致性 | 代码风格统一，命名规范 | 风格混乱 |
| 测试痕迹 | 评测结果、测试覆盖率 | 没有测试相关内容 |
| CI/CD | README 中有 CI 状态 badge | 没有 CI 配置 |
| 容器化 | 有 Dockerfile 和 docker-compose | 手动部署 |
| 配置管理 | 有 .env.example 和配置文件 | 硬编码配置 |

### 文档模板生成器

```typescript
class DocGenerator {
  generateReadme(project: ProjectConfig): string {
    // 根据项目配置自动生成 README
    return `
# ${project.name}

${project.oneLiner}

## 核心亮点
${project.highlights.map(h => `- ${h}`).join("\n")}

## 快速开始
${this.generateQuickStart(project)}

## 核心概念
${project.concepts.map(c => `- **${c.name}**: ${c.description}`).join("\n")}

## 项目结构
${this.generateStructure(project.structure)}

## 技术栈
${project.techStack.map(t => `- ${t.name}: ${t.detail}`).join("\n")}

## 评测结果
${this.generateEvalTable(project.evalResults)}

## 对比测试
${this.generateComparisonTable(project.baselines)}
`;
  }

  private generateQuickStart(project: ProjectConfig): string {
    return `
\`\`\`bash
git clone ${project.repo}
cd ${project.repo}
${project.installCommand}
${project.runCommand}
\`\`\`
`;
  }

  private generateEvalTable(results: EvalResult[]): string {
    const header = "| 维度 | 指标 | 分数 |";
    const separator = "|------|------|------|";
    const rows = results.map(r => `| ${r.dimension} | ${r.metric} | ${r.score} |`);
    return [header, separator, ...rows].join("\n");
  }
}
```

## 常见误区

- **README 只有安装步骤**：面试官看完不知道项目做什么，也不知道技术深度在哪
- **文档和代码不一致**：配置说明和实际配置不符，使用示例无法运行
- **没有演示链接**：面试官无法直观感受项目效果
- **技术栈描述模糊**：只说"Python + 大模型"，没有具体框架和库
- **缺乏结构**：所有内容堆在一个 README 里，没有分层和导航
- **忽略英文受众**：虽然用中文写项目文档，但技术术语和代码注释应该用英文
- **没有设计决策的记录**：面试官想知道你为什么选 A 不选 B，但文档没有记录

## 追问方向

- **"你的 README 里写了工具调用成功率 97%，这个数是怎么测出来的？"**
  能引出评测文档的话，说明文档设计得好。

- **"如果要让一个完全没接触过 AI 的人看懂你的项目，你要怎么改文档？"**
  考察文档的面向对象意识。好的回答会针对不同读者调整文档内容和表述。

- **"你项目里的 ADR（架构决策记录）是怎么决定写什么不写什么的？"**
  考察对设计决策记录的取舍判断。只记录有价值、有替代方案的决策。

- **"你觉得文档写得好和写不好，对项目最大的影响是什么？"**
  考察对文档价值的理解。对求职项目来说，文档决定了面试官的第一印象。

- **"你的文档花多长时间维护？用什么工具？"**
  考察文档维护的实际投入。回答应该包括写作、review、更新的完整流程。

## 学习建议

写好项目文档最重要的方法是"把自己当成读者"。先试图从零开始跑一个不熟悉的开源项目，记录过程中遇到的困难，然后在自己的文档中避免这些问题。推荐参考一些高质量的 Agent 开源项目的文档结构，比如 LangChain、AutoGPT 的 README 和 docs 目录。写好初稿后，找一个不了解项目的人看，让他提问题——这些问题的答案就是文档需要补充的内容。文档不是一次性的，每次新增功能时同步更新文档，养成习惯。
