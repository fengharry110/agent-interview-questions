# 如何讲工具调用设计？

## 面试问题

你这个智能体项目的工具调用是怎么设计的？模型怎么知道用哪个工具？

## 考察重点

- 是否能说清楚工具调用在智能体系统中的完整流程
- 是否理解工具描述对模型选择的影响
- 是否有工具注册、发现、参数校验的实际设计经验
- 是否处理过工具选择错误、参数错误、执行超时等异常场景
- 是否了解 Function Calling 的底层原理和局限

## 核心结论

工具调用的设计核心不是"怎么调 API"，而是**如何让模型在正确的时机用正确的参数调用正确的工具**，这取决于工具注册机制、描述工程、参数校验、权限控制和错误处理的协同设计。

## 参考回答

工具调用是智能体区别于普通聊天的核心能力。面试官问这个问题时，真正想听的是你是否理解工具调用的完整链路，而不仅仅是调用了 OpenAI 的 `functions` 参数。

### 工具调用的完整链路

```
用户请求
   │
   ▼
1. 工具注册 ─── 工具描述生成（影响模型选择的输入）
   │
   ▼
2. 工具选择 ─── 模型根据上下文选择工具（概率推理）
   │
   ▼
3. 参数生成 ─── 模型根据 JSON Schema 生成参数
   │
   ▼
4. 参数校验 ─── 校验参数合法性（防止模型乱填）
   │
   ▼
5. 权限检查 ─── 检查该工具在当前上下文中是否允许执行
   │
   ▼
6. 工具执行 ─── 执行并获取结果
   │
   ▼
7. 结果处理 ─── 结果回填到上下文，或触发下一步
```

### 工具注册机制

```typescript
// 工具注册的核心数据结构
interface ToolDefinition {
  // 工具标识
  name: string;                    // 唯一名称，模型用来引用
  description: string;             // 描述文本，模型用来判断用途
  parameters: JSONSchema;          // 参数 Schema，模型用来生成参数

  // 执行逻辑
  handler: (args: any, context: ToolContext) => Promise<ToolResult>;

  // 运行时元信息
  category: string;                // 工具分类
  permissionLevel: "read" | "write" | "admin";  // 权限等级
  timeoutMs: number;               // 超时控制
  retryPolicy: RetryConfig;        // 重试策略
}

// 注册框架
class ToolRegistry {
  private tools: Map<string, ToolDefinition> = new Map();

  register(tool: ToolDefinition): void {
    this.validateToolDefinition(tool);
    this.tools.set(tool.name, tool);
  }

  registerFromModule(module: ToolModule): void {
    for (const tool of module.tools) {
      this.register(tool);
    }
  }

  // 根据权限过滤可用的工具列表
  getAvailableTools(permissionLevel: string): ToolDefinition[] {
    return Array.from(this.tools.values())
      .filter(t => this.hasPermission(permissionLevel, t.permissionLevel));
  }

  // 动态发现（MCP 协议）
  async discoverFromMCPServer(serverUrl: string): Promise<void> {
    const tools = await mcpClient.listTools(serverUrl);
    for (const tool of tools) {
      this.register({
        name: `mcp__${tool.serverName}__${tool.method}`,
        description: tool.description,
        parameters: tool.inputSchema,
        handler: (args, ctx) => mcpClient.callTool(serverUrl, tool.method, args),
        category: "mcp",
        permissionLevel: "admin",
        timeoutMs: 30000,
        retryPolicy: { maxRetries: 2, backoff: "exponential" },
      });
    }
  }

  private validateToolDefinition(tool: ToolDefinition): void {
    if (!tool.name || !tool.description || !tool.handler) {
      throw new Error("Tool definition missing required fields");
    }
    if (!/^[a-z][a-zA-Z0-9_]*$/.test(tool.name)) {
      throw new Error("Tool name must be snake_case");
    }
  }
}
```

### 工具描述工程

工具描述是影响模型选择的最关键因素。描述写不好，模型就选错工具。

| 问题 | 差的描述 | 好的描述 |
|------|---------|---------|
| 模糊 | 查用户信息 | 根据用户 ID 查询用户的基本信息，包括姓名、部门、邮箱、角色。适用于用户身份验证和权限判断场景 |
| 重复 | 获取天气、获取天气信息 | 调用第三方天气 API 获取指定城市的实时天气数据，返回温度、湿度、风速。注意：需要 latitude 和 longitude 参数 |
| 遗漏约束 | 重置密码 | 重置用户密码并发送临时密码到用户注册邮箱。操作不可逆，需要管理员权限。执行后用户需要修改密码 |
| 缺少边界 | 执行 SQL 查询 | 对只读副本执行 SQL SELECT 查询。不支持 INSERT/UPDATE/DELETE。返回最多 100 行结果。适用于数据分析和报表生成场景 |

```typescript
// 描述生成的工程化方法
function generateToolDescription(
  tool: ToolDefinition,
  options?: { includeExamples?: boolean; includeErrors?: boolean }
): string {
  const parts: string[] = [];
  
  // 功能描述（必填）：做什么
  parts.push(tool.description);

  // 使用场景（推荐）：什么时候用
  if (tool.useCases?.length) {
    parts.push(`适用场景：${tool.useCases.join("、")}`);
  }

  // 参数说明（推荐）：参数之间的关系
  if (tool.parameterHints?.length) {
    for (const hint of tool.parameterHints) {
      parts.push(`注意：${hint}`);
    }
  }

  // 约束条件（推荐）：不能做什么
  if (tool.constraints?.length) {
    parts.push(`约束：${tool.constraints.join("; ")}`);
  }

  return parts.join(". ");
}
```

### 参数校验

模型生成的参数不一定合法，必须在执行前校验。

```typescript
class ToolParameterValidator {
  validate(params: any, schema: JSONSchema): ValidationResult {
    const errors: ValidationError[] = [];

    // 1. 类型校验
    if (schema.type === "object") {
      if (typeof params !== "object" || params === null) {
        errors.push({ field: "", message: "必须是一个对象" });
      }
    }

    // 2. 必填字段校验
    if (schema.required) {
      for (const field of schema.required) {
        if (params[field] === undefined || params[field] === null) {
          errors.push({ field, message: `缺少必填字段` });
        }
      }
    }

    // 3. 枚举值校验
    if (schema.properties) {
      for (const [key, prop] of Object.entries(schema.properties)) {
        if (params[key] !== undefined) {
          if (prop.enum && !prop.enum.includes(params[key])) {
            errors.push({ field: key, message: `值必须在 [${prop.enum}] 范围内` });
          }
        }
      }
    }

    // 4. 边界校验（数字范围、字符串长度）
    for (const [key, prop] of Object.entries(schema.properties || {})) {
      if (params[key] !== undefined) {
        if (prop.minimum !== undefined && params[key] < prop.minimum) {
          errors.push({ field: key, message: `不能小于 ${prop.minimum}` });
        }
        if (prop.maximum !== undefined && params[key] > prop.maximum) {
          errors.push({ field: key, message: `不能大于 ${prop.maximum}` });
        }
      }
    }

    return {
      valid: errors.length === 0,
      errors,
      sanitized: errors.length === 0 ? params : this.sanitize(params, schema),
    };
  }

  private sanitize(params: any, schema: JSONSchema): any {
    // 移除多余字段，填充默认值
    const result: any = {};
    for (const [key, prop] of Object.entries(schema.properties || {})) {
      result[key] = params[key] ?? prop.default;
    }
    return result;
  }
}
```

### 工具执行与错误处理

```typescript
async function executeToolWithSafety(
  tool: ToolDefinition,
  args: any,
  context: ToolContext
): Promise<ToolResult> {
  const startTime = Date.now();
  
  try {
    // 1. 参数校验
    const validation = validator.validate(args, tool.parameters);
    if (!validation.valid) {
      return {
        success: false,
        error: `参数校验失败: ${validation.errors.map(e => e.message).join("; ")}`,
        duration: Date.now() - startTime,
      };
    }

    // 2. 权限检查
    if (!permissionChecker.check(context.userRole, tool.permissionLevel)) {
      return {
        success: false,
        error: `权限不足: 需要 ${tool.permissionLevel} 级别权限`,
        duration: Date.now() - startTime,
      };
    }

    // 3. 执行（带超时和重试）
    const result = await retryWithBackoff(
      () => Promise.race([
        tool.handler(validation.sanitized, context),
        timeout(tool.timeoutMs),
      ]),
      tool.retryPolicy
    );

    return {
      success: true,
      data: result,
      duration: Date.now() - startTime,
    };
  } catch (error) {
    return {
      success: false,
      error: error.message,
      duration: Date.now() - startTime,
    };
  }
}
```

## 深度展开

### Function Calling 的底层原理

面试官可能会追问 Function Calling 的底层机制。核心要点：

1. **不是"理解"，是"匹配"**：模型不是真的理解工具，而是基于描述文本做概率匹配
2. **描述是唯一的信号**：没有微调的情况下，工具名 + 描述 + 参数 Schema 是模型选择工具的全部依据
3. **参数是生成的，不是解析的**：模型生成 JSON 文本，系统解析 JSON，不是结构化的 API 调用
4. **上下文长度影响选择**：工具列表太长时，模型会忽略靠后的工具（注意力衰减）

```typescript
// Model 内部视角：工具选择的概率过程（示意）
interface ModelToolCalling {

    //  参数需要 city 和 date，用户说了北京和今天。
    //  所以我应该调用 get_weather(city='北京', date='2024-01-15')"

  }

  // 影响选择的因素（按重要性排序）：
  // 1. 工具描述与用户 query 的语义相似度
  // 2. 工具 name 的语义相关性
  // 3. 参数 Schema 与上下文中可用信息的匹配
  // 4. 工具在列表中的位置（靠前有利）
}
```

### 工具调用的六种失败模式

| 失败模式 | 现象 | 原因 | 解决方案 |
|---------|------|------|---------|
| 选择错误 | 用 A 工具做了 B 的事 | 描述不准确或多义 | 优化描述，加使用场景和反例 |
| 参数幻觉 | 传入不存在的参数或值 | 参数 Schema 不清晰 | 枚举值 + 边界校验 + 示例 |
| 漏调工具 | 应该用工具但直接硬答 | 工具列表太长被忽略 | 根据上下文动态注入工具 |
| 循环调用 | 反复调用同一个工具 | 停止条件不充分 | 加去重检测 + 循环检测 |
| 并行错误 | 工具之间有依赖但模型并行调用 | 缺少依赖声明 | 加工具依赖图/工作流 |
| 上下文污染 | 前一个工具的结果干扰后一个调用 | 结果回填策略不当 | 结构化结果存储，隔离上下文 |

### 工具调用的可观测性

```typescript
interface ToolCallTrace {
  id: string;
  toolName: string;
  args: any;
  result: any;
  success: boolean;
  duration: number;
  timestamp: number;
  user: string;
  sessionId: string;
  error?: string;
  retryCount: number;
  permissionCheck: PermissionCheckResult;
}

class ToolCallLogger {
  private traces: ToolCallTrace[] = [];

  log(call: ToolCallTrace): void {
    this.traces.push(call);
    // 写入持久化存储
    this.writeToStorage(call);
    
    // 失败告警
    if (!call.success) {
      this.reportError(call);
    }
    
    // 异常模式检测
    if (this.detectAnomaly(call)) {
      this.alert(call);
    }
  }

  // 工具选择分布统计
  getToolCallDistribution(): Map<string, number> {
    const distribution = new Map();
    for (const trace of this.traces) {
      distribution.set(trace.toolName, (distribution.get(trace.toolName) || 0) + 1);
    }
    return distribution;
  }

  // 工具成功率统计
  getToolSuccessRate(toolName: string): number {
    const calls = this.traces.filter(t => t.toolName === toolName);
    if (calls.length === 0) return 1;
    return calls.filter(t => t.success).length / calls.length;
  }
}
```

## 常见误区

- **工具描述写得像函数注释**：`get_user(id: int) -> User`，模型无法理解使用场景
- **工具数量过多不加筛选**：把 50 个工具全部注入上下文，模型决策质量严重下降
- **不校验模型生成的参数**：直接传给后端，导致 SQL 注入等安全问题
- **忽略工具之间的依赖关系**：模型串行调用两个有依赖的工具，但顺序反了
- **结果回填不做切割**：工具返回大量数据直接塞回上下文，很快就满
- **不对工具调用做限流**：模型循环调用同一个工具，快速消耗 token

## 追问方向

- **"如果两个工具的描述很相似，模型经常选错，你怎么解决？"**
  好答案会说：分析混淆矩阵，修改描述增加区分度，或合并为一个工具加 discriminator 参数。

- **"工具执行超时了怎么处理？重试策略怎么设计？"**
  考察工程容错能力。要区分幂等和非幂等操作，给出不同的重试策略。

- **"MCP 协议的动态工具发现怎么实现的？安全风险是什么？"**
  考察对 MCP 协议的理解。安全风险包括恶意工具注册、工具名冲突、权限逃逸。

- **"你的系统中怎么防止模型循环调用同一个工具？"**
  考察循环检测机制。方案包括：去重（相同的输入只执行一次）、次数限制、人工打断。

- **"工具描述的长度有限制吗？太长会有什么问题？"**
  考察对模型上下文窗口和注意力机制的理解。太长会占用 token，且靠后的描述容易被忽略。

## 学习建议

要深入理解工具调用，建议做三件事：第一，阅读 OpenAI Function Calling 文档和源码实现，理解底层原理；第二，在自己的项目中实现一个 Tool Registry，包含描述生成、参数校验、错误处理、调用追踪；第三，收集工具调用失败的 Case，分析失败原因，优化描述和流程。实践中你会发现，工具调用的 80% 问题都出在描述工程上。
