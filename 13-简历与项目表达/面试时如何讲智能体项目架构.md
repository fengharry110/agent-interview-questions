# 面试时如何讲智能体项目架构？

## 面试问题

请介绍一下你这个智能体项目的整体架构设计。

## 考察重点

- 是否能从功能架构和技术架构两个维度进行介绍
- 是否能把复杂系统拆解成清晰的模块和层次
- 是否能说出每个模块的设计原因和替代方案
- 是否理解关键设计决策的工程权衡
- 是否能在有限时间内传递最核心的信息

## 核心结论

讲智能体架构的核心不是画一张大而全的架构图，而是**用一条用户请求的完整链路，串联起所有核心模块**，让面试官在 3-5 分钟内理解系统全貌和你的设计思路。

## 参考回答

面试中介绍架构和写文档不同。面试官只有几分钟注意力，需要用"用户请求视角"来组织讲述逻辑。

### 三步讲架构法

```
Step 1: 一句话定义系统（30秒）
  "这是一个面向 XX 场景的智能体系统，核心能力是 XX，主要解决 XX 问题。"

Step 2: 用户请求链路（3分钟）
  从用户输入到最终输出的完整流程，逐个模块展开。

Step 3: 关键设计决策（2分钟）
  选型理由、替代方案、工程权衡、踩过的坑。
```

### 参考回答框架

以"智能客服 Agent"为例：

**Step 1 - 一句话定义**

"这是一个面向企业 IT 工单处理的智能客服 Agent 系统。用户在 IM 中提出 IT 问题，系统自动理解意图、调用相应工具完成操作，复杂场景下转人工。核心要解决的是工单自动处理率低的问题。"

**Step 2 - 请求链路**

```
用户提问："重置我的 VPN 密码"
   │
   ▼
┌─ 1. 意图识别 ────────────────────────┐
│  准确分类：VPN密码重置（置信度 98%）   │
│  触发安全校验：需二次确认             │
└───────────────┬───────────────────────┘
                ▼
┌─ 2. 上下文组装 ───────────────────────┐
│  系统消息：角色定义 + 权限说明         │
│  用户消息：当前问题                    │
│  工具描述：动态注入可用工具            │
└───────────────┬───────────────────────┘
                ▼
┌─ 3. Agent Loop ───────────────────────┐
│  Round 1: 模型选择 tool_get_user_info  │
│   → 执行结果：用户张三，部门研发       │
│  Round 2: 模型选择 tool_reset_vpn_pwd │
│   → 权限检查通过 → 执行 → 成功        │
│  Round 3: 无更多工具调用 → 输出结果    │
└───────────────┬───────────────────────┘
                ▼
┌─ 4. 结果处理 ────────────────────────┐
│  格式化回复、记录日志、更新工单状态    │
└───────────────────────────────────────┘
```

**Step 3 - 关键决策**

"架构中最关键的一个决策是为什么自研 Agent Loop 而不是用 LangChain。两个原因：一是 LangChain 的回调机制在复杂工具链下调试困难，我们遇到的错误有 30% 来自框架本身；二是我们需要支持 Plan-and-Execute 和 ReAct 混合模式，LangChain 的默认实现在模式切换时上下文管理不够灵活。自研后，工具调用成功率从 82% 提升到 97%，调试效率提升也很大。"

### 架构描述的层次控制

| 面试阶段 | 时间 | 详细程度 | 重点 |
|---------|------|---------|------|
| 自我介绍 | 1-2 分钟 | 全局概览 | 一句话定义 + 核心指标 |
| 项目深挖 | 5-10 分钟 | 模块级 | 请求链路 + 关键决策 |
| 系统设计 | 15-30 分钟 | 代码级 | 核心接口 + 设计模式 |
| 追问环节 | 不限 | 问题级 | 针对面试官关注的方面 |

## 深度展开

### 架构讲述的三个常见失败模式

**失败一：由顶向下抽象轰炸**
"系统分为数据层、服务层、应用层、展示层……"面试官听完还是不知道你的系统到底做了什么。

**失败二：陷入过多实现细节**
"我用 FastAPI 写了路由，用 Redis 做缓存，用 PostgreSQL 存储……"面试官只听到一堆技术名词，没有架构思维。

**失败三：只讲好的不讲坏的**
"我们就是这么设计的，所有决策都是最优的。"面试官会觉得你缺乏工程判断力。

### 好的架构讲述应该包含的要素

```
系统边界图（Context Diagram）
  - 系统与外部系统的交互
  - 用户、管理员、外部 API

模块分解图（Container Diagram）
  - Agent Loop、Tool Registry、Context Manager
  - 模块间的接口和数据流

关键序列图（Sequence Diagram）
  - 一个完整请求的模块交互时序
  - 正常路径 + 异常路径

设计决策表（Decision Log）
  - 自研 vs 框架
  - 同步 vs 异步
  - 有状态 vs 无状态
```

### 架构决策的工程权衡表

| 决策点 | 选项 A | 选项 B | 选择依据 |
|--------|--------|--------|---------|
| Agent Loop | 同步循环 | 事件驱动 | 同步调试简单，事件驱动适合高并发 |
| 工具注册 | 静态配置 | 动态发现 | 动态发现灵活，静态配置可控 |
| 上下文管理 | 全量保留 | 智能压缩 | 全量质量高但成本高，压缩省 token |
| 权限模型 | ACL | RBAC | ACL 粒度细，RBAC 管理方便 |
| 执行模式 | ReAct | Plan-and-Execute | ReAct 交互性好，Plan 适合复杂任务 |

## 工程实践

### 架构描述代码模板

```typescript

interface AgentArchitecture {
  // 系统边界
  system: {
    name: string;
    description: string;
    externalDependencies: string[];
  };

  // 核心模块
  modules: {
    agentLoop: AgentLoopSpec;
    toolSystem: ToolSystemSpec;

  dataFlow: {
    input: DataSpec;
    output: DataSpec;
    storage: StorageSpec;
  };

  // 部署拓扑
  deployment: DeploymentSpec;
}

// 用 Request-Response 链路来组织讲述
const requestFlow = [
  {
    step: "request_parse",

  },

  },
```

### 面试中画架构图的技巧

在无法使用白板的远程面试中，可以用文本描述引导面试官的想象：

```
不好：我们的系统有用户层、服务层、数据层。
好：用户通过 IM 发送消息，消息先到达 Gateway 做认证，然后进入 Agent Loop。
    Agent Loop 里依次经过意图路由、上下文拼装、工具选择三个子模块。
    工具选择找到匹配的工具后，调用 Permission Check 确认权限，
    然后执行工具，把结果返回给 Agent Loop，最后生成回复。
```

### 复杂架构的分层讲述法

当系统非常大时，不要试图一次讲完。使用"分层剥洋葱"策略：

```
第一层（30秒）：系统对外提供什么能力
第二层（2分钟）：核心架构 = Agent Loop + Tool System + Context Manager
第三层（5分钟）：每个核心模块的内部设计
第四层（按需）：特定模块的实现细节和代码

关键原则：面试官不问，就不讲第四层。
```

## 常见误区

- **一上来就画大架构图**：面试官还没建立上下文，看了也白看
- **用过多技术术语**："我们用了 RAG + Agent + MCP + SSE + WebSocket"，显得杂乱
- **只讲模块不讲交互**：每个模块都说了一遍，但面试官不知道它们怎么配合
- **低估"为什么这样设计"的重要性**：面试官更关心设计理由，而不是设计本身
- **不说失败经验**：只讲成功路径，面试官会觉得你缺乏工程体感
- **架构图太完美不真实**：真实系统一定有妥协和权衡，不讲这些反而显得假
- **时间分配失控**：在某个细节上讲 10 分钟，导致整体架构没讲完

## 追问方向

- **"你这个架构中最大的单点风险是什么？"**
  考察架构的容错意识。好的回答会说"我们的 Agent Loop 是单点"或者"依赖的某个外部 API"。

- **"如果用户量增长 10 倍，这个架构的瓶颈在哪里？"**
  考察架构的可扩展性设计。要能指出哪个模块最先扛不住。

- **"为什么这几个模块要分开而不是合并成一个？"**
  考察模块划分的合理性。好回答会给出具体的耦合度和复用性分析。

- **"你觉得这个架构中最让你不满意的地方是什么？"**
  考察自我批判和迭代意识。

- **"如果让你重新设计这个架构，你会改变什么？"**
  考察复盘能力和技术成长。

## 学习建议

准备这个问题时，建议用"一分钟版本"和"五分钟版本"两种粒度来练习。一分钟版本适用于自我介绍环节，五分钟版本适用于项目深挖环节。把你的系统画成 C4 模型（Context / Container / Component / Code）的四层架构图，确保每一层你都可以自然地展开。最重要的是，找一个朋友当面试官，让他听完后复述你的架构——如果他复述错了，说明你的讲述逻辑有问题。
