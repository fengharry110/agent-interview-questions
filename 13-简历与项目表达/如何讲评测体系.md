# 如何讲评测体系？

## 面试问题

你的智能体项目如何评测？怎么知道改了一个 Prompt 是变好还是变坏？

## 考察重点

- 是否建立了结构化的评测体系，而不是凭感觉判断
- 是否能说清楚评测维度和评测方法
- 是否有自动化评测流水线的实践经验
- 是否处理过评测指标和用户体验不一致的问题
- 是否了解智能体评测与传统 NLP 评测的差异

## 核心结论

智能体评测的核心不是单一的准确率指标，而是构建**多维度评测矩阵**，覆盖功能正确性、工具调用质量、安全合规、成本效率和用户体验，并通过自动化流水线实现可复现评测。

## 参考回答

评测是智能体项目从 Demo 到生产的关键一步。没有评测，你不知道改了一个 Prompt 是变好还是变坏，也不知道系统的真实水平。面试官问这个问题，是想确认你有没有做过系统性评测。

### 评测体系的核心框架

一个完整的评测体系分为四个层面：

```
Layer 1: 单元评测（Unit Eval）
  测试单个组件：意图识别、工具选择、参数生成

Layer 2: 端到端评测（End-to-End Eval）
  测试完整任务：输入 → 输出，覆盖典型场景

Layer 3: 轨迹评测（Trajectory Eval）
  测试中间过程：工具选择正确性、执行顺序、错误恢复

Layer 4: 在线评测（Online Eval）
  生产环境评测：A/B 测试、用户满意度、成本监控
```

### 评测跑分实现

```typescript
class EvalRunner {
  private testCases: TestCase[];
  private agent: Agent;

  constructor(testCases: TestCase[], agent: Agent) {
    this.testCases = testCases;
    this.agent = agent;
  }

  async run(): Promise<EvalReport> {
    const results: TestResult[] = [];

    for (const testCase of this.testCases) {
      const result = await this.runSingleTestCase(testCase);
      results.push(result);
    }

    return this.generateReport(results);
  }

  private async runSingleTestCase(testCase: TestCase): Promise<TestResult> {
    const startTime = Date.now();

    try {
      const response = await this.agent.handle(testCase.input);
      const toolCalls = this.agent.getLastTrajectory();

      return {
        testCaseId: testCase.id,
        passed: this.evaluate(testCase, response, toolCalls),
        response,
        toolCalls,
        latency: Date.now() - startTime,
        error: null,
      };
    } catch (error) {
      return {
        testCaseId: testCase.id,
        passed: false,
        response: null,
        toolCalls: [],
        latency: Date.now() - startTime,
        error: error.message,
      };
    }
  }

  private evaluate(
    testCase: TestCase,
    response: string | null,
    toolCalls: ToolCall[]
  ): boolean {
    // 1. 检查输出内容（使用 LLM 判断语义等价性）
    const contentMatch = testCase.expectedOutput
      ? this.llmJudge.semanticMatch(response, testCase.expectedOutput)
      : true;

    // 2. 检查工具调用（类型和数量）
    const toolMatch = testCase.expectedToolCalls
      ? this.checkToolCalls(toolCalls, testCase.expectedToolCalls)
      : true;

    return contentMatch && toolMatch;
  }

  private checkToolCalls(
    actual: ToolCall[],
    expected: ExpectedToolCall[]
  ): boolean {
    for (const exp of expected) {
      const actualCount = actual.filter(c => c.toolName === exp.toolName).length;
      if (exp.minCount !== undefined && actualCount < exp.minCount) return false;
      if (exp.maxCount !== undefined && actualCount > exp.maxCount) return false;
    }
    return true;
  }

  private generateReport(results: TestResult[]): EvalReport {
    const total = results.length;
    const passed = results.filter(r => r.passed).length;
    const byCategory = this.groupByCategory(results);
    const byDifficulty = this.groupByDifficulty(results);

    return {
      total,
      passed,
      failed: total - passed,
      passRate: (passed / total) * 100,
      byCategory,
      byDifficulty,
      avgLatency: results.reduce((s, r) => s + r.latency, 0) / total,
      details: results,
      timestamp: Date.now(),
    };
  }
}
```

### 用 LLM 做评测（LLM-as-Judge）

```typescript
class LLMJudge {
  constructor(private model: LLM) {}

  async semanticMatch(
    actual: string,
    expected: string
  ): Promise<boolean> {
    const prompt = `
比较以下两个回答是否语义等价（忽略表述差异，关注核心信息是否一致）：

期望回答：${expected}
实际回答：${actual}

请仅输出 YES 或 NO。`;

    const result = await this.model.generate(prompt);
    return result.trim().toUpperCase() === "YES";
  }

  async evaluateTrajectory(
    trajectory: ToolCall[],
    expectedSteps: ExpectedStep[]
  ): Promise<TrajectoryScore> {
    const evalPrompt = `
分析以下智能体的执行轨迹是否合理：

任务：${expectedSteps.map(s => s.description).join(" → ")}
执行轨迹：
${trajectory.map((t, i) => `Step ${i + 1}: ${t.toolName}(${JSON.stringify(t.args)}) → ${t.result}`).join("\n")}

请从以下维度评分（1-5）：
1. 工具选择正确性：每一步是否选择了合适的工具
2. 执行顺序合理性：工具调用的先后顺序是否正确
3. 参数正确性：每个工具的参数是否合理
4. 整体效率：是否有不必要的工具调用

请输出 JSON 格式评分。`;

    const result = await this.model.generate(evalPrompt);
    return JSON.parse(result);
  }
}
```

## 深度展开

### 评测中的关键挑战

**挑战一：答案不唯一**
同一任务可以有多种正确回答。准确率等精确匹配指标不适用。

解决方案：用 LLM Judge 做语义等价判断，或使用人工评测采样。

**挑战二：轨迹多样性**
同一任务可以有多种执行路径（不同工具组合）。

解决方案：轨迹评测关注"合理性"而非"精确性"，用 LLM 判断轨迹是否合理。

**挑战三：评测成本和速度**
用模型评测模型，成本和速度都是问题。

解决方案：分层评测策略——快速检查（规则匹配）+ 深度检查（LLM Judge）。

**挑战四：评测覆盖不全**
你的测试用例只能覆盖已知场景，但线上会遇到各种未知情况。

解决方案：线上失败回流——线上失败的 Case 自动加入测试集。

### 评测指标的对比

| 评测方式 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| 精确匹配 | 快、确定性强 | 忽略语义等价 | 结构化输出（JSON） |
| LLM Judge | 灵活、语义理解 | 成本高、延迟大、有偏见 | 开放域回答评测 |
| 人工评测 | 准确 | 慢、成本极高 | 最终验收、采样评测 |
| 用户反馈 | 真实反映体验 | 噪声大、反馈率低 | 线上持续监控 |
| A/B 测试 | 因果推断 | 实现复杂、周期长 | 重大版本变更 |
| 回归测试 | 防止退化 | 维护成本随用例增加 | 持续集成/部署 |

### 评测流水线的 CI/CD 集成

```yaml
# .github/workflows/eval.yml
name: Agent Evaluation

on:
  pull_request:
    paths:
      - 'prompts/**'
      - 'tools/**'
      - 'agent/**'

jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Run quick eval suite
        run: python eval/run.py --suite quick
        # 快速套件：20 个关键用例，< 2 分钟
      
      - name: Check quality gate
        run: |
          python eval/check_gate.py \
            --min-pass-rate 90 \
            --max-latency-ms 5000 \
            --report eval-report.json
      
      - name: Run full eval suite (if gate passes)
        if: success()
        run: python eval/run.py --suite full
        # 完整套件：200+ 用例，~15 分钟
      
      - name: Post evaluation report
        uses: actions/github-script@v6
        with:
          script: |
            const report = require('./eval-report.json');
            const summary = `评测结果: ${report.passRate}% 通过
            | 类别 | 通过率 |
            |------|--------|
            ${report.byCategory.map(c => `| ${c.name} | ${c.passRate}% |`).join('\n')}
            
            质量门禁: ${report.gatePassed ? '通过' : '未通过'}`;
            
            github.rest.issues.createComment({
              ...context.repo,
              issue_number: context.issue.number,
              body: summary
            });
```

## 常见误区

- **只评测成功路径**：忽略异常输入、边界条件、安全攻击，系统在线上很容易出问题
- **评测用例和训练数据重叠**：用例和 Prompt 优化用的数据重叠，过拟合到测试集
- **忽视成本评测**：只关心效果不看成本，优化到 99% 准确率但成本翻了 10 倍
- **LLM Judge 自评自**：用 GPT-4 评测 GPT-4 的输出，会放大模型的偏好偏差
- **评测不回归**：改了一个新功能，旧的功能退化了也没有发现
- **一次性评测**：只在项目开始时做了评测，后续迭代不再跑

## 追问方向

- **"你的评测用例是人工写的还是自动生成的？覆盖率如何保证？"**
  考察评测数据构建方法。好的回答会说先用人工写核心用例，再用模型辅助生成变体。

- **"如果评测通过了但线上效果不好，可能是什么原因？"**
  考察评测和线上效果的 GAP 意识。原因包括测试分布和线上分布不一致、用户行为变化等。

- **"你的质量门禁（Quality Gate）怎么设计？"**
  考察版本发布的自动化控制。要包括通过率阈值、关键场景零容忍、成本上限。

- **"LLM Judge 有偏见怎么办？怎么校准？"**
  考察对评测方法局限性的理解。方案包括：多模型投票、人工采样校准、与规则混合使用。

- **"1000 个评测用例，跑一次要 1 小时，你怎么优化？"**
  考察工程效率。方案包括：分层评测、并行执行、增量评测（只跑相关用例）。

## 学习建议

搭建评测体系最好的方式是从小开始。先找 20 个核心用例，用最简单的精确匹配做评测，跑一个 Baseline。然后逐步增加用例数量，引入 LLM Judge，加入安全测试和性能测试。关键是要让评测成为 CI/CD 流程的一部分。在面试中，如果能展示你搭建的评测流水线和评测报告，比口述概念更有说服力。推荐阅读 Anthropic 的 Eval 设计文档和 OpenAI 的 Eval 开源项目。
