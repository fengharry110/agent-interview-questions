# 单 Agent 和多 Agent 如何选择？

## 面试问题

什么时候使用单 Agent，什么时候使用多 Agent？

## 考察重点

- 是否具备克制使用多 Agent 的判断力
- 是否理解多 Agent 会增加协调和评测复杂度
- 是否能说出适合拆分的条件

## 核心结论

大多数场景应先从单 Agent 开始。只有当任务天然存在清晰角色分工、工具边界和独立上下文时，才考虑多 Agent。

## 参考回答

单 Agent 更简单，容易调试、评测和控制成本。很多任务并不需要多个 Agent，只需要一个 Agent 加清晰工具和状态管理。

多 Agent 适合角色差异明显的场景，比如软件开发中 Planner 负责拆任务，Coder 负责实现，Reviewer 负责检查；或者研究任务中 Researcher 收集资料，Writer 组织报告，Critic 做事实核查。

但多 Agent 会带来通信成本、上下文同步、责任边界、冲突解决和循环讨论问题。所以不应该为了显得高级而使用多 Agent。

## 深度展开

多 Agent 的核心问题是协调。每个 Agent 的目标、工具、输入输出和停止条件都要清楚。如果只是把同一个模型复制几份互相聊天，很容易变成低效讨论。真正有价值的多 Agent 通常需要强约束工作流。

## 工程实践

- 先实现单 Agent baseline
- 只有瓶颈明确时再拆角色
- 每个 Agent 输出结构化结果
- 设置总控器负责冲突解决和停止条件

## 常见误区

- 认为多 Agent 一定比单 Agent 智能
- 多个 Agent 共享混乱上下文
- 没有评测就判断多 Agent 更好

## 追问方向

- 多 Agent 如何共享状态？
- Reviewer Agent 可靠吗？
- 多 Agent 的成本如何控制？

## 学习建议

面试时先说“默认单 Agent”，再讲清楚拆分条件，会显得更成熟。
