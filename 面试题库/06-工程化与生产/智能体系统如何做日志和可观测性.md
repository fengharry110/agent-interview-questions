# Agent 系统如何做日志和可观测性？

## 面试问题

Agent 系统如何设计日志和可观测性？

## 考察重点

- 是否理解 Agent 调试依赖完整执行轨迹
- 是否知道 trace 应记录模型、工具、状态和成本
- 是否关注隐私脱敏和线上监控

## 核心结论

Agent 可观测性的核心是记录完整任务轨迹，让工程师知道 Agent 每一步为什么这样做、调用了什么、结果如何。

## 参考回答

Agent 日志不能只记录最终回答。应该记录用户任务、系统提示版本、模型输入输出、计划、工具调用、工具参数、工具结果、错误、重试、耗时、token 使用量和成本。

这些信息应该串成一次 trace，可以按任务 ID 查询。这样当 Agent 失败时，工程师能还原完整过程，判断问题来自目标理解、检索、工具、权限还是生成。

线上还需要指标监控，比如任务成功率、工具失败率、平均步骤数、平均延迟、成本趋势、安全拦截次数和人工确认率。

## 深度展开

Agent 是多组件系统：模型、工具、检索、权限、业务 API 都可能失败。如果没有可观测性，失败只能靠猜。好的可观测性不仅用于排错，也用于评测、成本优化和安全审计。

同时，trace 中可能包含用户隐私、密钥片段、业务数据，因此必须做脱敏、访问控制和保留周期管理。

## 工程实践

- 每次任务生成唯一 trace_id
- 每个工具调用记录输入、输出摘要、耗时和错误类型
- prompt 版本和模型版本写入日志
- 敏感字段脱敏后再存储

## 常见误区

- 只保存最终答案
- 日志没有关联 ID，无法串起完整任务
- 为了调试保存过多敏感信息

## 追问方向

- trace 和普通日志有什么区别？
- 如何从 trace 中发现高频失败类型？
- 如何监控 Agent 成本异常？

## 学习建议

做 Agent 项目时，把执行轨迹展示出来，会显得比普通聊天 demo 专业很多。
