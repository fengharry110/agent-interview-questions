# 如何设计一个好的工具 Schema？

## 面试问题

你会如何设计一个适合 Agent 使用的工具 Schema？

## 考察重点

- 是否理解工具 Schema 会影响模型选择和参数生成
- 是否知道输入输出都要结构化
- 是否具备安全边界意识

## 核心结论

好的工具 Schema 应该让模型清楚知道工具何时使用、需要什么参数、不能做什么、会返回什么结果。

## 参考回答

工具 Schema 通常包含工具名、描述、输入参数、输出结构和约束。工具名要表达动作，例如“读取工作区文件”“搜索代码文本”。描述要说明适用场景，不要只写“用于处理文件”这种模糊表达。

参数设计要少而清晰。每个参数都应该有类型、含义、是否必填、取值范围。比如文件读取工具可以接受相对路径，但不应该接受任意系统路径。命令执行工具更应该限制命令范围，而不是把 shell 字符串完全交给模型。

输出也要结构化。成功时返回数据，失败时返回错误类型、是否可重试、建议下一步。这样 Agent 才能根据工具结果继续决策。

## 深度展开

工具 Schema 本质上是模型和真实系统之间的契约。Schema 太弱，模型输出自由度过高，系统难以校验；Schema 太复杂，模型又容易填错参数。一个好的 Schema 通常是“最小但完整”的：只暴露完成任务所需的参数，不暴露内部实现细节。

## 工程实践

- 使用枚举限制模式参数，例如 `read`、`write`、`append`
- 使用路径白名单限制文件范围
- 对危险参数设置二次确认
- 工具返回统一格式，便于日志和评测

## 常见误区

- 参数设计过宽，比如直接给 `command: string`
- 工具描述只写实现，不写使用时机
- 忽略输出 Schema，导致 Agent 无法稳定理解结果

## 追问方向

- 工具参数越详细越好吗？
- 如何减少模型幻造参数？
- 什么时候应该拆成多个工具？

## 学习建议

用 Pydantic 或 JSON Schema 给工具建模。你能把工具输入输出讲清楚，面试官会更相信你真的做过 Agent。
