# 什么是 Tool Calling？

## 面试问题

什么是 Tool Calling？为什么它是 Agent 系统里的关键能力？

## 考察重点

- 是否理解模型本身不会真正执行外部动作
- 是否知道工具调用需要结构化参数和程序执行
- 是否能说明工具结果如何回到 Agent 循环

## 核心结论

Tool Calling 是让模型选择外部工具并生成结构化参数，由程序执行真实动作，再把结果返回给模型继续推理的机制。

## 参考回答

大语言模型本身只能生成文本，不能真正访问数据库、读取文件、发送邮件或运行命令。Tool Calling 的作用是把模型的意图转成受控的函数调用。模型决定“应该调用哪个工具、传入什么参数”，但真正执行的是应用程序。

在 Agent 中，Tool Calling 是模型连接外部世界的接口。比如 Agent 想知道某个文件内容，就调用 read_file 工具；想验证代码，就调用 run_tests 工具；想查知识库，就调用 search_docs 工具。工具执行后返回 observation，模型再决定下一步。

## 深度展开

Tool Calling 的难点不在“能不能调用函数”，而在稳定性和边界。工具描述写得不清楚，模型会误选工具；参数 schema 太宽，模型会传入危险或无效参数；工具错误返回不清楚，模型就无法恢复。

生产系统通常会把工具调用看成一个受控协议：模型只能从已注册工具中选择，参数必须经过校验，执行必须经过权限检查，结果必须结构化返回。

## 工程实践

- 工具名要具体，例如 `读取文件` 比 `处理文件` 更清楚
- 参数使用严格 schema，不要传一大段自由文本
- 工具返回要区分 success、error_type、message、data
- 对高风险工具加入审批、沙箱和审计日志

## 常见误区

- 把工具调用理解成模型真的执行代码
- 只写工具函数，不设计权限和错误处理
- 工具描述含糊，导致模型乱选工具

## 追问方向

- Tool Calling 和插件有什么区别？
- 模型选错工具怎么办？
- 工具返回自然语言还是 JSON 更好？

## 学习建议

可以先写 3 个工具：读取文件、搜索文本、运行测试。用这 3 个工具就能理解大部分编码 Agent 的执行模式。
