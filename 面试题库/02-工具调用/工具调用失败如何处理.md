# Agent 调用工具失败时应该怎么办？

## 面试问题

Agent 调用工具失败时，你会如何处理？

## 考察重点

- 是否能区分可重试和不可重试错误
- 是否知道错误信息也要结构化返回给 Agent
- 是否能避免失败循环

## 核心结论

工具失败不是简单抛异常，而要分类、结构化返回、限制重试，并让 Agent 根据错误类型选择修正、重试、换路径或停止。

## 参考回答

我会先把工具错误分类。参数校验错误通常说明模型传参不对，可以把具体字段错误返回，让 Agent 修正后重试。网络超时可以有限重试。权限错误不能反复重试，应该请求授权或停止。业务规则错误则需要告诉 Agent 为什么失败，让它选择替代方案。

错误返回最好包含 `error_type`、`message`、`retryable`、`suggestion` 等字段。不要只把 Python traceback 或 HTTP 错误原样丢给模型，因为模型很难稳定判断下一步。

同时系统要设置最大重试次数和最大步骤数。如果 Agent 连续以相同方式失败，应该触发停止或人工介入。

## 深度展开

工具失败处理反映 Agent 系统是否可生产化。Demo 往往只处理成功路径，但真实业务里失败是常态：文件不存在、接口限流、权限不足、网络超时、数据格式变化、工具返回为空。Agent 必须能从失败中恢复，而不是把失败包装成看似合理的回答。

## 工程实践

- 统一错误格式，避免每个工具随意返回
- 对相同错误参数做去重，避免重复尝试
- 把失败 trace 写入日志，用于后续评测
- 对不可恢复错误给出清楚的用户反馈

## 常见误区

- 让模型自己阅读原始异常堆栈
- 所有错误都盲目重试
- 工具失败后仍然生成肯定答案

## 追问方向

- 如何判断错误是否可重试？
- 如何避免 Agent 一直修同一个参数？
- 工具失败是否应该暴露给用户？

## 学习建议

把错误至少分成 validation、permission、timeout、not_found、business_rule、unknown 六类，会让回答更有工程感。
