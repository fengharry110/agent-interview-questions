# 如何设计安全的文件和命令执行工具？

## 面试问题

如果 Agent 需要读写文件或执行命令，你会如何保证安全？

## 考察重点

- 是否理解文件和命令工具具有真实破坏性
- 是否知道沙箱、白名单、路径限制和人工确认
- 是否能区分读、写、删、执行的不同风险

## 核心结论

文件和命令工具必须遵循最小权限原则：限制范围、校验参数、记录日志，并对危险操作进行人工确认。

## 参考回答

文件工具应该首先限制工作区。Agent 可以读取项目内文件，但不能随意读取系统目录、用户密钥或其他项目文件。读文件、写文件、删除文件最好拆成不同工具，因为风险级别不同。写文件前可以生成 diff，让用户或系统校验后再应用。

命令执行工具风险更高，不应该直接暴露无限制 shell。更安全的方式是使用命令白名单，比如只允许 `pytest`、`npm test`、`rg` 等明确命令。涉及安装依赖、删除文件、网络访问、修改系统配置的命令，需要显式授权。

工具执行后要记录命令、参数、退出码、输出摘要和操作者，便于审计。

## 深度展开

真正危险的不是模型“想坏事”，而是模型可能误解上下文、被注入攻击诱导，或者把用户的自然语言误转成破坏性操作。因此安全不能依赖模型自觉，而要依赖系统边界。

## 工程实践

- 所有路径解析后必须确认仍在工作区内
- 删除和覆盖操作默认需要确认
- 命令执行设置超时和输出截断
- 日志中隐藏密钥、token、cookie 等敏感信息

## 常见误区

- 给 Agent 一个无限制终端
- 只在 prompt 里告诉模型“不要做危险操作”
- 写工具时不区分读取和修改权限

## 追问方向

- 如何防止路径穿越？
- 命令 allowlist 如何维护？
- 如果工具输出包含密钥怎么办？

## 学习建议

可以参考 Codex 类工具的权限思路：默认允许低风险读操作，高风险写操作和外部影响操作必须受控。
