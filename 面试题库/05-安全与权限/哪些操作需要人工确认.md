# 哪些操作需要 Human-in-the-loop？

## 面试问题

Agent 系统中哪些操作需要 Human-in-the-loop？

## 考察重点

- 是否能判断自动化边界
- 是否理解人工确认不是简单弹窗
- 是否能按风险设计确认策略

## 核心结论

不可逆、高价值、涉及隐私、影响外部用户或生产环境的操作，通常需要人工确认。

## 参考回答

需要人工确认的操作包括删除数据、付款、提交代码、部署生产、发送邮件、修改客户信息、访问敏感文件、调用外部服务产生费用等。这类操作一旦执行，可能造成真实损失。

Human-in-the-loop 不只是问“是否继续”。系统应该展示 Agent 打算做什么、为什么做、影响哪些对象、风险是什么、是否可撤销。这样人才能做出有效判断。

对于低风险重复任务，可以在系统积累足够评测和日志后逐步减少确认，但要保留审计和回滚能力。

## 深度展开

人工确认也是产品设计问题。如果每一步都确认，用户体验会很差；如果完全不确认，风险太高。比较好的方式是按风险动态确认：低风险自动执行，中风险预览确认，高风险强审批。

## 工程实践

- 确认界面展示 diff、目标对象和影响范围
- 对高风险动作要求二次确认
- 审批结果写入 trace
- 支持用户设置权限策略

## 常见误区

- 所有工具调用都让用户确认，导致 Agent 不可用
- 确认信息太模糊，用户无法判断
- 用户确认后就不记录责任链

## 追问方向

- 如何平衡安全和效率？
- 什么时候可以取消人工确认？
- 用户误批准了怎么办？

## 学习建议

回答时用“发邮件前预览”和“生产删除必须审批”两个例子，容易讲清楚分级思想。
