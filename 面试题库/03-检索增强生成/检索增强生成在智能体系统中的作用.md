# RAG 在 Agent 系统中起什么作用？

## 面试问题

RAG 在 Agent 系统中通常起什么作用？

## 考察重点

- 是否理解 RAG 是知识获取能力
- 是否知道 Agent 可以把检索作为工具调用
- 是否能说明 RAG 结果会影响后续决策

## 核心结论

RAG 为 Agent 提供外部知识来源，使 Agent 能基于私有文档、历史案例、代码说明或业务规则进行决策。

## 参考回答

Agent 需要完成任务时，模型自身知识往往不够。RAG 可以把外部知识库接入 Agent，让它在需要时查询相关资料。比如客服 Agent 可以检索产品文档，代码 Agent 可以检索项目规范，合规 Agent 可以检索制度条款。

在架构上，RAG 通常表现为一个检索工具。Agent 判断需要知识时，调用检索工具获得相关片段，再根据片段继续回答或执行任务。

## 深度展开

RAG 对 Agent 的影响比普通问答更大。普通 RAG 检索错了，可能只是回答错；Agent 检索错了，可能基于错误资料继续调用工具，造成错误操作。因此 Agent 场景下更需要关注检索质量、来源可信度、时间有效性和引用可追溯性。

RAG 也不能解决所有问题。它提供的是知识，不是行动策略。Agent 仍然需要规划、权限、工具调用和评测。

## 工程实践

- 检索结果带来源、段落位置和更新时间
- 对长结果做压缩，但保留关键证据
- 对关键决策要求引用依据
- 通过测试集验证 top-k 召回质量

## 常见误区

- 认为接入向量数据库就等于完成 Agent
- 忽略检索错误会影响工具调用
- 不保存引用来源，导致结果不可验证

## 追问方向

- Agent 应该何时触发检索？
- 检索结果冲突时怎么办？
- 如何评估 RAG 对 Agent 成功率的贡献？

## 学习建议

把 RAG 视为 Agent 的“查资料能力”，而不是 Agent 的全部。
