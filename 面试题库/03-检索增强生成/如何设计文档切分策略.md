# 如何设计文档切分策略？

## 面试问题

做 RAG 时，你会如何设计文档切分策略？

## 考察重点

- 是否理解 chunk 会影响召回和生成质量
- 是否知道切分应尊重文档结构和语义边界
- 是否能结合不同文档类型调整策略

## 核心结论

好的切分策略不是固定字数切一刀，而是在语义完整性、检索精度和上下文长度之间做平衡。

## 参考回答

文档切分要先看文档类型。普通文章可以按标题、段落和小节切分；技术文档可以按 API、函数、配置项切分；FAQ 可以按问答对切分；代码可以按类、函数或模块切分。

chunk 太小会丢上下文，模型拿到片段后不知道背景。chunk 太大又会引入噪声，导致检索命中但答案不精确。通常会设置 chunk size 和 overlap，并保存标题、来源、章节、时间等 metadata。

## 深度展开

切分策略不是一次性拍脑袋决定的。应该用真实问题测试：某个问题的答案是否被完整切进一个或少数几个 chunk？检索出来的 top-k 是否包含答案？如果答案分散在多个 chunk，是否需要父子 chunk、上下文扩展或重排？

在 Agent 场景中，切分还影响后续决策。如果 chunk 缺少限制条件，Agent 可能只看到一半规则就执行错误动作。

## 工程实践

- 优先按标题层级和语义边界切分
- 给 chunk 保存丰富 metadata
- 对长文档保留父章节信息
- 用固定问题集比较不同切分策略

## 常见误区

- 所有文档都用同一个 chunk size
- 只看 embedding 分数，不看返回内容质量
- 切分后丢失标题和来源

## 追问方向

- overlap 的作用是什么？
- PDF 表格如何切分？
- 父子 chunk 有什么用？

## 学习建议

做实验比背概念更重要。拿同一份文档，用 3 种切分策略跑同一批问题，差异会非常明显。
