# 召回结果不准时如何排查？

## 面试问题

如果 RAG 召回结果不准，你会如何系统排查？

## 考察重点

- 是否能从数据、切分、查询、embedding、排序逐层定位
- 是否能区分召回问题和生成问题
- 是否具备评测意识

## 核心结论

RAG 排查要先确认答案是否在库里，再检查切分是否保留语义，随后检查 query、embedding、metadata 过滤和 rerank。

## 参考回答

我会先确认知识库是否包含正确答案。如果源文档都没有，召回不准其实是数据覆盖问题。然后检查文档切分，看答案是否被拆散、标题是否丢失、表格是否解析错误。

接着看 query。用户问题可能太口语化，需要 query rewrite；也可能需要关键词检索和向量检索混合。再检查 embedding 模型是否适合语言和领域，metadata 过滤是否过严。最后看 top-k 排序，如果相关内容进了候选但排位低，可以加入 reranker。

## 深度展开

召回问题和生成问题要分开看。如果 top-k 里没有正确资料，是检索问题；如果 top-k 有正确资料但模型答错，是上下文组织、提示词或生成忠实度问题。很多人只看最终答案，会导致改错方向。

## 工程实践

- 打印每次检索的 query、top-k、score、source
- 建立黄金测试集，标注每题应命中文档
- 分别评估 recall@k 和最终答案质量
- 对线上失败问题回流到测试集

## 常见误区

- 一看到回答错就改 prompt
- 不检查知识库是否真的有答案
- 盲目提高 top-k，导致上下文噪声变多
