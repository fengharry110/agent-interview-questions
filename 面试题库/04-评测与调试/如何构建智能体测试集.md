# 如何构建 Agent 测试集？

## 面试问题

你会如何为 Agent 构建测试集？

## 考察重点

- 是否知道测试集要来自真实任务
- 是否能设计成功标准和评分规则
- 是否覆盖异常、权限和攻击场景

## 核心结论

Agent 测试集不是问答列表，而是一组带环境、工具、预期行为和评分规则的任务案例。

## 参考回答

我会先从真实用户任务或业务流程中收集案例，把每个案例整理成任务描述、初始状态、可用工具、期望结果和评分标准。对于 Agent 来说，有些任务没有唯一答案，所以成功标准要描述行为和结果，而不是只写固定文本。

测试集要覆盖简单任务、多步任务、工具失败、权限不足、信息缺失、恶意输入和边界情况。每次执行都保存 trace，方便判断失败发生在哪一步。

## 深度展开

测试集要避免只覆盖 happy path。一个 Agent 在 demo 中成功，不代表在真实场景可靠。真实用户会给模糊目标、错误信息、不完整上下文，工具也会超时、返回空结果或权限不足。测试集越接近真实环境，价值越高。

## 工程实践

- 用表格维护 case_id、任务、环境、工具、期望行为、评分规则
- 给每个 case 标注难度和风险等级
- 把线上失败案例回流到测试集
- 定期检查测试集是否过拟合 prompt

## 常见误区

- 只测试模型回答，不测试工具执行
- 只保留成功案例
- 成功标准含糊，导致评测不可重复

## 追问方向

- 没有唯一答案的任务怎么评测？
- 如何构造安全攻击用例？
- 测试集是否会被模型记住？

## 学习建议

可以先为自己的项目写 30 个测试任务，这是面试中非常能证明工程能力的材料。
