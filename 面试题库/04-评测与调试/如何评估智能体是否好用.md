# 如何评估一个 Agent 是否好用？

## 面试问题

你会如何评估一个 Agent 是否真正好用？

## 考察重点

- 是否知道 Agent 不能只靠主观体验评估
- 是否能同时评估最终结果和执行过程
- 是否理解评测集、指标和失败分析的重要性

## 核心结论

评估 Agent 要看它是否稳定完成真实任务，而不是只看某一次 demo 是否回答得漂亮。

## 参考回答

我会先构建代表性任务集，每个任务定义输入、可用工具、初始环境和成功标准。Agent 的评估不仅看最终答案是否正确，还要看过程是否合理，例如是否调用了正确工具、是否遵守权限、是否在合理步骤内完成。

常见指标包括任务成功率、工具调用正确率、平均步骤数、失败率、延迟、成本、安全违规率和人工介入率。对于开放式输出，还可以设计评分 rubric，评估完整性、准确性、可追溯性和格式合规性。

## 深度展开

Agent 的失败往往不是单点失败，而是链式失败。比如检索结果错了，导致计划错；计划错了，导致调用工具错；工具错了，模型又编造一个合理解释。因此评测必须保存执行轨迹，而不是只保存最终回答。

生产中还要做回归评测。每次改 prompt、模型、工具 schema 或检索策略，都要跑同一批任务，确认没有引入新问题。

## 工程实践

- 建立 20 到 100 个代表性任务作为基准集
- 保存每次执行的 trace
- 给失败打标签：规划、检索、工具、权限、生成、上下文
- 对关键业务场景设置上线门槛

## 常见误区

- 只看最终回答，不看工具调用过程
- 用几个成功 demo 代替系统评测
- 没有固定测试集，导致每次优化都凭感觉
