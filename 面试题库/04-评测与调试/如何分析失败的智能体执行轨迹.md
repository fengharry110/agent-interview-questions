# 如何分析一次失败的 Agent 执行轨迹？

## 面试问题

如果一次 Agent 执行失败了，你会如何分析它的 trace？

## 考察重点

- 是否能逐步定位失败来源
- 是否知道 trace 应该记录哪些内容
- 是否能把失败转化为系统改进

## 核心结论

分析失败 trace 要沿着目标理解、计划、工具选择、参数、工具结果、后续推理的链路逐步排查。

## 参考回答

我会先看 Agent 是否正确理解了用户目标。如果目标理解错，后面都会错。然后看计划是否合理，是否选择了合适工具。接着检查工具参数是否符合 schema，工具返回是否成功，返回内容是否被 Agent 正确理解。

如果使用 RAG，还要看检索结果是否相关。如果涉及权限，要看是否正确触发审批。如果模型忽略了前面的 observation，可能是上下文组织或 prompt 问题。

最后要把失败打标签，并决定修复方式：改 prompt、改工具 schema、改检索策略、改权限逻辑、补测试用例，或者调整产品交互。

## 深度展开

好的 trace 不是普通日志，而是 Agent 决策链。它应该能回答：Agent 当时知道什么、为什么这么做、调用了什么、结果是什么、下一步为何改变。没有 trace 的 Agent 系统很难生产化，因为失败时只能猜。

## 工程实践

- 记录每一步输入、输出、工具、参数、结果和耗时
- 敏感字段脱敏
- 失败案例自动生成回归测试候选
- 用 failure taxonomy 统计高频失败类型

## 常见误区

- 只看最后回答，不看中间过程
- 日志太散，无法还原完整任务
- 发现失败后只改 prompt，不改系统边界

## 追问方向

- trace 如何存储？
- 如何保护 trace 中的隐私？
- 如何从 trace 自动生成评测样本？

## 学习建议

调 Agent 要像调分布式系统一样看链路，不要像调普通 prompt 一样只看输出文本。
