# Agent 和 RAG 应用有什么区别？

## 面试问题

Agent 和 RAG 应用有什么区别？二者之间是什么关系？

## 考察重点

- 是否能区分“知识增强”和“任务执行”
- 是否理解 RAG 可以作为 Agent 的一个工具
- 是否知道 RAG 解决不了所有 Agent 问题

## 核心结论

RAG 主要解决模型回答时“参考什么知识”，Agent 主要解决为了完成目标“下一步做什么动作”。RAG 可以被 Agent 调用，但 RAG 本身不等于 Agent。

## 参考回答

RAG 是 Retrieval-Augmented Generation，也就是检索增强生成。它的典型流程是先根据用户问题检索相关文档，再把文档内容放进上下文，让模型基于资料回答。RAG 的核心价值是补充私有知识、降低幻觉、提供引用来源。

Agent 的核心是决策和执行。它不仅回答问题，还会决定是否需要检索、是否需要调用 API、是否需要写文件、是否需要继续下一步。RAG 在 Agent 系统中通常是一个知识工具，而不是整个系统本身。

举例来说，企业知识库问答可以只是 RAG：用户问问题，系统检索文档并回答。但如果用户说“阅读这三份制度，判断我的报销单是否合规，并生成修改建议”，系统可能需要检索、比较规则、调用表单工具、生成报告，这就更接近 Agent。

## 深度展开

很多面试候选人会把 Agent 和 RAG 混在一起，这是常见扣分点。RAG 的失败通常来自知识库覆盖、切分、召回、排序和上下文组织。Agent 的失败还包括目标理解、规划错误、工具选择错误、权限处理错误和多步状态管理错误。

所以二者的评测方法也不同。RAG 要看召回率、引用准确性和答案忠实度；Agent 还要看任务完成率、工具调用正确率、步骤数、成本、安全违规率等。

## 工程实践

- 把检索设计成 Agent 可调用的工具
- 让 Agent 在需要知识时检索，而不是每一步都检索
- 检索结果要带来源、时间和置信度
- 对检索内容进行压缩，避免上下文被噪声淹没

## 常见误区

- 认为做了向量数据库就是 Agent
- 认为 Agent 一定要有 RAG
- 忽略 RAG 错误会放大 Agent 后续决策错误

## 追问方向

- Agent 如何判断什么时候需要检索？
- RAG 检索错了，Agent 应该如何自我纠错？
- 什么场景只需要 RAG，不需要 Agent？

## 学习建议

回答时可以用一句话收束：RAG 是知识层能力，Agent 是行动层能力，二者可以组合，但不能混为一谈。
