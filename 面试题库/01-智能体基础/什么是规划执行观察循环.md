# 什么是 Plan-Act-Observe 循环？

## 面试问题

请解释 Agent 中常见的 Plan-Act-Observe 循环，它解决了什么问题？

## 考察重点

- 是否理解 Agent 的多步执行闭环
- 是否知道每一步的输入输出
- 是否能说明停止条件、错误处理和状态更新

## 核心结论

Plan-Act-Observe 是 Agent 将复杂目标拆成连续动作的基本执行模式：先计划下一步，再执行工具，最后观察结果并更新状态。

## 参考回答

Plan 阶段，Agent 根据用户目标、当前上下文和已有观察结果，决定下一步应该做什么。Act 阶段，系统把这个决策转换成具体动作，比如调用搜索工具、读取文件、执行命令或调用业务 API。Observe 阶段，工具返回结果，Agent 根据新信息判断任务是否完成，或者进入下一轮循环。

这个循环的价值在于，它允许 Agent 在不确定环境中逐步探索，而不是一次性生成完整答案。比如代码修复任务中，Agent 可能先运行测试，观察失败信息，再定位文件，再修改代码，再重新测试。

## 深度展开

真正的工程问题在于循环边界。没有边界的 Agent 可能无限重试、重复调用工具、消耗大量 token，甚至执行危险动作。因此系统必须设置最大步骤数、超时、预算、工具权限和失败退出策略。

此外，Plan 不一定总是长篇自然语言规划。生产系统里，Plan 可以是结构化状态，也可以是当前步骤的短决策。过度规划可能增加成本，完全不规划又可能导致短视行为。

## 工程实践

- 每一轮记录 goal、state、action、observation、decision
- 失败时返回结构化错误，而不是只返回异常文本
- 对可重试错误设置次数上限
- 对不可逆操作加入人工确认

## 常见误区

- 以为 Agent 必须先生成完整长计划
- 忽略 observation 的质量，导致后续推理建立在错误信息上
- 没有停止条件，导致循环失控

## 追问方向

- 如何防止 Agent 无限循环？
- Plan 应该展示给用户吗？
- Observation 太长时如何压缩？

## 学习建议

自己写一个最小循环：输入目标，模型选择工具，执行工具，拿结果继续问模型。只要做一遍，就会真正理解 Agent 的骨架。
