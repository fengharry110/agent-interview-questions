# 什么是 Agent？它和普通聊天机器人有什么区别？

## 面试问题

什么是 LLM Agent？它和普通聊天机器人最大的区别是什么？

## 考察重点

- 是否能把 Agent 从“会聊天”提升到“能执行任务”的层面理解
- 是否知道 Agent 由目标、状态、工具、执行循环和停止条件组成
- 是否能说清楚 Agent 为什么更强，也为什么更危险

## 核心结论

Agent 不是一种单独的模型，而是一种围绕大模型构建的任务执行系统。普通聊天机器人主要生成回答，Agent 则可以根据目标主动选择动作、调用工具、观察结果并继续推进任务。

## 参考回答

普通聊天机器人通常是输入一段文字，然后输出一段文字，它的边界主要在对话本身。Agent 的边界更大，它会围绕一个目标持续行动。比如用户说“帮我修复这个项目的测试失败”，普通聊天机器人可能给出建议；Agent 则可能读取报错、搜索代码、修改文件、运行测试，然后总结改动。

从系统结构看，Agent 至少包含几个部分：任务目标、可用工具、当前状态、决策逻辑、工具执行结果、停止条件和错误处理。大模型在其中通常扮演“决策器”或“控制器”的角色，但真正的能力来自模型和工具系统的组合。

因此，Agent 的关键不是模型会不会说得像人，而是它能否可靠地把目标拆成行动，并通过外部工具改变任务状态。

## 深度展开

面试中可以进一步说明，Agent 的难点主要来自不确定性。模型可能理解错目标、选择错工具、生成错误参数、误解工具返回结果，或者在多步执行中丢失上下文。所以 Agent 工程不是简单接一个模型 API，而是要设计工具边界、权限控制、执行日志、评测集和失败恢复机制。

## 工程实践

- 给 Agent 明确的工具列表，而不是让它拥有无限能力
- 每一步 action 和 observation 都要记录，便于调试
- 设置最大步骤数、最大重试次数和预算限制
- 对写文件、发请求、删数据等操作加入权限控制

## 常见误区

- 误以为“接入大模型就是 Agent”
- 误以为 Agent 越自主越好
- 只关注 prompt，不关注工具、状态和评测

## 追问方向

- Agent 是否一定需要规划能力？
- 为什么 Agent 比普通聊天机器人更难评测？
- 如何判断一个需求是否值得做成 Agent？

## 学习建议

用一个代码修复 Agent 做例子最容易讲清楚：它需要读文件、查错误、改代码、跑测试，这些动作能体现 Agent 和聊天机器人的本质区别。
