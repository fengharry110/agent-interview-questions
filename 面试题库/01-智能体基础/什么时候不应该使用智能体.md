# 什么时候不应该使用 Agent？

## 面试问题

什么情况下不应该使用 Agent？你如何判断一个需求是否适合 Agent？

## 考察重点

- 是否具备工程取舍能力
- 是否理解 Agent 的成本、延迟和不确定性
- 是否能用任务特征判断方案，而不是盲目追热点

## 核心结论

流程固定、规则清晰、结果必须高度确定的任务，通常不应该优先使用 Agent。Agent 适合目标明确但路径不固定、需要动态决策和工具协作的任务。

## 参考回答

如果一个需求可以用固定规则、普通后端流程或工作流引擎稳定解决，就不应该为了使用 Agent 而使用 Agent。比如表单校验、固定报表生成、简单 FAQ、确定性数据转换，这些任务用传统程序更便宜、更稳定、更容易测试。

Agent 的优势在于处理开放路径任务。比如用户目标明确，但中间步骤依赖环境反馈：代码调试、复杂资料研究、多系统信息收集、异常工单处理等。这些任务很难预先写死所有分支，Agent 才有价值。

## 深度展开

Agent 引入了模型不确定性、多步成本、权限风险和调试复杂度。一个成熟工程师应该先问：这个任务是否真的需要模型决策？是否需要工具调用？路径是否会随观察结果变化？失败后是否可恢复？如果答案都是否，那么 Agent 可能是过度设计。

## 工程实践

- 先用规则和工作流覆盖稳定路径
- 只把不确定、开放、多分支部分交给 Agent
- 对 Agent 输出增加校验和审批
- 用评测数据判断 Agent 是否真的提升效果

## 常见误区

- 认为所有 LLM 应用都应该 Agent 化
- 低估 Agent 的运维和评测成本
- 用 Agent 替代本该确定执行的业务逻辑

## 追问方向

- Agent 和传统工作流引擎如何结合？
- 如何把一个复杂任务拆成确定部分和 Agent 部分？
- Agent 失败时如何回退？

## 学习建议

面试时敢于说“不该用 Agent”的场景，反而能体现你的工程判断力。
