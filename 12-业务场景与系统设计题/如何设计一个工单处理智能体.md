# 如何设计一个工单处理智能体？

## 面试问题

如何设计一个工单处理智能体？

## 考察重点

- 是否理解工单处理的核心流程（接收→分类→分配→处理→反馈）
- 是否能说清楚智能体在工单处理中做哪些、人做哪些
- 是否了解工单系统的特殊要求（准确性要求高、可追溯、可升级）
- 是否有工单系统或客服系统的设计经验

## 核心结论

工单处理智能体的核心设计思路是"人机协作"：智能体负责重复性、标准化的工作（工单分类、信息提取、标准回复），复杂或敏感的工单升级到人工处理。关键设计点是工单状态机的严谨性——智能体可以自由行动，但不能违反状态转换规则。

### 系统架构

```python
class TicketAgentSystem:
    """
    工单处理智能体系统架构
    """
    
    def __init__(self):
        self.classifier = TicketClassifier()
        self.processor = TicketProcessor()
        self.escalation = EscalationManager()
        self.state_machine = TicketStateMachine()
    
    async def process_ticket(self, ticket: Ticket) -> TicketResult:
        """处理一张工单"""
        # 1. 工单分类
        classification = await self.classifier.classify(ticket)
        ticket.category = classification.category
        ticket.priority = classification.priority
        
        # 2. 状态转换：已接收 → 处理中
        self.state_machine.transition(ticket, "processing")
        
        # 3. 自动处理或升级
        if classification.should_auto_process():
            result = await self.processor.process(ticket)
            
            if result.success:
                self.state_machine.transition(ticket, "resolved")
                return TicketResult(
                    status="resolved",
                    resolution=result.resolution
                )
            else:
                return await self.escalation.escalate(
                    ticket, reason=result.failure_reason
                )
        else:
            return await self.escalation.escalate(
                ticket, reason="需要人工处理"
            )
```

### 工单状态机

```python
class TicketStateMachine:
    """
    工单状态机——智能体必须遵守的状态转换规则
    """
    
    STATES = [
        "new",          # 新创建
        "processing",   # 处理中
        "waiting_user", # 等待用户回复
        "escalated",    # 已升级
        "resolved",     # 已解决
        "closed"        # 已关闭
    ]
    
    ALLOWED_TRANSITIONS = {
        "new": ["processing", "closed"],
        "processing": ["waiting_user", "escalated", "resolved"],
        "waiting_user": ["processing", "closed"],
        "escalated": ["processing", "resolved"],
        "resolved": ["closed"],
        "closed": []
    }
    
    def transition(self, ticket: Ticket, target: str):
        current = ticket.status
        if target not in self.ALLOWED_TRANSITIONS.get(current, []):
            raise InvalidTransition(
                f"不能从 {current} 转换到 {target}"
            )
        ticket.status = target
        ticket.transition_history.append({
            "from": current,
            "to": target,
            "timestamp": datetime.now(),
            "agent": "system"
        })
```

### 自动处理逻辑

```python
class TicketAutoProcessor:
    """
    工单自动处理引擎
    """
    
    async def process(self, ticket: Ticket) -> ProcessResult:
        if ticket.category == "technical_issue":
            return await self._process_technical(ticket)
        elif ticket.category == "billing":
            return await self._process_billing(ticket)
        elif ticket.category == "account":
            return await self._process_account(ticket)
        else:
            return ProcessResult(
                success=False,
                failure_reason="不支持的自动处理类型"
            )
    
    async def _process_technical(self, ticket: Ticket) -> ProcessResult:
        solutions = await self.knowledge_base.search(
            query=ticket.description,
            category="technical"
        )
        
        if not solutions:
            return ProcessResult(
                success=False,
                failure_reason="知识库中未找到相关解决方案"
            )
        
        best_solution = await self.llm.select_best(
            problem=ticket.description,
            solutions=solutions
        )
        
        if best_solution.is_executable:
            execution = await self.execute_solution(best_solution)
            if execution.success:
                return ProcessResult(
                    success=True,
                    resolution=f"已自动执行解决方案：{best_solution.title}"
                )
        
        return ProcessResult(
            success=True,
            resolution=f"建议方案：{best_solution.content}"
        )
```

### 升级策略

```python
class EscalationManager:
    """
    工单升级管理
    """
    
    ESCALATION_RULES = [
        {
            "condition": lambda t: t.priority == "P0",
            "target": "senior_team",
            "timeout": 300
        },
        {
            "condition": lambda t: t.priority == "P1" and t.waiting_time > 3600,
            "target": "team_lead",
            "timeout": 1800
        },
        {
            "condition": lambda t: t.auto_attempts >= 3,
            "target": "specialist",
            "timeout": 3600
        }
    ]
    
    async def escalate(self, ticket: Ticket, reason: str) -> TicketResult:
        for rule in self.ESCALATION_RULES:
            if rule["condition"](ticket):
                ticket.escalated_to = rule["target"]
                break
        
        return TicketResult(
            status="escalated",
            escalation_reason=reason,
            assigned_to=ticket.escalated_to
        )
```

## 常见误区

- **追求 100% 自动处理**：有些工单天生需要人工判断。设定合理的自动处理率目标（60-80%）比盲目追求高自动化更实际
- **状态机不够严谨**：智能体可能跳过中间状态直接标记为"已解决"。状态转换必须由系统校验
- **升级策略太死板**：所有工单都走同一个升级流程会导致紧急工单被延误。需要根据优先级动态调整

## 追问方向

- 自动解决率的目标定多少合理？如果达不到怎么办？
- 如何设计工单的 SLA 并确保智能体能遵守？
- 智能体处理失败后转人工，怎么保证人工接手时信息不丢失？

## 学习建议

从一类最简单的工单开始（如密码重置），让智能体全自动处理。跑通后逐步扩展到更复杂的工单类型。重点是设计好状态机和升级策略——这决定了系统是"有序的自动化"还是"混乱的自动化"。
