# 如何设计一个数据分析智能体

## 面试问题

设计一个数据分析智能体，用户可以用自然语言提出问题，智能体自动完成数据查询、分析、可视化，并输出分析报告。你会如何设计这个系统？

## 考察重点

- 是否理解 NL2SQL 的局限性：不是所有分析需求都能转成 SQL
- 是否能说清楚数据分析的工作流：数据理解 -> 数据清洗 -> 探索分析 -> 建模 -> 可视化 -> 结论
- 是否了解如何控制分析的安全边界：防止 SQL 注入、防止泄露敏感数据
- 是否有实际的数据分析平台或 BI 工具的设计经验

## 核心结论

数据分析智能体的本质是一个"自动化数据分析 Pipeline"的调度系统，核心挑战不是生成 SQL，而是**在不确定中做出正确的分析决策**——选择什么数据、用什么分析方法、如何解读结果。

### 核心组件设计

#### 1. 需求理解与拆解

```typescript
interface AnalysisGoal {
  objective: string;            // 分析目标
  questionType: QuestionType;   // 问题类型
  metrics: string[];            // 涉及的指标
  dimensions: string[];         // 涉及的维度
  timeRange: TimeRange;         // 时间范围
  filters: Filter[];            // 过滤条件
  outputFormat: OutputFormat;   // 输出格式
}

type QuestionType =
  | 'descriptive'    // 描述性分析: "上个月销售额多少"
  | 'diagnostic'     // 诊断分析: "为什么销售额下降了"
  | 'comparative'    // 对比分析: "A 和 B 哪个更好"
  | 'trend'          // 趋势分析: "过去 12 个月的趋势"
  | 'predictive'     // 预测分析: "下个月会怎么样"
  | 'segmentation'   // 分群分析: "用户分群情况"
  | 'correlation';   // 相关性分析: "哪些因素相关"

class AnalysisPlanner {
  async plan(userQuestion: string, schema: Schema): Promise<AnalysisPlan> {
    // 1. 确定问题类型
    const goal = await this.extractGoal(userQuestion);

    // 2. 匹配数据源
    const dataSources = await this.matchDataSources(goal, schema);

    // 3. 生成分析步骤
    const steps = this.generateAnalysisSteps(goal, dataSources);

    return {
      goal,
      dataSources,
      steps,
      estimatedComplexity: this.estimateComplexity(steps)
    };
  }

  private generateAnalysisSteps(
    goal: AnalysisGoal,
    sources: DataSource[]
  ): AnalysisStep[] {
    // 不同类型的分析有不同的步骤
    const templates: Record<QuestionType, AnalysisStep[]> = {
      descriptive: [
        { type: 'query', description: '查询基础数据' },
        { type: 'aggregate', description: '聚合计算指标' },
        { type: 'visualize', description: '生成可视化' }
      ],
      diagnostic: [
        { type: 'query', description: '查询相关数据' },
        { type: 'segment', description: '维度下钻' },
        { type: 'compare', description: '对比分析' },
        { type: 'visualize', description: '生成对比图' },
        { type: 'conclude', description: '分析原因' }
      ],
      predictive: [
        { type: 'query', description: '获取历史数据' },
        { type: 'feature_engineer', description: '特征工程' },
        { type: 'model', description: '训练预测模型' },
        { type: 'forecast', description: '生成预测' },
        { type: 'visualize', description: '生成预测图' }
      ]
    };

    return templates[goal.questionType] ?? templates.descriptive;
  }
}
```

#### 2. 分析执行引擎

```typescript
interface AnalysisStep {
  type: 'query' | 'aggregate' | 'segment' | 'compare'
      | 'model' | 'visualize' | 'conclude';
  description: string;
  code?: string;        // 生成的代码
  result?: any;         // 执行结果
  error?: string;
}

class AnalysisExecutor {
  private sqlGenerator: SQLGenerator;
  private pythonExecutor: PythonSandbox;
  private vizGenerator: VisualizationGenerator;

  async execute(plan: AnalysisPlan): Promise<AnalysisResult> {
    const results: StepResult[] = [];

    for (const step of plan.steps) {
      // 根据步骤类型选择执行方式
      let result: StepResult;

      switch (step.type) {
        case 'query':
          result = await this.executeQuery(step, plan.dataSources);
          break;
        case 'aggregate':
        case 'segment':
        case 'compare':
          result = await this.executeAnalysis(step, results);
          break;
        case 'model':
          result = await this.executeModeling(step, results);
          break;
        case 'visualize':
          result = await this.executeVisualization(step, results);
          break;
        case 'conclude':
          result = await this.executeConclusion(step, results);
          break;
      }

      results.push(result);

      // 检查中间结果是否正常
      if (result.error || this.needsReplan(result)) {
        return this.handleFailure(plan, results, result);
      }
    }

    return {
      steps: results,
      finalReport: await this.generateReport(results, plan.goal)
    };
  }

  private async executeQuery(
    step: AnalysisStep,
    sources: DataSource[]
  ): Promise<StepResult> {
    // 1. 生成 SQL
    const sql = await this.sqlGenerator.generate({
      question: step.description,
      schema: sources[0].schema,
      format: step.outputFormat
    });

    // 2. SQL 安全检查
    const safe = await this.sqlValidator.validate(sql);
    if (!safe) {
      return { error: 'SQL validation failed' };
    }

    // 3. 执行 SQL（带超时和行数限制）
    const data = await this.database.query(sql, {
      timeout: 30000,
      maxRows: 100000
    });

    return { data, sql, rowCount: data.length };
  }
}
```

#### 3. NL2SQL 生成器

```typescript
class SQLGenerator {
  private model: LLM;

  async generate(params: {
    question: string;
    schema: Schema;
    previousSteps?: StepResult[];
  }): Promise<string> {
    const prompt = `
You are a SQL expert. Convert the following question to SQL.

## Schema
${formatSchema(params.schema)}

## Question
${params.question}

## Rules
- Use proper JOINs, not subqueries when possible
- Always use LIMIT 1000 unless specified
- Use table aliases for readability
- Do not modify data (SELECT only)
- Use EXISTS instead of IN for subqueries

## Available Tables and Relationships
${formatRelations(params.schema.relations)}

Generate ONLY the SQL query, no explanation.
`;

    const sql = await this.model.generate(prompt);

    // SQL 验证
    const parsed = this.parser.parse(sql);
    if (!this.validator.validate(parsed, params.schema)) {
      // 无效 SQL，重试一次
      return this.retryWithFix(sql, parsed.errors);
    }

    return sql;
  }

  private retryWithFix(badSql: string, errors: ParseError[]): Promise<string> {
    const fixPrompt = `
The previous SQL has errors:
${errors.map(e => `- ${e.message}`).join('\n')}

SQL:
${badSql}

Fix the SQL:
`;
    return this.model.generate(fixPrompt);
  }
}
```

#### 4. Python 分析环境

```typescript
class PythonSandbox {
  private readonly ALLOWED_LIBRARIES = [
    'pandas', 'numpy', 'scipy', 'sklearn',
    'matplotlib', 'seaborn', 'plotly',
    'statsmodels', 'datetime', 'json', 'math'
  ];

  private readonly FORBIDDEN_PATTERNS = [
    /import\s+(os|sys|subprocess|shutil)/,
    /__import__\(/,
    /exec\(/,
    /eval\(/,
    /open\(/,
    /\.\.\/\.\.\//
  ];

  async execute(code: string, data: DataFrame): Promise<ExecutionResult> {
    // 安全检查
    for (const pattern of this.FORBIDDEN_PATTERNS) {
      if (pattern.test(code)) {
        return { error: 'Security violation: forbidden pattern detected' };
      }
    }

    // 在沙箱中执行
    return this.sandbox.runPython({
      code: `
import pandas as pd
import numpy as np

# 注入数据
df = pd.DataFrame(${JSON.stringify(data)})

${code}
      `,
      timeout: 60000,
      memoryLimit: '512MB'
    });
  }
}
```

```typescript
class VisualizationGenerator {
  chartTypeMap: Record<string, string[]> = {
```

## 深度展开

### 分析模式分类

| 模式 | 用户提问示例 | 实现方式 | 复杂度 |
|-----|-------------|---------|--------|
| 查询型 | "上月销售额多少" | NL2SQL + 直接查询 | 低 |
| 对比型 | "A 和 B 的转化率差异" | 分组聚合 + 统计检验 | 中 |
| 趋势型 | "过去 6 个月用户增长趋势" | 时间序列聚合 + 可视化 | 中 |
| 归因型 | "为什么订单转化率下降了" | 下钻分析 + 相关性分析 | 高 |
| 预测型 | "下季度营收预测" | 历史数据 + 时间序列模型 | 高 |
| 探索型 | "看看数据里有什么有趣的发现" | 自动 EDA + 异常检测 | 最高 |

### 处理不确定性的策略

```typescript
class UncertaintyHandler {
  async handleUncertainty(
    result: AnalysisResult,
    question: string
  ): Promise<AnalysisResult> {
    // 1. 统计显著性检测
    if (result.type === 'comparison') {
      result.statisticalSignificance = await this.significanceTest(result);
      if (!result.statisticalSignificance) {
        result.warnings.push('差异在统计上不显著');
      }
    }

    // 2. 数据质量检查
    const quality = await this.checkDataQuality(result.sourceData);
    if (quality.hasMissingData) {
      result.warnings.push(`数据中存在 ${quality.missingRate}% 的缺失值`);
    }
    if (quality.hasOutliers) {
      result.warnings.push('检测到异常值');
    }

    // 3. 置信区间
    if (result.type === 'predictive') {
      result.confidenceInterval = this.calculateCI(result);
    }

    return result;
  }
}
```

### 安全控制

```typescript
class DataSecurityGuard {
  private readonly SENSITIVE_COLUMNS = [
    'phone', 'email', 'id_card', 'password',
    'credit_card', 'ssn', 'address'
  ];

  async sanitizeResult(result: AnalysisResult): Promise<AnalysisResult> {
    // 1. 脱敏敏感列
    if (result.data) {
      for (const col of this.SENSITIVE_COLUMNS) {
        if (result.data.columns.includes(col)) {
          result.data = result.data.dropColumn(col);
          result.warnings.push(`已移除敏感字段: ${col}`);
        }
      }
    }

    // 2. 小样本抑制
    if (result.data && result.data.length < 10) {
      result.warnings.push('数据量太小，统计结果可能不准确');
    }

    // 3. 差分隐私
    if (this.containsPII(result)) {
      result.aggregatedOnly = true;
      result.warnings.push('涉及个人数据，仅展示聚合结果');
    }

    return result;
  }
}
```

### 1. 多轮分析对话的状态管理

```typescript
interface AnalysisSession {
  sessionId: string;
  history: AnalysisTurn[];
  currentPlan: AnalysisPlan;
  intermediateResults: Map<string, any>;
  userPreferences: {
    preferredVizType?: string;
    defaultTimeRange?: string;

class AnalysisSessionManager {
  async continueAnalysis(
    sessionId: string,
    newQuestion: string
  ): Promise<AnalysisResult> {
    const session = await this.loadSession(sessionId);

    // 检查新问题是独立问题还是对之前分析的追问
    const relation = await this.analyzeQuestionRelation(
      newQuestion,
      session.history
    );

    if (relation.type === 'follow_up') {
      // 基于已有结果继续分析
      return this.refineAnalysis(session, relation);
    } else {
      // 新的独立分析
      return this.startNewAnalysis(newQuestion, session);
    }
  }
}
```

### 2. 执行计划和实际调度的差异

```typescript
class AdaptiveExecutor {
  async execute(plan: AnalysisPlan): Promise<AnalysisResult> {
    const actualSteps: ExecutedStep[] = [];
    let currentPlan = plan;

    for (let i = 0; i < plan.steps.length; i++) {
      const step = plan.steps[i];
      const result = await this.executeStep(step, actualSteps);

      // 根据结果动态调整后续计划
      if (result.surpriseDetected) {
        // 数据中发现意外模式,调整计划
        currentPlan = await this.replan(
          currentPlan,
          i,
          result
        );
      }

      actualSteps.push(result);
    }

    return { plan: currentPlan, executedSteps: actualSteps };
  }
}
```

## 常见误区

### 误区 1：NL2SQL 可以解决所有分析问题

NL2SQL 只能处理"查询型"问题。对于需要统计建模、机器学习或复杂推理的分析任务，需要使用 Python 代码生成或直接调用分析库。

### 误区 2：认为模型生成的图表自动就是好的

模型选图表时经常选错类型。例如用饼图展示 20 个分类，或用折线图展示没有时间维度的数据。需要在模型选择后做一层规则校验。

### 误区 3：忽视数据质量的前置检查

在查询之前应该先做数据质量检查：字段缺失率、异常值、数据类型一致性。否则分析结果可能充满误导。

### 误区 4：允许无限期的分析执行

分析任务可能因为数据量大或模型复杂度高而执行很久。必须设置超时阈值，并在超时时提供中间结果或降级输出。

## 追问方向

1. **"如果用户问了一个模糊的问题，比如'分析一下用户数据'，系统应该怎么做？"**

    系统应该主动引导用户明确分析目标。可以反问用户最关心的指标是什么、时间范围是什么，或者提供常见的分析模板供选择。在无明确方向时，可以做自动 EDA 来发现数据的基本特征和异常。

2. **"如何防止用户通过分析系统获取未授权的敏感数据？"**

    在数据访问层做列级别权限控制。敏感字段默认不暴露，聚合结果中抑制小样本（如少于 10 条的数据不展示）。同时记录所有查询操作日志，便于审计。

3. **"如何验证模型生成的文本分析结论是正确的？"**

    结论需要和数据交叉验证：对结论中的每个断言，要求在数据中找到支持证据。例如，如果模型说"转化率在 3 月显著下降"，需要同时输出 3 月和 2 月的转化率数据供验证。

4. **"当分析结果和用户的直觉预期不一致时，系统怎么做？"**

    不要盲目迎合用户的预期。展示数据支持的结论和置信度，同时提供可能的解释（如数据偏差、样本问题）。让用户决定是否信任分析结果。
