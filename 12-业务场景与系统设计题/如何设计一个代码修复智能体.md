# 如何设计一个代码修复智能体

## 面试问题

设计一个代码修复智能体，能够自动定位代码中的 Bug、安全漏洞或风格问题，并生成修复方案。你会如何设计这个系统？

## 考察重点

- 是否理解代码修复与一般代码生成的区别：修复需要理解"正确行为"和"当前行为"的差异
- 是否能说清楚静态分析、动态分析和 LLM 三者如何配合，而不是把所有工作交给模型
- 是否考虑过修复的安全性问题：模型可能引入更严重的漏洞
- 是否有实际生产环境的代码修复经验，包括如何防止修复产生回归

## 核心结论

代码修复智能体的本质是一个"差异生成"系统，核心挑战不是让模型写代码，而是让模型在**理解代码上下文**和**保持修复前后行为一致性**的前提下，生成最小且正确的差异。

### 核心组件设计

#### 1. 代码理解层

代码理解层不能只靠 LLM 的代码理解能力，需要结合静态分析工具：

```typescript
interface CodeUnderstandingResult {
  repoStructure: FileTree;
  affectedFiles: FileAnalysis[];
  relevantContext: {
    imports: string[];
    typeDefinitions: TypeDef[];
    relatedFunctions: FunctionSig[];
    testCases: TestCase[];
  };
}

class CodeAnalyzer {
  async analyze(repoPath: string, problem: string): Promise<CodeUnderstandingResult> {
    // 1. 静态分析: 获取代码结构
    const ast = await parseAST(repoPath);
    const deps = await buildDependencyGraph(ast);

    // 2. 问题相关代码定位
    const relevantFiles = await this.locateRelevantFiles(problem, deps);

    // 3. 提取局部上下文
    const context = await this.extractContext(relevantFiles, ast);

    return { repoStructure: ast, affectedFiles: relevantFiles, relevantContext: context };
  }

  private async locateRelevantFiles(
    problem: string,
    deps: DependencyGraph
  ): Promise<FileAnalysis[]> {
    // 使用代码搜索 + 语义匹配定位相关文件
    const candidates = await semanticSearch(problem, deps.allFiles);
    return this.pruneByDependency(candidates, deps);
  }
}
```

#### 2. 问题定位模块

问题定位需要分层策略：

| 问题类型 | 定位方法 | 工具链 |
|---------|---------|-------|
| 语法错误 | 编译器/解析器报错 | TypeScript Compiler, ESLint |
| 类型错误 | 类型检查器 | tsc, mypy, pyright |
| 运行时 Bug | 异常追踪 + 日志分析 | Sentry, 堆栈解析 |
| 逻辑错误 | 差分测试/断言失败 | 测试框架输出 |
| 安全漏洞 | SAST 扫描 | Semgrep, CodeQL |
| 性能问题 | Profiling 数据 | 性能分析工具 |

```typescript
interface ProblemReport {
  type: 'syntax' | 'type' | 'runtime' | 'logic' | 'security' | 'performance';
  location: { file: string; line: number; column: number };
  message: string;
  stacktrace?: string;
  severity: 'critical' | 'major' | 'minor';
  expectedBehavior?: string;
  actualBehavior?: string;
}

class ProblemLocalizer {
  async localize(
    errorOutput: string,
    repoContext: CodeUnderstandingResult
  ): Promise<ProblemReport[]> {
    // 尝试多种定位策略，合并结果
    const fromCompiler = this.parseCompilerError(errorOutput);
    const fromAST = this.analyzeASTAnomalies(repoContext);
    return this.mergeReports([fromCompiler, fromAST]);
  }
}
```

#### 3. 修复生成模块

```typescript
interface FixCandidate {
  patch: UnifiedDiff;
  explanation: string;
  confidence: number;  // 0-1
  risk: 'safe' | 'moderate' | 'risky';
  affectedTests: string[];
}

class FixGenerator {
  private model: LLM;
  private validator: FixValidator;

  async generateFixes(
    problems: ProblemReport[],
    context: CodeUnderstandingResult
  ): Promise<FixCandidate[]> {
    const prompt = this.buildFixPrompt(problems, context);
    const modelOutput = await this.model.generate(prompt);

    const candidates = this.parseModelOutput(modelOutput);
    return this.rankCandidates(candidates, context);
  }

  private buildFixPrompt(
    problems: ProblemReport[],
    context: CodeUnderstandingResult
  ): string {
    // 构建包含以下信息的 Prompt:
    // 1. 问题描述 + 错误信息
    // 2. 相关代码片段（带行号）
    // 3. 函数签名和类型定义
    // 4. 已有的测试用例
    // 5. 修复约束（不改公共接口、保持风格等）
    return `
## Problem
${problems.map(p => `- ${p.type}: ${p.message} at ${p.location.file}:${p.location.line}`).join('\n')}

## Context
${formatCode(context.affectedFiles)}

## Constraints
- Do not change public API signatures
- Follow existing code style (${context.repoStructure.styleGuide})
- Ensure all existing tests pass
- Minimize diff size

## Task
Generate a minimal fix for each problem.
`;
  }
}
```

#### 4. 验证执行模块

修复必须经过验证才能应用：

```typescript
class FixValidator {
  async validate(candidate: FixCandidate, context: CodeUnderstandingResult): Promise<ValidationResult> {
    // 1. 语法检查
    const syntaxOk = await this.checkSyntax(candidate.patch);

    // 2. 类型检查
    const typeOk = await this.checkTypes(candidate.patch);

    // 3. 运行测试（在沙箱中）
    const testResults = await this.runTestsInSandbox(candidate.patch);

    // 4. 安全检查：对比修复前后的安全属性
    const securityOk = await this.securityAudit(candidate.patch);

    return {
      passed: syntaxOk && typeOk && testResults.passed && securityOk,
      details: {
        syntaxOk, typeOk, testResults, securityOk
      },
      regressions: testResults.failedTests
    };
  }

  private async runTestsInSandbox(patch: UnifiedDiff): Promise<TestResult> {
    // 在隔离沙箱中:
    // 1. 应用 patch
    // 2. 安装依赖（如有变化）
    // 3. 运行关联测试
    // 4. 清理沙箱
    return sandbox.execute(async () => {
      await applyPatch(patch);
      return await runTestSuite();
    });
  }
}
```

#### 5. 修复决策和提交

```typescript
class FixOrchestrator {
  private analyzer: CodeAnalyzer;
  private localizer: ProblemLocalizer;
  private generator: FixGenerator;
  private validator: FixValidator;

  async fix(problemInput: string, repoPath: string): Promise<FixResult> {
    // 阶段 1: 理解代码
    const context = await this.analyzer.analyze(repoPath, problemInput);

    // 阶段 2: 定位问题
    const problems = await this.localizer.localize(problemInput, context);

    // 阶段 3: 生成修复
    const candidates = await this.generator.generateFixes(problems, context);

    // 阶段 4: 验证
    for (const candidate of candidates) {
      const validation = await this.validator.validate(candidate, context);
      if (validation.passed) {
        return {
          success: true,
          patch: candidate.patch,
          explanation: candidate.explanation,
          validation
        };
      }
    }

    // 如果自动修复都失败，生成诊断报告
    return {
      success: false,
      diagnosis: this.generateDiagnosis(problems, candidates),
      suggestions: this.generateSuggestions(problems)
    };
  }
}
```

## 深度展开

### 修复策略的四种模式

| 模式 | 适用场景 | 实现方式 | 风险 |
|-----|---------|---------|------|
| 直接替换 | 语法错误、类型错误 | 基于编译器报错信息直接生成替换 | 低 |
| 插入代码 | 缺少空检查、缺少错误处理 | 在特定位置插入新代码 | 中 |
| 重构等价 | 逻辑 Bug、性能优化 | 保持语义不变的重写 | 高 |
| 多文件修复 | API 变更、接口升级 | 跨文件分析并同步修改 | 最高 |

### 防止修复导致回归

```typescript
class RegressionGuard {
  // 对比修复前后的行为差异
  async checkBehavioralEquivalence(
    original: CodeUnit,
    fixed: CodeUnit
  ): Promise<EquivalenceResult> {
    // 方法 1: 符号执行比对
    const symbolicDiff = await this.symbolicCompare(original, fixed);

    // 方法 2: 随机输入测试
    const randomInputs = this.generateRandomInputs(original.parameters);
    const fuzzResult = await this.fuzzCompare(original, fixed, randomInputs);

    // 方法 3: 已有测试回归
    const testResult = await this.runRegressionTests(fixed);

    return {
      symbolicallyEquivalent: symbolicDiff.isEqual,
      fuzzConsistent: fuzzResult.allMatch,
      testsPass: testResult.passed
    };
  }
}
```

### 安全修复的特殊处理

安全修复和其他修复不同，因为：
1. 安全修复可能改变程序的外部行为
2. 引入的新依赖可能有漏洞
3. 修复可能被攻击者用来识别漏洞位置

```typescript
class SecurityFixHandler {
  async handleSecurityFix(vuln: Vulnerability): Promise<SecurityFixPlan> {
    return {
      vulnerability: vuln,
      // 安全修复需要额外的验证
      extraValidations: [
        'backward_compat_check',
        'dependency_audit',
        'behavioral_diff_analysis'
      ],
      // 安全修复可能需要静默处理
      disclosurePolicy: 'coordinated_disclosure',
      // 生成修复的多个变体供选择
      variants: await this.generateVariants(vuln)
    };
  }
}
```

## 工程实践

### 1. 增量修复 vs 整体修复

对于大型代码库，应该用增量方式：

```typescript
class IncrementalFixer {
  private readonly MAX_FILES_PER_FIX = 5;

  async fixIncremental(
    errorList: ProblemReport[],
    context: CodeUnderstandingResult
  ): Promise<FixCandidate[]> {
    // 按文件分组
    const byFile = this.groupByFile(errorList);

    // 按依赖顺序处理
    const sorted = this.topologicalSort(byFile, context.dependencies);

    // 逐文件修复，每修复一个文件就运行一次验证
    const fixes: FixCandidate[] = [];
    for (const file of sorted) {
      const fileFixes = await this.fixFile(file, context);
      const validated = await this.validateEach(fileFixes);
      fixes.push(...validated.filter(v => v.passed));
    }

    return fixes;
  }
}
```

### 3. 修复质量评估指标

| 指标 | 计算方法 | 目标值 |
|-----|---------|-------|
| 编译通过率 | 修复后代码能否编译 | > 99% |
| 测试通过率 | 已有测试是否全部通过 | > 95% |
| 修复维持率 | 修复后 30 天内未被回滚 | > 90% |
| 安全达标率 | 修复是否消除漏洞 | 100% |
| 最小化率 | 实际 diff 行数 / 允许最大行数 | < 30% |

### 4. 沙箱隔离

```typescript
class Sandbox {
  async executeInSandbox<T>(fn: () => Promise<T>): Promise<T> {
    // Docker 容器或 VM 中执行
    const container = await this.createContainer({
      image: 'code-fixer-sandbox',
      // 限制资源
      memoryLimit: '2GB',
      cpuLimit: 1,
      networkAccess: false,
      // 挂载代码副本
      mounts: [{ source: '/tmp/code', target: '/workspace' }],
      // 超时控制
      timeout: 120_000
    });

    try {
      await container.start();
      return await container.execute(fn);
    } finally {
      await container.destroy();  // 确保清理
    }
  }
}
```

## 常见误区

### 误区 1：让 LLM 直接输出完整文件

**错误做法**：让 LLM 输出整个修复后的文件，然后用新文件替换旧文件。

**问题**：
- 模型可能改变未涉及的部分
- 格式/缩进可能不一致
- diff 过大，难以 review

**正确做法**：让 LLM 输出统一格式的 diff（unified diff），并逐行验证。

### 误区 2：忽视多文件修复的协调

**错误做法**：一次只修改一个文件，不考虑跨文件的 API 调用关系。

**问题**：修复 A 文件的签名变更导致 B 文件出错。

**正确做法**：构建完整的依赖关系图，按拓扑顺序修复，修复每个文件后立即回归验证。

### 误区 3：信任模型的安全修复

**错误做法**：认为模型生成的安全修复天然安全。

**问题**：模型可能：
- 引入不安全的依赖
- 错误地使用了加密 API
- 修复了表面问题但留下更深层的漏洞

**正确做法**：安全修复必须经过 SAST 扫描和人工审核。

### 误区 4：无差别的全量回归测试

**错误做法**：每次修复都运行整个测试套件。

**问题**：大项目的全量测试可能耗时数小时，影响开发效率。

**正确做法**：基于依赖分析，只运行受影响模块的关联测试。

## 追问方向

1. **"如果模型连续生成的 3 个修复方案都导致测试失败，系统应该怎么做？"**

    应该设计降级策略：回退到问题诊断模式，生成详细的失败分析报告，标注哪些文件需要人工介入，而不是无限尝试。

2. **"如何防止代码修复智能体引入新的安全漏洞？"**

    核心是在验证阶段加入安全审计步骤。具体做法包括：保持修复前后安全策略一致、在新引入的依赖上运行漏洞扫描、以及对比修复前后的数据流变化。

3. **"对于多语言的 monorepo，代码修复智能体如何跨语言处理？"**

    需要构建跨语言的抽象语法树映射和依赖分析。例如，TypeScript 调用 Go 微服务时，修复需要在两端同步。实践中，可以按语言拆分处理，然后在接口契约层面做一致性校验。

4. **"如何处理修复优先级——多个问题中哪些应该自动修复，哪些需要人工？"**

    根据风险等级划分：语法/类型错误自动修复，运行时异常自动修复+人工确认，安全漏洞自动修复+强制人工审核，架构/设计问题仅生成建议。

## 学习建议

- 阅读 OpenAI Codex CLI 的源码，理解其 fix 工具的 prompt 设计和验证机制
- 熟悉 unified diff 格式和 patch 工具的使用
- 掌握至少一种静态分析工具（ESLint, Semgrep, CodeQL）
- 了解 Jiwei 等代码修复论文中的评估方法
- 实践：给一个开源项目的历史 commit 做回放，看你的修复智能体能否生成同样的 patch
